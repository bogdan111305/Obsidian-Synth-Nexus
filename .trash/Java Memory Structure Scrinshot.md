# Полная структура памяти в Java (JVM)

## 1. **Heap (куча)**

### 📌 Где располагается:

- Внутри **управляемой JVM памяти**.
    
- Выделяется при старте JVM (настраивается `-Xms`, `-Xmx`).
    
- Может расширяться или сжиматься динамически.
### Что хранит:

- **Объекты** и **массивы**, созданные с помощью `new`.
    
- Все объекты, на которые указывают **ссылки (references)**.
    
- Замыкающие объекты (closures), лямбды, строки.
### Управляется GC:

- Да, **полностью**.
    
- Объекты очищаются, когда на них **нет сильных ссылок**.
### Структура кучи:

#### Молодое поколение (Young Generation)

- **Eden** — сюда попадают новые объекты.
    
- **Survivor S0/S1** — объекты, пережившие один или несколько GC.
#### Старое поколение (Old Generation, Tenured)

- Объекты, которые "пережили" несколько сборок в Young.
    
- Часто долго живущие объекты: кеши, singletons и пр.
#### Примерный жизненный цикл:

`Object o = new Object(); // создаётся в Eden`

- Первый GC → перемещается в S0
    
- Следующий → в S1
    
- Потом → в Old Gen
### Управление:

`-Xms512m                 # начальный размер heap -Xmx2048m                # максимальный размер heap -XX:NewRatio=2           # отношение old/new (2:1) -XX:SurvivorRatio=6      # Eden/S0/S1 = 6:1:1`

##  2. **Metaspace**

### Где располагается:

- В **native-памяти (off-heap)**, **вне JVM heap**.
    
- Начиная с Java 8, заменила `PermGen`.
### Что хранит:

- **Метаданные загруженных классов**:
    
    - Названия классов, методы, сигнатуры, аннотации
        
    - Информация о класс-лоадерах
        
    - Скомпилированный байт-код (частично)
### Как работает:

- Когда JVM загружает класс через ClassLoader, она выделяет память в Metaspace.
    
- После выгрузки класса память **освобождается**, если GC считает, что класс-лоадер больше не нужен.
### OutOfMemoryError:

- Возможен, если Metaspace **не ограничен** и память ОС заканчивается:

`Exception in thread "main" java.lang.OutOfMemoryError: Metaspace`
### Управление:

`-XX:MetaspaceSize=128m          # стартовый размер -XX:MaxMetaspaceSize=512m       # ограничение роста`
### Что такое **native memory** (off-heap)?

- Это **оперативная память**, выделенная **не самой JVM** напрямую (не управляется сборщиком мусора).
    
- JVM использует системные вызовы ОС (через `malloc` или `mmap`), чтобы получить доступ к обычной **"сырой" памяти**, минуя Java-кучу (heap).
### Чем отличается от обычной кучи (`Heap`)?

|Параметр|Java Heap|Native (off-heap) memory|
|---|---|---|
|Где размещается|Внутри JVM|Вне JVM, через ОС|
|Кто управляет|JVM (через GC)|Программа вручную / JVM частично|
|Что хранит|Java-объекты|Метаданные, буферы, кеши|
|Пример|`new Object()`|`ByteBuffer.allocateDirect()`|
### Что размещается в native memory:

1. **Metaspace** (метаданные классов)
    
2. **Direct ByteBuffer** (`allocateDirect()`)
    
3. **JIT Code Cache**
    
4. **Native Stacks** (JNI, потоки ОС)
    
5. **Thread data**, мониторинг, TLS
    
6. **Компоненты GC (ZGC, Shenandoah)** — часть их структур тоже в native
### Почему JVM использует off-heap?

- Снижает нагрузку на сборщик мусора.
    
- Быстрее для **IO и NIO** операций.
    
- Позволяет использовать больше памяти (heap ограничен `-Xmx`, а native — пределами ОС).
### Важный нюанс:

Off-heap-память **не освобождается автоматически** JVM, если ты сам её не очищаешь (например, через `cleaner`, `Unsafe` или finalize в случае DirectBuffer'ов).
## 3. **Java Stack (стек вызовов)**

### Где располагается:

- Каждый поток имеет **собственную область памяти**.
    
- JVM выделяет стек при создании потока.
### Что хранит:

- Локальные переменные (в том числе ссылки на объекты в куче)
    
- Параметры метода
    
- Ссылки на `this`
    
- Стековые кадры (Stack Frames) вызванных методов
### Ошибки:

- `StackOverflowError`: рекурсия без выхода
    
- Размер ограничен:

`-Xss1m   # размер стека на поток`
##  4. **Program Counter (PC Register)**

### Где располагается:

- На каждый поток JVM выделяет **регистровую ячейку**.
    
- Минимальный объем памяти.
### Что хранит:

- **Адрес текущей инструкции**, которую исполняет поток.
    
- Если поток вызывает метод, PC указывает на следующую команду.

📌 Уникален для каждого потока.

## 5. **Native Method Stack**

### Где располагается:

- В **native memory** (вне JVM).
    
- Используется для вызова **не-Java** методов (через JNI).
### Что хранит:

- Адреса нативных библиотек
    
- Аргументы/результаты вызова нативных методов
## 6. **Code Cache**

### Где располагается:

- В native-памяти JVM.
    
- Хранит **скомпилированные версии байт-кода**, преобразованные JIT-компилятором.
### Что делает:

- Ускоряет выполнение часто вызываемых методов.
    
- Участвует в оптимизациях HotSpot.
### Настройка:

`-XX:ReservedCodeCacheSize=128m`
## 7. **Direct Memory (Off-Heap)**

### Где располагается:

- **Вне heap**, напрямую в **native memory**.
### Используется:

- `ByteBuffer.allocateDirect()` — для работы с сетевыми буферами
    
- Сторонними библиотеками (например, Netty, Spark, Cassandra)
    
- `Unsafe.allocateMemory()`
### Опасно:

- JVM **не управляет** этой памятью напрямую!
    
- Нужно **освобождать вручную**, иначе утечки памяти.
### Управление:

`-XX:MaxDirectMemorySize=512m`
# Как JVM управляет памятью

## Сборка мусора (GC)

JVM использует разные **алгоритмы GC** в зависимости от версии и флагов:

|GC Алгоритм|Подходит для|
|---|---|
|Serial GC|Малые приложения, отладка|
|Parallel GC|Высокая пропускная способность|
|CMS|Устаревший, предсказуемые паузы|
|G1 (по умолчанию)|Баланс пауз и скорости|
|ZGC / Shenandoah|Минимальные паузы, большие хипы|

📌 Управляется флагами:

`-XX:+UseG1GC -XX:+UseZGC`
## Инструменты анализа

|Инструмент|Назначение|
|---|---|
|`jmap -heap <pid>`|Показать структуру heap|
|`jvisualvm`|Визуализация памяти, профилирование|
|`jcmd <pid> GC.*`|Детальный анализ памяти и классов|
|`MAT (Eclipse)`|Анализ дампа памяти|
|`-Xlog:gc*`|Логирование GC|
### Как сборщик мусора работает со стеком?

- Сборщик мусора **не удаляет объекты из стека**, потому что там хранятся только примитивы и ссылки, а не объекты.
    
- **Задача сборщика — найти живые объекты в куче.**
    
- Для этого сборщик мусора сначала находит **корни (GC roots)** — это начальные точки отсчёта для поиска живых объектов.
    
- **Одним из таких корней являются ссылки из стека — локальные переменные и параметры методов.**
### Процесс поиска живых объектов

- GC начинает с **GC roots** — это:
    
    - Ссылки из стека (локальные переменные активных потоков),
        
    - Ссылки из статических полей,
        
    - Ссылки из регистров,
        
    - Другие специфичные корни (например, JNI).
        
- Затем, по этим ссылкам GC рекурсивно ищет все объекты в куче, на которые есть ссылки — это живые объекты.
    
- Объекты, на которые нет ссылок от корней (в том числе и от стека), считаются мертвыми и подлежат удалению.

**Metaspace** — это **не часть Heap (кучи)**. Это **отдельная область памяти вне кучи**, которая используется JVM для хранения **метаданных классов**.
### Подробно:

#### 🔹 До Java 8 — PermGen

- Метаданные классов (структура классов, методы, поля, константы) хранились в **PermGen** (Permanent Generation).
    
- **PermGen была частью кучи** (Heap), но со своими ограничениями и проблемами (например, OutOfMemoryError при подгрузке большого количества классов).
    

#### 🔹 С Java 8 — Metaspace

- Вместо PermGen была введена **Metaspace**.
    
- **Metaspace не входит в Heap.** Она находится **в native memory** (памяти операционной системы вне управления сборщиком мусора Java).
    
- Это значит, что размер Metaspace по умолчанию ограничен не JVM, а доступной памятью ОС (если явно не ограничен параметром).
### Что хранится в Metaspace?

- Метаданные классов: структура, методы, сигнатуры, constant pool и т.п.
    
- Символы классов.
    
- Некоторые внутренние структуры JVM.

> ❗Объекты экземпляров классов (`new MyClass()`) по-прежнему хранятся в **Heap**, а вот информация **о самом классе `MyClass`** — в **Metaspace**.

### Где физически находится Metaspace?

- В **native memory**, управляемой через C/C++ код JVM.
    
- JVM запрашивает память у ОС напрямую через `malloc` и другие системные вызовы.
    
- Это даёт JVM гибкость и устраняет ограничения старого PermGen.

### Как управлять Metaspace?

Можно указать максимальный размер:

`-XX:MaxMetaspaceSize=256m`

или начальный:

`-XX:MetaspaceSize=128m`
### Визуально:

`+---------------------------+ |       Native Memory       | |  +---------------------+  | |  |     Metaspace       |  |  <-- Метаданные классов |  +---------------------+  | |                           | +---------------------------+  +---------------------------+ |          Heap             | |  +---------------------+  | |  | Young Generation    |  | |  | Old Generation      |  |  <-- Объекты, строки, массивы и т.п. |  +---------------------+  | +---------------------------+`

## Как очищается Metaspace?

### 1. **Через сборку мусора (GC), но не напрямую**

- Метаспейс сам по себе **не собирается сборщиком мусора как куча**, потому что он находится **в native memory**.
    
- **Однако**: когда GC определяет, что **класс больше не используется (unreachable)**, он может инициировать **выгрузку этого класса**.
    
- **В момент выгрузки класса JVM освобождает его метаданные из Metaspace**, тем самым очищая память.
    

> 💡 Ключ: **class unloading** — единственный способ очистки Metaspace.
### 2. **Когда происходит class unloading?**

- Class unloading может происходить **только во время Full GC**.
    
- Это зависит от:
    
    - Используемого GC (например, G1, CMS, Serial),
        
    - Наличия выгружаемых классов (например, в динамических приложениях, где классы подгружаются и выгружаются, как в `ClassLoader`, OSGi, сервлет-контейнерах, плагинах и пр.).
### 3. **Условия для выгрузки классов:**

Чтобы класс был выгружен, должны выполниться **все** условия:

- У него **нет активных экземпляров** в куче.
    
- На его **ClassLoader нет активных ссылок**.
    
- Сам `Class` и связанные с ним ресурсы недоступны из GC roots.
## Как принудительно управлять этим?

- Можно **включить выгрузку неиспользуемых классов**:
    
    `-XX:+ClassUnloading -XX:+ClassUnloadingWithConcurrentMark`
    
    Обычно включены по умолчанию.
    
- Задать лимиты:
    
    `-XX:MaxMetaspaceSize=512m`
    
- Запустить вручную Full GC (например, из кода или через инструмент):
    
    `System.gc();`
    

> ❗ **Вызов `System.gc()` — не гарантия очистки Metaspace.** Это лишь рекомендация, и JVM может проигнорировать её.

## Важно: когда проблема с Metaspace возникает?

- При **утечках `ClassLoader`'ов** — например, в приложениях с горячей перезагрузкой плагинов, OSGi, динамическими сервлетами и т.п.
    
- Когда классы подгружаются повторно, но старые `ClassLoader` всё ещё висят в памяти — тогда **Metaspace растёт бесконечно**, и в итоге может произойти `OutOfMemoryError: Metaspace`.
## 1. **Статические поля (`static`)**

### Где хранятся:

- В **Metaspace** (до Java 8 — в PermGen)
    
- Конкретно — в **данных, привязанных к `Class<?>` объекту**
    
- Не относятся к какому-либо экземпляру объекта
### Когда создаются:

- При **загрузке класса JVM'ой**, обычно при первом обращении
### Пример:

`public class MyClass {     static int counter = 0; }`

- Поле `counter` хранится **один раз на весь класс**, не в куче.
## 2. **Обычные (нестатические) поля**

### Где хранятся:

- В **куче (Heap)**, внутри **каждого объекта**.
### Когда создаются:

- При **создании экземпляра класса (через `new`)**
### Пример:

`public class MyClass {     int value; }`

- Каждый объект `MyClass` имеет **своё** поле `value`.
## 3. **Статические методы**

### Где хранятся:

- В **Metaspace**, вместе с метаданными класса.
    
- Методы — это часть **структуры `Class<?>`**, не объекта.
### Особенности:

- Не участвуют в **виртуальном вызове** (нет динамической диспетчеризации)
    
- Вызываются напрямую по имени класса
### Пример:

`public class MyClass {     static void log() {         System.out.println("static log");     } }`

- Метод `log()` — существует **один раз**, не зависит от объектов
## 4. **Обычные (нестатические) методы**

### Где хранятся:

- Также в **Metaspace**, как часть метаданных класса
    
- Не дублируются на каждый объект
### Но:

- У объекта в куче есть **ссылка на `Class<?>`**, и через неё JVM вызывает нужный метод
    
- При вызове метода `this` ссылается на объект
### Пример:

`public class MyClass {     void print() {         System.out.println("Hello");     } }`

- Метод `print()` существует один раз, но может быть вызван на любом экземпляре
## Сводная таблица

| Элемент             | Где хранится | Привязан к           | Когда создаётся      |
| ------------------- | ------------ | -------------------- | -------------------- |
| `static` поле       | Metaspace    | Класс                | При загрузке класса  |
| Нестатическое поле  | Heap         | Объект               | При создании объекта |
| `static` метод      | Metaspace    | Класс                | При загрузке класса  |
| Нестатический метод | Metaspace    | Класс (через объект) | При загрузке класса  |
## Что такое `Class<?>` в Java?

- Это **обычный объект** (экземпляр класса `java.lang.Class`)
    
- Представляет **метаинформацию** о загруженном классе (его поля, методы, имя, суперкласс и т.д.)
    
- На каждый загруженный класс JVM создаёт **ровно один объект `Class<?>`**
## Где хранится `Class<?>`?

### 1. **Объект `Class<?>` хранится в Heap (куче)**

java

КопироватьРедактировать

`Class<?> clazz = MyClass.class;`

- Переменная `clazz` ссылается на объект `Class<MyClass>` в **куче**
    
- Это **тот же Heap**, где хранятся обычные объекты
    
- Поэтому на него можно применять обычные правила (например, `clazz.hashCode()` и т.п.)

### 2. **Метаданные класса (структура, методы, поля и т.п.) — в Metaspace**

- JVM выделяет **Metaspace** (начиная с Java 8) в **нативной памяти** (вне Heap)
    
- Там хранятся:
    
    - Имя класса
        
    - Метод-таблицы (`vtable`)
        
    - Таблицы полей и методов
        
    - Статические поля и методы
        
    - Константный пул (`Constant Pool`)
        
    - Дескрипторы типов
        
- Это внутренние структуры JVM, **необъектные**
## Схема:

        `Native Memory (Metaspace)         ┌─────────────────────────────┐         │  Метаданные класса          │         │  - Имя класса               │         │  - Суперкласс               │         │  - Список методов           │         │  - Список полей             │         │  - Статические поля         │         └────────────▲────────────────┘                      │       Heap           │       ┌──────────────┴──────────────┐       │     Class<MyClass>          │  ← объект Class<?>       │ (java.lang.Class instance)  │       └─────────────────────────────┘`
## Как это работает?

- При первом использовании класса JVM:
    
    1. Загружает байт-код через ClassLoader
        
    2. Создаёт метаданные в Metaspace
        
    3. Создаёт объект `Class<?>` в Heap, ссылающийся на эти метаданные
        
- Объект `Class<?>` доступен через `MyClass.class`, `obj.getClass()` и т.п.
    
- Этот объект используется для рефлексии, получения информации о типе и т.д.
## Как освобождается?

- Когда **класс становится не нужен** (например, при выгрузке ClassLoader'а, как в OSGi, WebApp-контейнерах):
    
    - `Class<?>` объект и связанные метаданные становятся **доступны для сборки мусора**
        
    - Metaspace очищается автоматически, **но не сразу**, а когда классы реально выгружены
        
    - Это особенно важно при **динамической загрузке/выгрузке классов**

- ## Что такое `ClassLoader`?

- Это **класс** в Java (`java.lang.ClassLoader`)
    
- Его задача — найти и загрузить байт-код класса (`.class` файл), превратить его в объект `Class<?>`, и передать JVM для использования
    
- Он используется **при первом обращении к классу**
## Как работает загрузка класса?

1. **ClassLoader находит файл** (обычно `.class` или из JAR)
    
2. Читает байт-код
    
3. Создаёт метаданные класса в **Metaspace**
    
4. Создаёт объект `Class<?>` в **куче**
    
5. Возвращает его
## Иерархия загрузчиков классов в JVM:

`Bootstrap ClassLoader    ↑ Extension ClassLoader    ↑ Application ClassLoader (System)    ↑ Ваши собственные ClassLoader'ы`

### 1. **Bootstrap ClassLoader** (написан на C++):

- Загружает основные классы Java (`java.lang.*`, `java.util.*` и т.д.)
    
- Работает без Java-кода
### 2. **Extension ClassLoader** (до Java 9):

- Загружал классы из `lib/ext`
    
- С Java 9 заменён на модульную систему
### 3. **Application ClassLoader**:

- Загружает классы из `classpath` (ваши `src`, JAR-файлы)
    
- Это основной загрузчик в большинстве Java-программ
### 4. **Custom ClassLoaders**:

- Вы можете создать свой `ClassLoader`, чтобы:
    
    - Загружать классы из сети, БД, архива
        
    - Перезагружать классы (например, плагины)
        
    - Ограничивать доступ к классам (песочница)
## Принцип делегирования (delegation model)

- ClassLoader **сначала делегирует загрузку родителю**
    
- Только если родитель не смог найти класс — пытается загрузить сам

`ClassLoader parent = getClass().getClassLoader(); ClassLoader myLoader = new MyCustomLoader(parent);`
## Пример использования

`ClassLoader loader = MyClass.class.getClassLoader(); System.out.println(loader); // sun.misc.Launcher$AppClassLoader`
## Пример кастомного загрузчика

`public class MyClassLoader extends ClassLoader {     @Override     protected Class<?> findClass(String name) throws ClassNotFoundException {         byte[] bytes = ... // читаем байт-код как массив байтов         return defineClass(name, bytes, 0, bytes.length);     } }`
## Зачем это нужно?

- **OSGi**, **Web-контейнеры (Tomcat, Jetty)**, **плагин-системы** создают свой ClassLoader
    
- Когда ClassLoader становится **неиспользуемым**, JVM может выгрузить все классы, которые он загрузил → **освобождение Metaspace**
    
- Это ключ к **динамической загрузке/выгрузке** в больших системах
## Cтатическая переменная — это поле, привязанное к классу, а не к объекту.

### Пример:

`public class MyClass {     static String name = "Hello"; }`
## Что здесь происходит на уровне памяти?

### 1. **Переменная `name` (сама ссылка)**

- Хранится в **Metaspace** (в структуре, ассоциированной с объектом `Class<MyClass>`).
    
- Это **не в куче (Heap)**, а в **нативной памяти** (начиная с Java 8 — Metaspace).
    
- Эта переменная — **ссылка** на объект типа `String`.
### 2. **Объект `String "Hello"`**

- Создан в **Heap** (или в String Pool — об этом ниже).
    
- Его можно изменить или заменить, если переменная `name` переназначится.
## Визуализация:

`┌────────────────────┐ │    Metaspace       │ │  (данные о классе) │ │  static name ----┐ │ └────────────────────┘│                       ▼                 ┌──────────────┐                 │  Heap        │                 │ "Hello"      │                 └──────────────┘`
## Если ты напишешь:

`MyClass.name = "World";`

- Ссылка `name` в Metaspace будет теперь указывать на другой объект `"World"` в Heap.
    
- Старый объект `"Hello"` может быть удалён сборщиком мусора (если больше не используется).
## Важно:

- Статическая переменная — **одна на весь класс**, она **не входит в объект**.
    
- Если статическая переменная ссылается на объект в куче — она **удерживает его от сборки мусора** (до тех пор, пока класс не выгрузится).

## Пример со своей ссылкой:

`public class Example {     static List<String> list = new ArrayList<>(); }`

- `list` (сама переменная-ссылка) — в **Metaspace**
    
- `new ArrayList<>()` — в **Heap**

## Если **статическая переменная** — это **примитивный тип**, то всё **ещё проще**:

### Пример:

`public class MyClass {     static int count = 42; }`
## Где хранится `count`?

- Переменная `count` — **статическая** → значит, она принадлежит **классу**, а **не объекту**.
    
- Она хранится в **Metaspace** — внутри специальной **внутренней структуры JVM** под названием _"Class metadata"_ (метаинформация о классе).
    
- Значение `42` хранится **прямо там**, **внутри Metaspace** — **без всякой кучи**.
### То есть:

- `count` — **не в Heap**
    
- `count` — **не в стеке**
    
- `count` — **в Metaspace** (в области, связанной с `Class<MyClass>`)
## Примитивы: где хранятся?

|Тип переменной|Где хранится значение примитива|
|---|---|
|Локальная переменная|В **стеке (stack)**|
|Поле экземпляра класса|В **куче (heap)**|
|Статическая переменная|В **Metaspace**|
## Вывод:

- Если **переменная статическая**, не важно — ссылка это или примитив — она принадлежит **классу**.
    
- Если это **примитив**, то JVM просто выделяет под него место в **структуре класса в Metaspace**, и **там его и хранит**.
## если переменная `static final` и примитивная (или `String`), то:

### ➤ Она становится **компилируемой константой**, то есть:

- Значение **встраивается прямо в байткод** всех классов, которые к ней обращаются.
    
- JVM даже **не обращается к полю класса**, потому что оно **инлайнится** во время компиляции.
## Пример:

`public class MyClass {     public static final int VALUE = 42; }`

Если другой класс делает:

`System.out.println(MyClass.VALUE);`

То **в байткоде** вместо обращения к `MyClass.VALUE` будет просто `ldc 42`.
## Где тогда хранится значение?

|Область|Что хранится|
|---|---|
|Metaspace|Само поле `VALUE` как часть описания класса (в Class metadata) — **да**|
|Другой класс|Значение `42` будет **вшито прямо в его байткод**|
|Heap|Ничего, потому что это **примитив**|
|Stack|Значение **временно копируется в стек** при использовании|
## То же самое с `String`:

`public static final String NAME = "HELLO";`

- Если строка известна на этапе компиляции, она тоже **инлайнится**.
    
- Строка при этом **попадает в String Pool**, а ссылка может быть встроена в байткод.
## Важно: что происходит при изменении значения?

Если ты изменишь `VALUE = 43` и перекомпилируешь только `MyClass`, а другие классы — нет, они **будут использовать старое значение**, пока не пересоберутся.

🔸 Поэтому `public static final` **часто используют только для настоящих констант**, а не для чего-то, что может меняться в будущем.
## Резюме:

| Случай                     | Где хранится                   | Доступ в байткоде        |
| -------------------------- | ------------------------------ | ------------------------ |
| `static`                   | В Metaspace                    | Через `getstatic`        |
| `static final` (константа) | Инлайнится, значение вшивается | Через `ldc` или напрямую |
## Как JVM загружает классы

Когда ты впервые обращаешься к классу (например, через `new Child()`), JVM:

1. Ищет `.class` файл (через `ClassLoader`)
    
2. Загружает байткод → создаёт структуру **Class** в **Metaspace**
    
3. Эта структура содержит:
    
    - Имя класса
        
    - Родительский класс
        
    - Таблицу методов (vtable)
        
    - Таблицу полей
        
    - Ссылки на константы, аннотации, интерфейсы
        
4. **Для каждого класса (родителя и потомка)** создаётся отдельный `Class<?>` объект в куче, и его данные — в Metaspace
## Metaspace (с 8-й Java)

- Это область **вне кучи** (в нативной памяти)
    
- Хранит **метаинформацию** о каждом классе:
    
    - Имена методов и полей
        
    - Их типы
        
    - Таблицы вызовов
        
    - Доступ к `static` полям
        
    - Ссылки на суперкласс, интерфейсы и т.д.
## 2. Что происходит при `new Child()`

`class Parent {     int a = 10; }  class Child extends Parent {     int b = 20; }  Child c = new Child();`

### Что делает JVM:

1. Убедилась, что **оба класса** уже загружены в Metaspace
    
2. В куче (Heap) создаёт **объект класса `Child`**
    
    - Он содержит:
        
        - Поле `a` (унаследованное от `Parent`)
            
        - Поле `b` (определено в `Child`)
            
3. Вызывает сначала конструктор `Parent`, затем `Child`
### Что реально лежит в объекте `c` в куче:

``Heap ┌────────────────────────────────┐ │   Объект класса Child          │ │ ┌─────────────┬──────────────┐ │ │ │ поле `a`=10 │ поле `b`=20  │ │ ← размещены подряд │ └─────────────┴──────────────┘ │ │ Служебные данные JVM (например, ссылка на Class<?>) │ └────────────────────────────────┘``

- Объект **цельный**: это один блок в куче, содержащий всё — и от родителя, и от потомка.
    
- **Не создаётся отдельный объект для родителя** — просто в одном объекте есть все поля, включая унаследованные.
## 3. Как работают методы при наследовании

В Metaspace создаётся таблица методов (vtable) — аналог виртуальных методов в C++.

- Каждый объект имеет **ссылку на свой `Class<?>`**
    
- В таблице указано, **какие методы он должен вызывать**
    
- При переопределении методов `Child` "переписывает" указатель на свою версию
## Цепочка загрузки классов

Если `Child` extends `Parent`, то:

1. JVM сначала загружает `Parent` (и его родителя, и так далее до `Object`)
    
2. Затем загружает `Child`, который ссылается на `Parent`
    
3. Каждый из них получает свою структуру в Metaspace
    
4. Эти структуры связаны: `ChildClass` → `ParentClass`
## Сравнение по памяти

|Что|Где хранится|
|---|---|
|Экземпляр объекта|Heap|
|Унаследованные поля|В том же объекте в Heap|
|Метаданные о классе|Metaspace|
|Таблицы методов и полей|Metaspace|
|Статические переменные|Metaspace|
|Локальные переменные|Stack|
## Как это всё связывается?

`Child c = new Child();`

JVM:

- Аллоцирует память в куче (для всех полей `Child`, включая от `Parent`)
    
- Привязывает к `Class<Child>` (внутри объекта)
    
- Метаданные (`Class`) уже в Metaspace
    
- Статические поля доступны через `Class<Child>`
    
- Методы виртуальны — через vtable
## Визуально:

`[Heap] ┌──────────────────────────────┐ │    Object: Child             │ │ ┌─────┬─────┐                │ │ │ a=10│ b=20│ ← поля         │ │ └─────┴─────┘                │ │ Ссылка на Class<Child>       │ └──────────────────────────────┘  [Metaspace] ┌───────────────────┐   ┌───────────────────┐ │ Class<Parent>     │   │ Class<Child>      │ │ - поле a:int      │   │ - поле b:int      │ │ - методы...       │   │ - методы...       │ │                   │   │ - superclass → ───┘ └───────────────────┘   └───────────────────┘`
