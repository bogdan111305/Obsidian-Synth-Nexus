
- **Map** — интерфейс коллекций в Java, который хранит пары **ключ-значение**.
    
- Ключ уникален, значение может повторяться.
    
- Основные операции:
    
    - `put(key, value)` — добавить или обновить пару,
        
    - `get(key)` — получить значение по ключу,
        
    - `remove(key)` — удалить по ключу,
        
    - `containsKey(key)`, `containsValue(value)` — проверить наличие.
### Иерархия Map

`java.util.Map<K,V> (интерфейс)  ├─ java.util.HashMap<K,V>  │    ├─ java.util.LinkedHashMap<K,V>  ├─ java.util.TreeMap<K,V>  ├─ java.util.concurrent.ConcurrentHashMap<K,V>  └─ java.util.WeakHashMap<K,V>`
### Основные реализации Map и их особенности

|Класс|Структура хранения|Порядок элементов|Потокобезопасность|Ключи отсортированы?|Особенности|
|---|---|---|---|---|---|
|`HashMap`|Хеш-таблица + массив|Нет|Нет|Нет|Быстрая работа (O(1) аморт.), неупорядоченный|
|`LinkedHashMap`|HashMap + связный список|В порядке вставки|Нет|Нет|Итерация в порядке вставки|
|`TreeMap`|Красно-черное дерево|Отсортированы|Нет|Да|Ключи упорядочены, операции O(log n)|
|`ConcurrentHashMap`|Разбитая на сегменты хеш-таблица|Нет|Да|Нет|Потокобезопасный, без блокировок|
|`WeakHashMap`|Хеш-таблица с ключами на weak references|Нет|Нет|Нет|Ключи удаляются сборщиком мусора при отсутствии других ссылок|
## Как устроен HashMap?

- HashMap — это **хеш-таблица**, базирующаяся на массиве бакетов.
    
- Каждый бакет — **цепочка** (связанный список) или с Java 8 — дерево (красно-черное), если много коллизий.
    
- Элемент — пара **ключ-значение** (Entry или Node), хранящаяся в бакете.
    
- Ключ хешируется с помощью `hashCode()`, дополнительно смешивается (hash spreading), и по результату вычисляется индекс в массиве:  
    `index = (n - 1) & hash` — где `n` — длина массива бакетов.
### Хранение и структура

- Основной контейнер — массив `Node<K,V>[] table` (бакеты).
    
- При добавлении:
    
    1. Вычисляется индекс бакета по хешу ключа.
        
    2. В бакете ищется совпадение по ключу через `equals`.
        
    3. Если совпадение есть — обновляем значение.
        
    4. Если нет — добавляем новый узел в цепочку или дерево.
        
- При превышении порога загрузки (load factor = 0.75 по умолчанию), массив бакетов **расширяется в 2 раза**, и все элементы перераспределяются.
### Как расширяется HashMap?

- При достижении количества элементов > `capacity * load factor` происходит **resize** — увеличивается длина массива бакетов в **2 раза**.
    
- Все элементы перераспределяются, т.к. индекс зависит от размера массива.
    
- Это дорогая операция — O(n), но она происходит нечасто, обеспечивая амортизированную сложность операций.
### **Big-O** операциям в `HashMap` в Java с пояснениями:

| Операция                                | Время выполнения (Big-O) | Пояснение                                                                                |
| --------------------------------------- | ------------------------ | ---------------------------------------------------------------------------------------- |
| **get(key)**                            | O(1) (амортизированное)  | Быстрый доступ через вычисление хеша и переход к бакету; поиск в цепочке или дереве O(1) |
| **put(key, value)**                     | O(1) (амортизированное)  | Добавление с возможной коллизией; при переполнении — resize O(n), но редко               |
| **remove(key)**                         | O(1) (амортизированное)  | Поиск и удаление узла в бакете, аналогично get                                           |
| **containsKey(key)**                    | O(1) (амортизированное)  | Поиск ключа по хешу, аналогично get                                                      |
| **containsValue(value)**                | O(n)                     | Придётся перебрать все элементы, т.к. поиск по значению не индексируется                 |
| **iteration (по всем элементам)**       | O(n)                     | Пробегает по всем бакетам и элементам                                                    |
| **resize (увеличение массива бакетов)** | O(n)                     | При достижении порога загрузки происходит перераспределение всех элементов               |
**Почему амортизированное O(1)?**

- Большинство операций выполняется быстро (приблизительно константное время).
    
- Однако периодически при добавлении вызывается операция **resize** — удвоение массива бакетов и перераспределение элементов — это O(n).
    
- Но так как resize происходит редко, средняя стоимость операции остаётся O(1).

**Примечания:**

- Если хеш-функция распределяет ключи плохо (много коллизий), то время операций может деградировать до O(n).
    
- Начиная с Java 8, в случае большого количества коллизий цепочка превращается в сбалансированное красно-чёрное дерево, что уменьшает время поиска до O(log n).
## Устройство LinkedHashMap

- LinkedHashMap наследуется от HashMap и хранит элементы в **двусвязном списке** в порядке вставки (или в порядке доступа, если включён accessOrder).
    
- Внутренние структуры:
    
    - **Хеш-таблица**: массив бакетов (Entry[]), как у HashMap.
        
    - **Двусвязный список**: каждый элемент (Entry) дополнительно содержит ссылки на предыдущий и следующий элементы.
        
- Каждый элемент — объект Entry, расширяющий HashMap.Node, с полями:
    
    - key, value, hash
        
    - next (для цепочки коллизий)
        
    - before и after (для двусвязного списка)
### Расширение (resize)

- При превышении порога загрузки (load factor, по умолчанию 0.75) происходит **resize** — удвоение размера массива бакетов.
    
- Элементы перераспределяются по новому массиву.
    
- Двусвязный список при этом не меняется, только хеш-таблица обновляется.
    
- Resize — O(n) операция, но происходит редко.
### Big-O операций

| Операция                        | Время (Big-O)         | Пояснения                                                       |
| ------------------------------- | --------------------- | --------------------------------------------------------------- |
| **get(key)**                    | O(1) амортизированное | Поиск по хешу и цепочке в бакете, потом доступ к Entry.         |
| **put(key, value)**             | O(1) амортизированное | Добавление в хеш-таблицу и в конец двусвязного списка.          |
| **remove(key)**                 | O(1) амортизированное | Поиск и удаление из хеш-таблицы и из двусвязного списка.        |
| **containsKey(key)**            | O(1) амортизированное | Аналогично get.                                                 |
| **containsValue(value)**        | O(n)                  | Поиск по значениям — приходится перебирать все элементы.        |
| **итерация по всем элементам**  | O(n)                  | Проходит по двусвязному списку в порядке вставки (или доступа). |
| **resize (расширение массива)** | O(n)                  | Удвоение массива бакетов и перераспределение элементов.         |
|                                 |                       |                                                                 |
|                                 |                       |                                                                 |
### Особенности операций по позициям

- LinkedHashMap не поддерживает доступ по индексу — операции типа get по индексу **нет**.
    
- Добавление в начало или середину **явно нельзя**, элементы добавляются в конец двусвязного списка (или перемещаются в конец при accessOrder = true).
    
- Удаление "с начала" — можно, вызвав remove на первом элементе (через итератор или key).
    
- Добавление в конец — всегда при put.
    
- Удаление из середины или конца — удаление по ключу, работает за O(1) благодаря хеш-таблице + двусвязному списку.
### Итог

- LinkedHashMap — это **хеш-таблица + двусвязный список для порядка элементов**.
    
- Операции поиска, вставки, удаления — амортизированное O(1).
    
- Итерация идёт в порядке вставки или доступа.
    
- Нет доступа по индексу.
    
- Расширение массива бакетов происходит как в HashMap — удвоение, O(n).
## Устройство TreeMap

- **TreeMap** реализован на основе **красно-чёрного дерева (Red-Black Tree)** — сбалансированного бинарного дерева поиска.
    
- Каждый узел дерева содержит пару ключ-значение (`Map.Entry`), ссылки на левое и правое поддерево, а также на родителя.
    
- Элементы в TreeMap хранятся **отсортированными по ключу** — либо с помощью естественного порядка ключей (Comparable), либо с помощью `Comparator`, переданного в конструктор.
    
- Внутренних массивов или хеш-таблиц у TreeMap нет — это именно древовидная структура.
### Расширение и масштабирование

- Так как структура — дерево, никакого расширения массива не происходит.
    
- Вставка, удаление и поиск приводят к перестройкам (поворотам, перекраске) дерева, чтобы сохранять балансировку.
    
- Размер дерева увеличивается динамически с добавлением элементов.
### Big-O операций

|Операция|Время (Big-O)|Пояснения|
|---|---|---|
|**get(key)**|O(log n)|Поиск по сбалансированному дереву, логарифмическая сложность.|
|**put(key, value)**|O(log n)|Вставка с поддержанием баланса красно-чёрного дерева.|
|**remove(key)**|O(log n)|Удаление узла с балансировкой.|
|**containsKey(key)**|O(log n)|Поиск ключа аналогично get.|
|**containsValue(value)**|O(n)|Значения не индексируются — перебор всех узлов.|
|**итерация по ключам или значениям**|O(n)|Итерируем элементы в отсортированном порядке — последовательный обход дерева.|
### Операции по позициям (индексу, началу, середине, концу)

- В **TreeMap нет прямого доступа по индексу**, так как это не массив, а дерево.
    
- Доступ к элементам по порядку осуществляется через **итератор** — последовательный обход в отсортированном порядке.
    
- Добавление в **начало, середину, конец** в терминах индексов не применимо — новые элементы вставляются согласно их ключу.
    
- Удаление из начала (минимального ключа) или конца (максимального ключа) можно сделать через методы:
    
    - `firstKey()`, `firstEntry()` и `pollFirstEntry()`
        
    - `lastKey()`, `lastEntry()` и `pollLastEntry()`
        
- Добавление и удаление всегда в O(log n).
### Итог

- TreeMap — **красно-чёрное дерево**, поддерживает отсортированные ключи.
    
- Нет массива и расширений — дерево динамическое.
    
- Основные операции (по ключу) — O(log n).
    
- Нет доступа по индексу, но есть методы для первого/последнего элемента.
    
- Итерация происходит в отсортированном порядке ключей.
