# Надёжность передачи данных

#### Как кафка может потерять сообщения? 

Как мы помним работает репликация лидер-ведомый. При этом кафка не редко меняет лидера.

И если она сменит лидера, когда новый лидер не будет иметь все данные из старого (не успеет српелицировать), то данные потеряются: 
![[Pasted image 20221023140220.png|500]]

Минимизируем риск появление этой пролемы: 

![[Pasted image 20221023140533.png|300]]

`asks` - определяем подтверждение записи, для надёжности сохранности сообщений
может быть: 
- *none* - когда нам не критично потеряется сообщение или нет 
- one - лидер должен подтвердить что получил сообщение 
- all  - сообщение должно быть записано на все in-sync реплики, при такой настройке, нужно обязательно указать следующий параметр: 

`min.insync.replicas = 1` - минимальное количество *in-sync* реплик, которые должны подтвердить запись, чтобы запись считалась успешной

это всё настраивается для каждого топика отдельно.
при `acks=all` возрастает время записи 

### In-sync и not in-sync реплики

*In-sync* - реплики, до которых данные доезжают за определённое время (по умолчанию 10 секунд)
*Not in-sync* - реплики, до которых данные доезжают медленнее 

То есть если за 10 секунд реплики успевают обновлять данные с лидера, он исчитаются *in-sync* в противном случае они считаются *not in-sync*

На самом деле 10 секунд это невероятно большое время, так как во все эти 10 секунд пока kafka будет думать убирать ли эту реплику в not in-sync читать данные нельзя и соответственно будет большая задержка. 

# Формат хранения данных 

Kafka хранит данные в формате *лога*, т.е. данные записываются в конец и мы не имеем возможности записать их в середину или в начало

Формат сообщения: 
- <i style="color:#f3ff00">offcet</i> - смещение, уникально для каждого сообщения в рамках партиции 
- <i style="color:#f3ff00">created time</i> - время записи 
- <i style="color:#f3ff00">key</i> - ключ записи, важны при выборе партиции при записи. Все сообщения с одинаковым ключом попадут в одну партицию
- <i style="color:#f3ff00">value</i> - сама запись, данные которые мы передаём 

так же как данные состоят из страниц, данные в kafka сотоят и батчей(пакетов)

При записи множество сообщений собираются в пакет и этим пакетом отправляются в kafka:  
![[Pasted image 20221023175708.png|600]]
createdTime - создание последнего сообщения из пакета 

Достоинства такого подхода: 
- Высокая производительность
- Добавляет латентность, т.к. пока не накопится весь пакет сообщения не отправятся.

В Kafka есть 2 настройки которые определяют размер пакета: 
- размер пакета в Кб 
- сколько времени ждать пока заполнится пакет перед отправкой (по умолчанию 10ms)

### Формат смещения

Любое сообщение в топике можно обрисовать использованием всего двух значений:
- номер партиции
- смещение в партиции 

Смещения позваляют нескольим consumer-ам независимо читать данные из партиций
Также добаляется возможность перечитывать данные повторно 

# Удаление данных 

Kafka удаляет данные целыми файлами, т.к. она записывает в файл, то когда файл превышает определённый объём, она начинает записывать в новый.

Удаление и размер файлов может настраиваться, вплоть до того что данные не будут удаляться никогда.

Удаление данных активно использует переменную времени. Мы можем настроить так, что бы все сообщение старше 2-ух недель удалились. 

### Zero-copy 

Kafka способна отправлять данные в сеть минуя процессор, что повышает производительность

### Как Kafka хранит смещения 

Для нескольких consumer-ов создаётся <i style="color:#f3ff00">consumer group</i> - именованный объект в кафке, который лежит в отдельном топике и хранит для группы читателей хранит смещения.
