## Карта памяти JVM

```
+-------------------+   +-------------------+
|     Heap          |   |     Stack         |
|-------------------|   |-------------------|
|  Eden             |   |  frame: main()    |
|  Survivor S0/S1   |   |  frame: foo()     |
|  Old (Tenured)    |   |  ...              |
+-------------------+   +-------------------+

+-------------------+
|   Metaspace       |  (метаданные классов)
+-------------------+
|   Code Cache      |  (скомпилированный JIT-код)
+-------------------+
|   Native Stack    |  (стек нативного кода/JNI)
+-------------------+
|   Direct Memory   |  (NIO, off-heap)
+-------------------+
```

- **Heap** — для объектов, массивов, строк (вся "жизнь" Java-объектов)
- **Stack** — для локальных переменных, параметров, ссылок, вызовов методов (у каждого потока свой)
- **Metaspace** — для метаданных классов (структура, байт‑код, runtime constant pool, таблицы методов и т.п.)
- **Code Cache** — для JIT-компилированного кода
- **Native Stack** — для вызовов JNI (C/C++)
- **Direct Memory** — для быстрых буферов вне Heap
## Жизненный цикл объекта: от создания до удаления
### Как создаётся объект

- `new` — объект появляется в Eden (Heap)
- Ссылка на объект хранится в Stack (frame текущего метода), static-поле, массиве и т.д.

```java
void foo() {
    MyClass obj = new MyClass(); // obj — ссылка в стеке, объект — в Eden
}
```
### Как объект становится "мертвым"

- Пока есть хотя бы одна strong-ссылка — объект жив
- Как только ВСЕ strong-ссылки исчезли — объект становится недостижимым
## Подробно про Heap: Eden, Survivor, Old, продвижение

1. **Создание:**

   - Большинство новых объектов появляется в Eden
   - Крупные объекты (large objects) могут сразу попадать в Old Generation, минуя Eden

2. **Minor GC (сборка мусора в Young Gen):**

   - GC проходит по всем ссылкам (GC Roots)
   - Все живые объекты в Eden копируются в один из Survivor (например, S0)
   - Все мёртвые (недостижимые) объекты в Eden удаляются
   - Объекты, которые уже были в Survivor (например, S1), и снова выжили — копируются в другой Survivor (S0), их возраст увеличивается на 1
   - Если объект пережил N циклов (обычно 15, настраивается `-XX:MaxTenuringThreshold`), он перемещается в Old

3. **Переключение ролей (ping-pong):**

   - После GC роли S0 и S1 меняются местами (ping-pong)

```
[Heap: Young Gen]
+-------+    Minor GC    +---------+    Promotion    +------+
| Eden  | ─────────────▶ | Survivor| ──────────────▶ | Old  |
+-------+                |  S0/S1  |                +------+
```

>- Большинство объектов "умирает молодыми" (в Eden)
>- Только немногие доживают до Old (кэш, сессии, long-lived data)
>- Такой подход ускоряет GC: быстро очищать Eden, редко трогать Old
## Как устроен Stack и stack frame

- Каждый поток — свой Stack
- Каждый вызов метода — новый frame (кадр стека)
- В frame хранятся:
  - Локальные переменные (примитивы и ссылки)
  - Параметры метода
  - Операндный стек (для промежуточных вычислений)
  - this (для нестатических методов)
  - Адрес возврата
- Когда метод завершён — frame удаляется (pop)
### Stack Overflow: причины и предотвращение

**Stack Overflow** возникает, когда стек потока переполняется (превышен лимит `-Xss`).

**Основные причины:**
- Бесконечная рекурсия без базового случая
- Слишком глубокая рекурсия (например, обход глубокого дерева)
- Очень крупные локальные переменные (большие массивы в методах)
- Чрезмерное количество вложенных вызовов методов

**Как избежать:**
- Проверяйте базовые случаи в рекурсии
- Используйте итеративные алгоритмы вместо рекурсивных для глубоких структур
- Увеличьте размер стека `-Xss` (например, `-Xss2m`)
- Выносите крупные массивы из локальных переменных в поля класса

```java
// Плохо - бесконечная рекурсия
public int factorial(int n) {
    return n * factorial(n - 1); // нет базового случая!
}

// Хорошо - с базовым случаем
public int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}
```
## Metaspace и хранение static

**Metaspace** в Java — это область памяти, которая используется для хранения метаданных классов JVM (Java Virtual Machine). Она заменила PermGen (Permanent Generation) начиная с Java 8. Что хранится в Metaspace:

1. **Метаданные классов**:
    - Информация о структуре классов (имя, поля, методы, конструкторы).
    - Суперклассы и интерфейсы.
    - Аннотации классов.
2. **Константы**:
     - Таблица констант класса (runtime constant pool) — метаданные.
     - Пул строк (String Pool) для `String.intern()` находится в Heap (начиная с Java 7).
3. **Байт-код методов классов**:
    - Байт-код методов хранится как часть метаданных в Metaspace.
    - Машинный код JIT хранится в Code Cache.
4. **Таблицы методов и интерфейсов**:
    - Виртуальные таблицы методов (vtable) для поддержки полиморфизма.
    - Информация для диспетчеризации вызовов методов.
5. **Дополнительные данные**:
    - Информация о загрузчике классов (ClassLoader).
    - Данные для рефлексии (reflection) и динамической загрузки классов.

### Где находятся значения static-полей

- Значения static‑полей находятся в Heap как часть объекта `java.lang.Class` (class mirror).
- Метаданные самих полей (имя, тип, модификаторы) — в Metaspace.
- Ссылки из static‑полей удерживают объекты достижимыми через GC Roots.
### Особенности Metaspace:

- **Динамическое управление**: В отличие от PermGen, Metaspace автоматически расширяется (ограничено только доступной оперативной памятью).
- **Сборщик мусора**: Метаданные классов удаляются, если их загрузчик классов становится недостижимым (например, при выгрузке классов).
- **Настройка**: Размер можно ограничить параметрами JVM, например, -XX:MaxMetaspaceSize.
## Как работает GC

- GC запускается автоматически, когда Heap почти заполнен
- Minor GC — очищает Eden/Survivor, быстрый
- Full GC — очищает Old, медленный
- GC не трогает Stack! Только Heap
- GC не удаляет объекты, на которые есть хотя бы одна strong-ссылка из Stack, static, JNI

**GC Roots** — стартовые точки: локальные переменные в стеке, static-поля, ссылки из JNI
- GC строит граф достижимости: идёт по всем ссылкам от GC Roots
- Всё, что не достижимо — мусор
- **Mark**: GC помечает все достижимые объекты
- **Sweep**: GC удаляет все неотмеченные объекты, освобождая память
- **Compact**: (иногда) GC сдвигает живые объекты, чтобы убрать "дыры" в памяти

### Виды GC и их особенности (актуально для HotSpot)

- Serial GC: однопоточный, Stop-The-World (STW), подходит для маленьких heap и однопроцессорных систем. Включается `-XX:+UseSerialGC`.
- Parallel GC (Throughput): многопоточный, ориентирован на пропускную способность. `-XX:+UseParallelGC`.
- CMS (Concurrent Mark-Sweep): устаревший, частично конкурентный, минимизирует паузы, но фрагментирует память. Deprecated в Java 9, удалён в Java 14.
- G1 GC: региональный, дефолтный с Java 9 для серверных JVM. Предсказуемые паузы, young/mixed сборки, SATB-конкурентная маркировка. `-XX:+UseG1GC`.
- ZGC: низкие паузы (мс), гигабайтные/терабайтные heap, цветные указатели/релокация с барьерами. `-XX:+UseZGC`.
- Shenandoah: низкие паузы, конкурентная компакция. `-XX:+UseShenandoahGC` (в некоторых билдах).

Практика:
- С Java 9 по умолчанию G1 на серверных JVM, Parallel GC на клиентских. Для экстремально низких пауз рассмотрите ZGC/Shenandoah.
- Для максимальной пропускной способности и простоты — Parallel GC.

### Minor, Major, Full и Mixed паузы

- Minor GC: сборка только young generation (Eden+Survivor). Быстрая копирующая.
- Major GC: сборка old generation (термин исторический; детализация зависит от коллектора).
- Full GC: стоп-мир, сборка всего heap и Metaspace; максимально дорогая. Часто признак проблемы.
- Mixed GC (G1): одновременно young регионы и часть old регионов после фазы маркировки.

### GC Roots (расширенный список) и safepoints

GC Roots включают: активные потоки, их локальные переменные/операндные стеки, static-поля загруженных классов, JNI Global refs, зеркала классов (`java.lang.Class`), системные структуры JVM (например, ClassLoader-ы), JIT-профили и др. Маркировка/эвакуация выполняется в safepoint — момент, когда все потоки остановлены в точках безопасной остановки.

### Write barriers и card marking

- При записи ссылки в объект GC добавляет накладной write barrier.
- Генерационные коллекторы используют card table: память разбита на карточки (обычно 512 байт). При записи ссылки из old→young помечается карточка (card marking), чтобы при Minor GC просматривать только помеченные карточки, а не весь old.

### TLAB и путь выделения памяти

- У каждого потока есть TLAB (Thread-Local Allocation Buffer) в Eden. Выделение — быстрый bump-the-pointer в TLAB.
- Если объект не помещается в текущий TLAB — запрашивается новый TLAB; очень крупные объекты могут идти мимо TLAB напрямую в Eden/Old (в зависимости от коллектора).
- В G1 «очень крупные» (humongous) объекты (> 50% размера региона) размещаются в специальных регионах, часто логически относящихся к old.

### Escape analysis и scalar replacement

- JIT анализирует «убегание» объектов из метода/потока. Если объект не утекает, возможны оптимизации:
  - удаление синхронизации,
  - разбиение на скаляры (scalar replacement),
  - размещение на стеке «как бы» (не создавая объект в heap). В HotSpot фактического «heap→stack» перемещения нет; JIT просто избегает аллокации.

### Фазы G1: как достигаются предсказуемые паузы

- Initial Mark (STW, часто совмещается с Minor GC)
- Concurrent Mark (SATB, конкурентно)
- Remark (короткая STW для закрытия «дырок»)
- Cleanup (выбор наборов регионов old для Mixed GC)
- Evacuation Pauses: young/mixed копирующие паузы с эвакуацией регионов

```java
public class Leaky {
    private static List<Object> cache = new ArrayList<>();
    public static void add(Object o) { cache.add(o); }
}
// Объекты в cache не будут удалены, пока класс Leaky загружен
```

## Code Cache и JIT

- Code Cache хранит скомпилированный JIT-код. Есть уровни компиляции (C1 «быстрый», C2 «оптимизирующий», Tiered Compilation по умолчанию).
- Заполняемость Code Cache может вызвать OOME («CodeCache is full») и деградацию (падение в интерпретатор). Настройка: `-XX:ReservedCodeCacheSize`.

## Direct Memory и NIO off-heap

- `ByteBuffer.allocateDirect` выделяет память вне heap («Direct Memory»). Лимит: `-XX:MaxDirectMemorySize` (по умолчанию равен размеру heap минус survivor spaces).
- Off-heap также активно использует JNI/Unsafe/пулы Netty. Утечки off-heap не видны обычному GC.
- Диагностика нативной памяти: Native Memory Tracking — `-XX:NativeMemoryTracking=summary|detail`, просмотр через `jcmd VM.native_memory`.

## OutOfMemoryError: основные виды и что настраивать

- Java heap space: увеличить `-Xmx`, снизить удержание, включить GC с компакцией, поиск утечек.
- GC overhead limit exceeded: слишком много времени уходит на GC с малым возвратом памяти; часто симптом недостатка heap или утечек.
- Metaspace: увеличить `-XX:MaxMetaspaceSize`, проверить утечки загрузчиков классов.
- Compressed class space: при включённых сжатых класс-указателях; ограничивается отдельно (`-XX:CompressedClassSpaceSize`).
- Direct buffer memory: увеличить `-XX:MaxDirectMemorySize`, убедиться в своевременном освобождении direct-буферов.
- Unable to create new native thread: лимиты ОС/контейнера на потоки/память стека (`-Xss`).
- CodeCache is full: увеличить `-XX:ReservedCodeCacheSize`.

## Диагностика памяти и GC (инструменты)

- jcmd: `jcmd <pid> GC.heap_info`, `GC.run`, `VM.native_memory summary`.
- jstat: `jstat -gcutil <pid> 1s` — метрики GC.
- jmap: `jmap -histo[:live] <pid>` — топ объектов; `jmap -dump:live,format=b,file=heap.hprof <pid>` — дамп heap.
- JFR/JMC: низконакладочная профилировка (Java Flight Recorder) и анализ (Java Mission Control).
- jvisualvm, async-profiler (CPU/alloc), jhsdb (SA) для детального анализа.

## Оптимизация использования памяти и JVM tuning

### Основные параметры JVM для настройки памяти:

**Heap размеры:**
- `-Xms<size>` — начальный размер heap (рекомендуется = Xmx для избежания расширения)
- `-Xmx<size>` — максимальный размер heap
- `-XX:NewRatio=N` — соотношение old/young (по умолчанию 2, т.е. old в 2 раза больше young)
- `-XX:SurvivorRatio=N` — соотношение Eden/Survivor (по умолчанию 8, т.е. Eden в 8 раз больше одного Survivor)

**Другие области:**
- `-Xss<size>` — размер стека потока (по умолчанию 1MB на Linux/Windows)
- `-XX:MetaspaceSize=<size>` — начальный размер Metaspace
- `-XX:MaxMetaspaceSize=<size>` — максимальный размер Metaspace
- `-XX:ReservedCodeCacheSize=<size>` — размер Code Cache
- `-XX:MaxDirectMemorySize=<size>` — лимит Direct Memory

### Стратегии оптимизации:

1. **Выбор правильного GC:**
   - Для низких пауз: G1, ZGC, Shenandoah
   - Для высокой пропускной способности: Parallel GC
   - Для небольших приложений: Serial GC

2. **Настройка размеров поколений:**
   - Если много кратковременных объектов — увеличьте Young Generation
   - Если много долгоживущих объектов — увеличьте Old Generation

3. **Мониторинг и профилирование:**
   - Включите GC логи: `-Xlog:gc*:gc.log:time`
   - Используйте JFR для детального анализа аллокаций
   - Мониторьте метрики: частоту GC, длительность пауз, пропускную способность

4. **Избежание утечек памяти:**
   - Всегда закрывайте ресурсы (try-with-resources)
   - Очищайте ThreadLocal после использования
   - Следите за размерами кэшей и коллекций
   - Используйте WeakReference для слушателей событий

## Java Memory Model (JMM) и happens-before

Базовые правила happens-before:
- Внутри одного потока — порядок программный.
- `unlock` монитора происходит раньше последующего `lock` того же монитора в другом потоке.
- Запись в `volatile` происходит раньше последующего чтения того же `volatile`.
- Завершение `Thread.start()` происходит раньше начала кода потока; завершение потока происходит раньше успешного `Thread.join()`.
- Инициализация `final` полей публикуется корректно при завершении конструктора.

### Memory Barriers (памятные барьеры)

Memory barriers — это специальные инструкции процессора, обеспечивающие упорядочивание операций с памятью. JIT компилятор вставляет их для гарантий JMM.

**Основные типы барьеров:**
- **LoadLoad**: гарантирует, что чтения до барьера завершились до чтений после барьера
- **LoadStore**: чтения до барьера завершились до записей после барьера  
- **StoreLoad**: записи до барьера завершились до чтений после барьера (самый дорогой)
- **StoreStore**: записи до барьера завершились до записей после барьера

**Где вставляются:**
- При входе в `synchronized` — LoadLoad + LoadStore
- При выходе из `synchronized` — LoadStore + StoreStore  
- После записи `volatile` — StoreStore + StoreLoad
- Перед чтением `volatile` — LoadLoad + LoadStore

Барьеры предотвращают переупорядочивание операций процессором и компилятором, обеспечивая корректную видимость изменений между потоками.

## volatile и synchronized: кратко о гарантиях

- `volatile` обеспечивает видимость и упорядочивание для конкретного поля, но не атомарность составных операций (кроме чтения/записи примитивов/ссылок).
- `synchronized` обеспечивает взаимное исключение и все гарантии видимости для критической секции; выход из монитора публикует изменения.

## Дополнительно: полезные детали

- Compressed oops/klass pointers: на 64-битных JVM с умеренным heap используются сжатые указатели для экономии памяти и кэша CPU.
- String Deduplication (G1): `-XX:+UseStringDeduplication` уменьшает дубликаты внутренних массивов (`byte[]` в Java 9+, `char[]` в более ранних версиях) у одинаковых строк.
- Safepoint bias: Чрезмерное число safepoint-проверок может влиять на производительность; избегайте горячих циклов с частыми барьерами.

## ThreadLocal: как это устроено и почему бывают утечки (подробно)

- **ThreadLocal** — это специальный механизм для хранения переменных, уникальных для каждого потока.
- Каждый поток видит своё собственное значение, даже если переменная объявлена как static.
- Используется для хранения контекста, кэшей, соединений с БД, временных буферов, user session и т.д.
### ThreadLocal внутри JVM

- У каждого потока (Thread) есть скрытая структура — **ThreadLocalMap**.
- Ключ — объект ThreadLocal, значение — объект для этого потока.
- Ключи в ThreadLocalMap — слабые ссылки (`WeakReference<ThreadLocal>`).
- Когда ты вызываешь `threadLocal.set(value)`, значение кладётся в ThreadLocalMap текущего потока.
- Когда вызываешь `threadLocal.get()`, возвращается значение из ThreadLocalMap текущего потока.
- Нет синхронизации между потоками: каждый поток работает только со своей копией.

```
[Thread-1]
  ThreadLocalMap: [ThreadLocalA] → valueA1
                 [ThreadLocalB] → valueB1
[Thread-2]
  ThreadLocalMap: [ThreadLocalA] → valueA2
                 [ThreadLocalB] → valueB2
```

### Как работает очистка и почему бывают утечки

- Если поток долго живёт (например, в thread pool), а ThreadLocal не очищается(`remove()`), то значение остаётся в памяти навсегда.
- Даже если сам объект ThreadLocal становится unreachable, его ключ в ThreadLocalMap становится null, но значение (value) может остаться и не быть удалено GC, пока жив поток.
- Это приводит к утечкам памяти, особенно если value — крупный объект (например, массив, кэш, соединение).

**GC и ThreadLocal:**
- GC видит ThreadLocalMap как часть потока (Thread).
- Пока поток жив, его ThreadLocalMap живёт вместе с ним.
- Если ThreadLocal не удалён, value остаётся достижимым через Thread → ThreadLocalMap → value.

**Всегда** вызывайте `threadLocal.remove()` после использования, особенно в пулах потоков.
Всегда удаляйте значения после использования.
Не используйте ThreadLocal для хранения объектов, которые могут ссылаться на большие графы данных.
В современных фреймворках (Spring, сервера приложений) всегда проверяйте, что ThreadLocal очищается после запроса.

## Типы ссылок в Java: strong, soft, weak, phantom

### Strong Reference (сильная)

- Обычная ссылка: объект не удаляется GC, пока есть хотя бы одна strong-ссылка

  ```java
  Object o = new Object();
  ```
### Soft Reference (мягкая)

- GC удаляет объект только при нехватке памяти
- Используется для кэшей

  ```java
  SoftReference<Object> ref = new SoftReference<>(new Object());
  ```
### Weak Reference (слабая)

- GC удаляет объект при следующем проходе, если нет strong/soft-ссылок
- Используется в WeakHashMap, для кэшей, слушателей

  ```java
  WeakReference<Object> ref = new WeakReference<>(new Object());
  ```
### Phantom Reference (фантомная)

- Не даёт получить объект, используется для финализации и post-mortem cleanup

  ```java
  PhantomReference<Object> ref = new PhantomReference<>(new Object(), queue);
  ```

---

| Тип     | GC удаляет          | Когда использовать            |
| ------- | ------------------- | ----------------------------- |
| Strong  | Нет                 | Обычные объекты               |
| Soft    | При нехватке памяти | Кэш, LRU-кэш                  |
| Weak    | При следующем GC    | WeakHashMap, слушатели        |
| Phantom | После финализации   | Управление ресурсами, cleanup |

**Роль ссылок в достижимости:**
- Только strong-ссылки гарантируют "жизнь" объекта
- Soft/Weak/Phantom позволяют GC удалить объект раньше
- GC всегда начинает обход с GC Roots (stack, static, JNI), далее по всем типам ссылок
