# Как устроена память в Java: подробное руководство

---

## 1. Карта памяти JVM и обзор

```
+-------------------+   +-------------------+
|     Heap          |   |     Stack         |
|-------------------|   |-------------------|
|  Eden             |   |  frame: main()    |
|  Survivor S0/S1   |   |  frame: foo()     |
|  Old (Tenured)    |   |  ...              |
+-------------------+   +-------------------+

+-------------------+
|   Metaspace       |  (метаданные классов, static)
+-------------------+
|   Code Cache      |  (скомпилированный JIT-код)
+-------------------+
|   Native Stack    |  (для JNI)
+-------------------+
|   Direct Memory   |  (NIO, off-heap)
+-------------------+
```
- **Heap** — для объектов, массивов, строк (вся "жизнь" Java-объектов)
- **Stack** — для локальных переменных, параметров, ссылок, вызовов методов (у каждого потока свой)
- **Metaspace** — для описания классов, static-полей, методов, констант
- **Code Cache** — для JIT-компилированного кода
- **Native Stack** — для вызовов JNI (C/C++)
- **Direct Memory** — для быстрых буферов вне Heap

## 2. Жизненный цикл объекта: от создания до удаления

### 2.1. Как создаётся объект
- `new` — объект появляется в Eden (Heap)
- Ссылка на объект хранится в Stack (frame текущего метода), static-поле, массиве и т.д.

**Пример:**
```java
void foo() {
    MyClass obj = new MyClass(); // obj — ссылка в стеке, объект — в Eden
}
```

### 2.2. Как объект становится "мертвым"
- Пока есть хотя бы одна strong-ссылка — объект жив
- Как только ВСЕ strong-ссылки исчезли — объект становится недостижимым

---

## 3. Подробно про Heap: Eden, Survivor, Old, продвижение

### 3.1. Как объекты перемещаются из Eden в Survivor и Old (пошагово)
- **Eden** — область Heap (Young Generation), куда попадают все новые объекты (`new`)
- **Survivor S0/S1** — две небольшие области для "выживших" объектов
- **Old (Tenured)** — область для долгоживущих объектов

#### Как происходит перемещение (Minor GC)
1. **Создание:**
   - Каждый новый объект появляется в Eden
2. **Minor GC (сборка мусора в Young Gen):**
   - GC проходит по всем ссылкам (GC Roots)
   - Все живые объекты в Eden копируются в один из Survivor (например, S0)
   - Все мёртвые (недостижимые) объекты в Eden удаляются
   - Объекты, которые уже были в Survivor (например, S1), и снова выжили — копируются в другой Survivor (S0), их возраст увеличивается на 1
   - Если объект пережил N циклов (обычно 15, настраивается `-XX:MaxTenuringThreshold`), он перемещается в Old
3. **Переименование:**
   - После GC роли S0 и S1 меняются местами (ping-pong)

**Схема:**
```
[Heap: Young Gen]
+-------+    Minor GC    +---------+    Promotion    +------+
| Eden  | ─────────────▶ | Survivor| ──────────────▶ | Old  |
+-------+                |  S0/S1  |                +------+
```

**Пример:**
```java
for (int i = 0; i < 1000; i++) {
    Object o = new Object(); // каждый раз — новый объект в Eden
    // ... если ссылка исчезает — объект будет удалён на следующем Minor GC
}
```
- Если объект всё ещё достижим после Minor GC — он перемещается в Survivor, его возраст увеличивается
- После нескольких циклов — promotion в Old

**Почему это важно:**
- Большинство объектов "умирает молодыми" (в Eden)
- Только немногие доживают до Old (кэш, сессии, long-lived data)
- Такой подход ускоряет GC: быстро очищать Eden, редко трогать Old

## 4. Как устроен Stack и stack frame

- Каждый поток — свой Stack
- Каждый вызов метода — новый frame (кадр стека)
- В frame хранятся:
  - Локальные переменные (примитивы и ссылки)
  - Параметры метода
  - Операндный стек (для промежуточных вычислений)
  - this (для нестатических методов)
  - Адрес возврата
- Когда метод завершён — frame удаляется (pop)

**Пример:**
```java
void foo(int x) {
    int y = x + 1;
    bar(y);
}
void bar(int z) {
    // z — локальная переменная в frame bar
}
```

**Схема:**
```
[Stack]
┌─────────────┐
│ frame: bar  │ ← top
│ z           │
└─────────────┘
┌─────────────┐
│ frame: foo  │
│ x, y        │
└─────────────┘
```

## 5. Metaspace и хранение static

- **static поля** — в Metaspace, одна копия на класс
- **static методы** — в Metaspace, вызываются через имя класса
- **static final константы** — часто "вшиваются" в байт-код (inlined)
- **нестатические поля** — в Heap, внутри каждого объекта
- **нестатические методы** — в Metaspace, вызываются через объект

**Пример:**
```java
class A {
    static int counter = 0; // Metaspace
    int value;              // Heap
    static final int CONST = 42; // Metaspace + inlined
}
```

## 6. Как работает GC

- GC запускается автоматически, когда Heap почти заполнен
- Minor GC — очищает Eden/Survivor, быстрый
- Full GC — очищает Old, медленный
- GC не трогает Stack! Только Heap
- GC не удаляет объекты, на которые есть хотя бы одна strong-ссылка из Stack, static, JNI

**GC Roots** — стартовые точки: локальные переменные в стеке, static-поля, ссылки из JNI
- GC строит граф достижимости: идёт по всем ссылкам от GC Roots
- Всё, что не достижимо — мусор
- **Mark**: GC помечает все достижимые объекты
- **Sweep**: GC удаляет все неотмеченные объекты, освобождая память
- **Compact**: (иногда) GC сдвигает живые объекты, чтобы убрать "дыры" в памяти

**Пример утечки:**
```java
public class Leaky {
    private static List<Object> cache = new ArrayList<>();
    public static void add(Object o) { cache.add(o); }
}
// Объекты в cache не будут удалены, пока класс Leaky загружен
```

## 7. ThreadLocal: как это устроено и почему бывают утечки (подробно)

### 7.1. Что такое ThreadLocal и зачем он нужен
- **ThreadLocal** — это специальный механизм для хранения переменных, уникальных для каждого потока.
- Каждый поток видит своё собственное значение, даже если переменная объявлена как static.
- Используется для хранения контекста, кэшей, соединений с БД, временных буферов, user session и т.д.

### 7.2. Как устроен ThreadLocal внутри JVM
- У каждого потока (Thread) есть скрытая структура — **ThreadLocalMap**.
- Ключ — объект ThreadLocal, значение — объект для этого потока.
- Когда ты вызываешь `threadLocal.set(value)`, значение кладётся в ThreadLocalMap текущего потока.
- Когда вызываешь `threadLocal.get()`, возвращается значение из ThreadLocalMap текущего потока.
- Нет синхронизации между потоками: каждый поток работает только со своей копией.

**Схема:**
```
[Thread-1]
  ThreadLocalMap: [ThreadLocalA] → valueA1
                 [ThreadLocalB] → valueB1
[Thread-2]
  ThreadLocalMap: [ThreadLocalA] → valueA2
                 [ThreadLocalB] → valueB2
```

### 7.3. Как работает очистка и почему бывают утечки
- Если поток долго живёт (например, в thread pool), а ThreadLocal не очищается (`remove()`), то значение остаётся в памяти навсегда.
- Даже если сам объект ThreadLocal становится unreachable, его ключ в ThreadLocalMap становится null, но значение (value) может остаться и не быть удалено GC, пока жив поток.
- Это приводит к утечкам памяти, особенно если value — крупный объект (например, массив, кэш, соединение).

**GC и ThreadLocal:**
- GC видит ThreadLocalMap как часть потока (Thread).
- Пока поток жив, его ThreadLocalMap живёт вместе с ним.
- Если ThreadLocal не удалён, value остаётся достижимым через Thread → ThreadLocalMap → value.

### 7.4. Best practices
- **Всегда** вызывайте `threadLocal.remove()` после использования, особенно в пулах потоков.
- Используйте try-finally:
  ```java
  try {
      threadLocal.set(...);
      // ... работа
  } finally {
      threadLocal.remove();
  }
  ```
- Не храните большие объекты в ThreadLocal без крайней необходимости.
- Для пула потоков используйте InheritableThreadLocal только если понимаете последствия.

### 7.5. Пример с thread pool и утечкой
```java
private static final ThreadLocal<byte[]> buf = new ThreadLocal<>();
ExecutorService pool = Executors.newFixedThreadPool(10);

for (int i = 0; i < 1000; i++) {
    pool.submit(() -> {
        buf.set(new byte[1024 * 1024]); // 1MB на поток
        // ...
        // buf.remove(); // если забыть — утечка!
    });
}
```
- Если не вызвать `remove()`, каждый поток в пуле будет удерживать свой 1MB навсегда!

### 7.6. Как избежать утечек
- Всегда удаляйте значения после использования.
- Не используйте ThreadLocal для хранения объектов, которые могут ссылаться на большие графы данных.
- В современных фреймворках (Spring, сервера приложений) всегда проверяйте, что ThreadLocal очищается после запроса.

## 8. Типы ссылок в Java: strong, soft, weak, phantom

### 8.1. Strong Reference (сильная)
- Обычная ссылка: объект не удаляется GC, пока есть хотя бы одна strong-ссылка
- Пример:
  ```java
  Object o = new Object();
  ```

### 8.2. Soft Reference (мягкая)
- GC удаляет объект только при нехватке памяти
- Используется для кэшей
- Пример:
  ```java
  SoftReference<Object> ref = new SoftReference<>(new Object());
  ```

### 8.3. Weak Reference (слабая)
- GC удаляет объект при следующем проходе, если нет strong/soft-ссылок
- Используется в WeakHashMap, для кэшей, слушателей
- Пример:
  ```java
  WeakReference<Object> ref = new WeakReference<>(new Object());
  ```

### 8.4. Phantom Reference (фантомная)
- Не даёт получить объект, используется для финализации и post-mortem cleanup
- Пример:
  ```java
  PhantomReference<Object> ref = new PhantomReference<>(new Object(), queue);
  ```

### 8.5. Таблица сравнения ссылок
| Тип        | GC удаляет | Когда использовать                |
|------------|------------|-----------------------------------|
| Strong     | Нет        | Обычные объекты                   |
| Soft       | При нехватке памяти | Кэш, LRU-кэш                |
| Weak       | При следующем GC    | WeakHashMap, слушатели      |
| Phantom    | После финализации   | Управление ресурсами, cleanup|

**Роль ссылок в достижимости:**
- Только strong-ссылки гарантируют "жизнь" объекта
- Soft/Weak/Phantom позволяют GC удалить объект раньше
- GC всегда начинает обход с GC Roots (stack, static, JNI), далее по всем типам ссылок
