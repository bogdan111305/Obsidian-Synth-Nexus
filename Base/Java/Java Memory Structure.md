## 1. Обзор Структуры Памяти JVM

JVM делит память на управляемые (Heap, частично Metaspace) и неуправляемые (native memory) области. Управляемая память автоматически контролируется JVM и GC, тогда как native-память требует ручного или частичного управления. Эта структура обеспечивает платформонезависимость Java, но требует тщательной настройки для высоконагруженных систем.

### 1.1. Преимущества

- Автоматическое управление памятью через GC.
- Поддержка многопоточности с отдельными стеками.
- Гибкость настройки через параметры JVM.

### 1.2. Вызовы

- Возможные паузы из-за GC.
- Риск утечек в native-памяти.
- Необходимость мониторинга и оптимизации под конкретные сценарии.

## 2. Основные Области Памяти

### 2.1. Heap (Куча)

#### Описание

Heap — основная область для хранения объектов и массивов, созданных с помощью `new`. Расположена внутри управляемой памяти JVM и динамически расширяется или сжимается.

#### Структура

- **Young Generation**:
    - **Eden**: Новые объекты размещаются здесь при создании.
    - **Survivor S0/S1**: Объекты, пережившие один или несколько циклов GC, перемещаются сюда для дальнейшего тестирования на "долгоживучесть".
- **Old Generation (Tenured)**: Долгоживущие объекты, такие как кэши, singletons или большие коллекции, которые пережили несколько сборок.
- **Humongous Objects**: Объекты размером более 50% региона в G1GC, размещаемые напрямую в Old Generation.

#### Жизненный Цикл Объекта

```java
Object o = new Object(); // Создается в Eden
// Первый Minor GC → перемещается в Survivor S0
// Следующий Minor GC → перемещается в Survivor S1
// После нескольких циклов (например, 15, настраиваемых через -XX:MaxTenuringThreshold) → в Old Generation
```

- Объекты в Old Generation очищаются только во время Full GC.

#### Управление

- `-Xms512m`: Начальный размер Heap.
- `-Xmx2048m`: Максимальный размер Heap.
- `-XX:NewRatio=2`: Соотношение Old/New (2:1, где 2 части — Old, 1 часть — Young).
- `-XX:SurvivorRatio=6`: Соотношение Eden:S0:S1 (6:1:1).
- Пример команды: `java -Xms512m -Xmx2048m -XX:NewRatio=2 -XX:SurvivorRatio=6 MyApp`

### 2.2. Metaspace

#### Описание

Metaspace, введенный в Java 8 вместо PermGen, хранит метаданные классов и расположен в native-памяти (off-heap), вне JVM Heap.

#### Что Хранит

- Метаданные загруженных классов: названия, методы, сигнатуры, аннотации.
- Информация о класс-лоадерах (ClassLoader).
- Частично скомпилированный байт-код и константный пул (Constant Pool).

#### Механизм Работы

- При загрузке класса через ClassLoader JVM выделяет память в Metaspace.
- Память освобождается при выгрузке класса (class unloading), если GC определяет, что ClassLoader больше не нужен.

#### Ошибки

- `java.lang.OutOfMemoryError: Metaspace`: Возникает, если Metaspace не ограничен и исчерпана память ОС.
- Пример: При динамической загрузке большого числа классов (например, в OSGi или Web-контейнерах).

#### Управление

- `-XX:MetaspaceSize=128m`: Начальный размер Metaspace.
- `-XX:MaxMetaspaceSize=512m`: Максимальный размер (по умолчанию неограничен, зависит от памяти ОС).
- Пример команды: `java -XX:MetaspaceSize=128m -XX:MaxMetaspaceSize=512m MyApp`.

#### Отличие от PermGen

- PermGen был частью Heap и имел фиксированный размер (настраивался через `-XX:MaxPermSize`), что приводило к частым `OutOfMemoryError`.
- Metaspace динамически расширяется в native-памяти, устраняя это ограничение.

### 2.3. Что Такое Native Memory (Off-Heap)?

- Это оперативная память, выделяемая JVM через системные вызовы ОС (например, `malloc` или `mmap`), вне управления GC.
- Используется для Metaspace, Direct ByteBuffer, JIT Code Cache и других структур.

#### Сравнение с Heap

|Параметр|Java Heap|Native (Off-Heap) Memory|
|---|---|---|
|Расположение|Внутри JVM|Вне JVM, через ОС|
|Управление|JVM (через GC)|Ручное/частично JVM|
|Содержимое|Java-объекты|Метаданные, буферы, кэши|
|Пример|`new Object()`|`ByteBuffer.allocateDirect()`|

#### Что Размещается в Native Memory

1. Metaspace (метаданные классов).
2. Direct ByteBuffer (`allocateDirect()`).
3. JIT Code Cache (скомпилированный код).
4. Native Stacks (JNI, потоки ОС).
5. Thread data (мониторинг, TLS).
6. Структуры GC (например, ZGC, Shenandoah).

#### Преимущества Использования Off-Heap

- Снижение нагрузки на GC.
- Ускорение операций ввода-вывода (IO, NIO).
- Возможность использования памяти сверх лимита `-Xmx`.

#### Риски

- Память не освобождается автоматически JVM. Например, `ByteBuffer.allocateDirect()` требует ручного управления через `Cleaner` или `Unsafe`.

### 2.4. Java Stack (Стек Вызовов)

#### Описание

Каждый поток имеет собственный стек, выделяемый JVM при создании потока.

#### Что Хранит

- Локальные переменные (примитивы и ссылки на объекты в Heap).
- Параметры методов.
- Ссылки на `this`.
- Стековые кадры (Stack Frames) для каждого вызова метода.

#### Ошибки

- `StackOverflowError`: Возникает при чрезмерной рекурсии или большом стеке вызовов.
- Управление: `-Xss1m` (размер стека на поток, по умолчанию 1 MB).

#### Пример

```java
public void recursive(int depth) {
    if (depth > 1000) return;
    recursive(depth + 1); // Может вызвать StackOverflowError
}
```

### 2.5. Program Counter (PC Register)

#### Описание

Каждый поток имеет собственный PC Register — минимальную область памяти.

#### Что Хранит

- Адрес текущей исполняемой инструкции в байт-коде.
- Обновляется при вызове методов или возврате.

#### Особенности

- Уникален для каждого потока и занимает минимальный объем.

### 2.6. Native Method Stack

#### Описание

Расположена в native-памяти для поддержки вызовов не-Java методов через Java Native Interface (JNI).

#### Что Хранит

- Адреса нативных библиотек (например, `.so` или `.dll`).
- Аргументы и результаты вызовов нативных методов.

#### Пример

```java
public native void nativeCall(); // Требует реализации в C/C++
```

### 2.7. Code Cache

#### Описание

Хранит скомпилированные версии байт-кода, преобразованные JIT-компиллятором, в native-памяти.

#### Назначение

- Ускоряет выполнение часто вызываемых методов (hot methods).
- Поддерживает оптимизации HotSpot JVM.

#### Управление

- `-XX:ReservedCodeCacheSize=128m`: Максимальный размер кэша (по умолчанию 240 MB в Java 11+).

### 2.8. Direct Memory (Off-Heap)

#### Описание

Расположена вне Heap в native-памяти для работы с `ByteBuffer.allocateDirect()` и сторонними библиотеками.

#### Использование

- Сетевые буферы (Netty, Spark, Cassandra).
- `Unsafe.allocateMemory()` для низкоуровневого доступа.

#### Риски

- Требует ручного освобождения, иначе утечки памяти.
- Управление: `-XX:MaxDirectMemorySize=512m` (по умолчанию равно `-Xmx`).

#### Пример

```java
ByteBuffer buffer = ByteBuffer.allocateDirect(1024); // Выделение в Direct Memory
```

## 3. Управление Памятью и Сборщик Мусора (GC)

### 3.1. Алгоритмы и Коллекторы GC

JVM использует разные алгоритмы в зависимости от версии и настроек:

- **Serial GC**: Однопоточный, для небольших приложений (`-XX:+UseSerialGC`).
- **Parallel GC**: Многопоточный, для высокой пропускной способности (`-XX:+UseParallelGC`).
- **CMS (Concurrent Mark-Sweep)**: Минимизирует паузы, устарел в Java 9 (`-XX:+UseConcMarkSweepGC`).
- **G1GC**: Баланс пауз и скорости, по умолчанию в Java 9+ (`-XX:+UseG1GC`).
- **ZGC/Shenandoah**: Минимальные паузы для больших куч (`-XX:+UseZGC`).

#### Сравнение

|Коллектор|Подходит для|Особенности|
|---|---|---|
|Serial GC|Малые приложения, отладка|Простота, одна нить|
|Parallel GC|Высокая пропускная способность|Многопоточность|
|CMS|Предсказуемые паузы|Устарел, сложная настройка|
|G1GC|Средние и большие хипы|Региональная сборка|
|ZGC|Крупные системы|Почти нулевые паузы|

### 3.2. Процесс Работы GC

- **Поиск Живых Объектов**:
    - GC начинает с **GC Roots** (ссылки из стека, статические поля, регистры, JNI).
    - Рекурсивно помечает все доступные объекты в Heap.
- **Очистка**:
    - Удаляет объекты без ссылок (Mark and Sweep).
    - Использует Copying для Young Generation и Mark-Compact для Old Generation.
- **Компактизация**: Устраняет фрагментацию в Old Generation.

#### Роль Стека в GC

- GC не удаляет объекты из стека, так как там хранятся только примитивы и ссылки.
- Ссылки из стека служат корнями (GC Roots) для поиска живых объектов в Heap.

### 3.3. Настройка

- `-XX:MaxGCPauseMillis=200`: Ограничивает максимальную паузу GC.
- `-XX:GCTimeRatio=9`: Соотношение времени GC к рабочему времени (например, 9:1).
- Пример: `java -XX:+UseG1GC -XX:MaxGCPauseMillis=200 MyApp`.

## 4. Загрузка Классов и ClassLoader

### 4.1. Роль ClassLoader

ClassLoader отвечает за поиск, загрузку и инициализацию байт-кода классов. Процесс включает:

1. Поиск `.class` файла (или JAR).
2. Чтение байт-кода.
3. Создание метаданных в Metaspace.
4. Создание объекта `Class<?>` в Heap.

### 4.2. Иерархия ClassLoader

- **Bootstrap ClassLoader** (написан на C++):
    - Загружает основные классы (`java.lang.*`, `java.util.*`).
    - Работает без Java-кода.
- **Extension ClassLoader** (до Java 9):
    - Загружает классы из `lib/ext`.
    - Заменен модульной системой в Java 9.
- **Application ClassLoader**:
    - Загружает классы из `classpath` (ваши `src`, JAR-файлы).
- **Custom ClassLoader**:
    - Используется для динамической загрузки (например, OSGi, плагины).

### 4.3. Принцип Делегирования

ClassLoader делегирует загрузку родителю. Если родитель не находит класс, текущий ClassLoader загружает его самостоятельно.

#### Пример

```java
public class MyClassLoader extends ClassLoader {
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] bytes = loadClassData(name); // Логика загрузки (например, из файла или сети)
        return defineClass(name, bytes, 0, bytes.length);
    }
}
```

#### Использование

```java
ClassLoader loader = MyClass.class.getClassLoader();
System.out.println(loader); // sun.misc.Launcher$AppClassLoader
```

### 4.4. Динамическая Загрузка и Выгрузка

- Custom ClassLoader позволяет перезагружать классы (например, в плагинах).
- Выгрузка классов происходит при удалении ссылок на ClassLoader, что освобождает Metaspace.

## 5. Хранение Статических и Нестатических Элементов

### 5.1. Статические Поля

- **Где Хранятся**: В Metaspace как часть метаданных `Class<?>`.
- **Когда Создаются**: При загрузке класса JVM (обычно при первом обращении).
- **Пример**:
    
    ```java
    public class MyClass {
        static int counter = 0; // Хранится в Metaspace
    }
    ```
    
- **Особенности**: Одна копия на весь класс, удерживает объекты в Heap от GC.

### 5.2. Нестатические Поля

- **Где Хранятся**: В Heap внутри каждого объекта.
- **Когда Создаются**: При создании экземпляра через `new`.
- **Пример**:
    
    ```java
    public class MyClass {
        int value; // Хранится в Heap для каждого объекта
    }
    MyClass obj = new MyClass(); // value выделяется в Heap
    ```
    

### 5.3. Статические Методы

- **Где Хранятся**: В Metaspace как часть метаданных класса.
- **Особенности**: Не участвуют в виртуальном вызове, вызываются напрямую.
- **Пример**:
    
    ```java
    public class MyClass {
        static void log() {
            System.out.println("static log");
        }
    }
    MyClass.log(); // Вызов без экземпляра
    ```
    

### 5.4. Нестатические Методы

- **Где Хранятся**: В Metaspace как часть метаданных класса.
- **Особенности**: Доступны через объект, используют `this` для динамического вызова.
- **Пример**:
    
    ```java
    public class MyClass {
        void print() {
            System.out.println("Hello");
        }
    }
    MyClass obj = new MyClass();
    obj.print(); // Вызов через объект
    ```
    

### 5.5. Статические final Константы

- **Особенности**: Встраиваются в байт-код как компилируемые константы (inlined).
- **Где Хранятся**: Поле в Metaspace, значение вшивается в байт-код.
- **Пример**:
    
    ```java
    public class MyClass {
        public static final int VALUE = 42; // Inlined как `ldc 42`
    }
    System.out.println(MyClass.VALUE); // Значение 42 встраивается в байт-код
    ```
    
- **Важно**: Изменение `VALUE` требует пересборки всех зависящих классов.

#### Сравнение

|Элемент|Где Хранится|Привязан к|Когда Создается|
|---|---|---|---|
|`static` Поле|Metaspace|Класс|При загрузке класса|
|Нестатическое Поле|Heap|Объект|При создании объекта|
|`static` Метод|Metaspace|Класс|При загрузке класса|
|Нестатический Метод|Metaspace|Класс (через объект)|При загрузке класса|
|`static final`|Metaspace + Inlined|Класс|При загрузке класса|

## 6. Объект `Class<?>` и Метаданные

### 6.1. Что Такое `Class<?>`?

- Это объект типа `java.lang.Class`, представляющий метаданные загруженного класса.
- Создается ровно один раз для каждого класса JVM.

### 6.2. Где Хранится?

- **Объект `Class<?>`**: В Heap (например, `MyClass.class`).
- **Метаданные**: В Metaspace (имя, методы, поля, vtable, константный пул).

#### Схема

```
[Native Memory (Metaspace)]
  ┌─────────────────────────────┐
  │ Метаданные класса          │
  │ - Имя класса               │
  │ - Суперкласс               │
  │ - Список методов           │
  │ - Список полей             │
  └──────────────▲─────────────┘
                │
[Heap]
  ┌─────────────────────────────┐
  │ Class<MyClass>             │ ← Объект Class<?>
  └─────────────────────────────┘
```

### 6.3. Процесс Создания

- При первом использовании класса (например, `new MyClass()`):
    1. ClassLoader загружает байт-код.
    2. Создаются метаданные в Metaspace.
    3. Создается объект `Class<?>` в Heap.

### 6.4. Освобождение

- Выгрузка происходит при удалении ссылок на ClassLoader (например, в OSGi).
- Требует Full GC и условия: нет активных экземпляров, ссылок на ClassLoader.

## 7. Наследование и Память

### 7.1. Загрузка Классов

При `new Child()` JVM загружает `Parent` и `Child`:

- Создает метаданные в Metaspace для каждого.
- Связывает `ChildClass` → `ParentClass`.

### 7.2. Хранение Полей

- Все поля (унаследованные и собственные) хранятся в одном объекте в Heap.
- Пример:
    
    ```java
    class Parent {
        int a = 10;
    }
    class Child extends Parent {
        int b = 20;
    }
    Child c = new Child(); // В Heap: a=10, b=20
    ```
    

#### Структура в Heap

```
[Heap]
┌────────────────────────────────┐
│ Объект класса Child           │
│ ┌─────┬─────┐                │
│ │ a=10│ b=20│ ← Поля         │
│ └─────┴─────┘                │
│ Ссылка на Class<Child>        │
└────────────────────────────────┘
```

### 7.3. Таблица Методов (vtable)

- Хранится в Metaspace.
- Обновляется при переопределении методов в `Child`.

## 8. Диагностика и Мониторинг

### 8.1. Инструменты

- `jmap -heap <pid>`: Показывает структуру Heap.
- `jvisualvm` или `Java Mission Control (JMC)`: Визуализация и профилирование.
- `jcmd <pid> GC.class_histogram`: Анализ распределения классов.
- `jmap -clstats <pid>`: Статистика классов.
- `-Xlog:gc*,class+unload`: Логирование GC и выгрузки классов.

### 8.2. Пример Логирования

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class MemoryMonitor {
    private static final Logger logger = LoggerFactory.getLogger(MemoryMonitor.class);

    public static void logMemoryUsage() {
        Runtime runtime = Runtime.getRuntime();
        logger.info("Использовано: {} MB, Свободно: {} MB, Всего: {} MB",
                (runtime.totalMemory() - runtime.freeMemory()) / 1024 / 1024,
                runtime.freeMemory() / 1024 / 1024,
                runtime.totalMemory() / 1024 / 1024);
    }
}
```

- Настройка `log4j.properties`:
    
    ```
    log4j.rootLogger=INFO, console
    log4j.appender.console=org.apache.log4j.ConsoleAppender
    log4j.appender.console.layout=org.apache.log4j.PatternLayout
    log4j.appender.console.layout.ConversionPattern=%d{yyyy-MM-dd HH:mm:ss} %-5p %c{1}:%L - %m%n
    ```
    

### 8.3. Лучшие Практики

- Настройте `-Xms`, `-Xmx`, `-XX:MaxMetaspaceSize` под нагрузку.
- Мониторьте Metaspace в динамических приложениях (OSGi, Tomcat).
- Избегайте утечек в Direct Memory (например, с `ByteBuffer`).

## 9. Проблемы и Решения

### 9.1. Проблемы

- **OutOfMemoryError: Heap**: Переполнение Heap (увеличьте `-Xmx`).
- **OutOfMemoryError: Metaspace**: Утечки ClassLoader (настройте `-XX:MaxMetaspaceSize`).
- **StackOverflowError**: Чрезмерная рекурсия (увеличьте `-Xss`).
- **Утечки Direct Memory**: Неправильное освобождение буферов.

### 9.2. Решения

- Используйте профилировщики (JProfiler, YourKit).
- Регулярно выполняйте Full GC (например, через `System.gc()` — как рекомендацию).
- Анализируйте дампы памяти с `jmap -dump`.
