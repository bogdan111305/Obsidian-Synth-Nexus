# Инициализация объектов с учётом наследования

> [!QUOTE] Суть
> Порядок инициализации при создании объекта: **1) Статические поля/блоки родителя → 2) Статические поля/блоки потомка → 3) Поля/блоки экземпляра родителя → 4) Конструктор родителя → 5) Поля/блоки экземпляра потомка → 6) Конструктор потомка**.

> [!WARNING] Ловушка: вызов переопределяемого метода в конструкторе
> Если конструктор родителя вызывает метод, переопределённый в подклассе, он выполнится до того, как инициализированы поля подкласса. Результат — NPE или некорректное поведение.

## Основы инициализации

### Концепция инициализации при наследовании

Когда создаётся объект подкласса, JVM выполняет инициализацию в строгом порядке, начиная с суперкласса. Это необходимо, так как подкласс зависит от полей и методов суперкласса. Все части объекта (суперкласс и подкласс) хранятся в **едином объекте в куче**, но инициализируются поэтапно.

```java
class Parent {
    String message = "Parent field";

    {
        System.out.println("Parent instance init block");
    }

    public Parent() {
        System.out.println("Parent constructor");
    }
}

class Child extends Parent {
    String message2 = "Child field";

    {
        System.out.println("Child instance init block");
    }

    public Child() {
        System.out.println("Child constructor");
    }
}

public class Main {
    public static void main(String[] args) {
        Child c = new Child();
    }
}
```

```
Parent instance init block
Parent constructor
Child instance init block
Child constructor
```
### Объяснение порядка

- Сначала инициализируются поля и блоки суперкласса (`Parent`).
- Затем вызывается его конструктор.
- После этого инициализируются поля и блоки подкласса (`Child`), а затем его конструктор.
## Полный порядок инициализации

При создании объекта подкласса JVM выполняет следующие шаги:

### 1. Загрузка классов

- JVM загружает класс `Child`, что вызывает загрузку `Parent` (если он ещё не загружен).
- Загрузка выполняется ClassLoader'ом, метаданные классов сохраняются в Metaspace.

### 2. Статическая инициализация суперкласса

- Инициализируются статические поля `Parent`.
- Выполняются статические блоки `Parent`.

### 3. Статическая инициализация подкласса

- Инициализируются статические поля `Child`.
- Выполняются статические блоки `Child`.

### 4. Выделение памяти

- JVM выделяет память в куче для объекта, включая поля суперкласса и подкласса.

### 5. Установка значений по умолчанию

- Поля объекта инициализируются значениями по умолчанию (`0`, `null`, `false`).

### 6. Инициализация суперкласса

- Инициализируются экземплярные поля `Parent`.
- Выполняются экземплярные блоки инициализации `Parent`.
- Вызывается конструктор `Parent` (по умолчанию `super()` добавляется компилятором).

### 7. Инициализация подкласса

- Инициализируются экземплярные поля `Child`.
- Выполняются экземплярные блоки инициализации `Child`.
- Вызывается конструктор `Child`.

### Байт-код (упрощённый, для создания `Child`)

```java
new Child          // Выделение памяти для Child
dup                // Дублирование ссылки
invokespecial Parent.<init>  // Вызов конструктора Parent
invokespecial Child.<init>   // Вызов конструктора Child
astore_1           // Сохранение объекта
```

## Статическая инициализация

Статические поля и блоки инициализируются **один раз** при загрузке класса в JVM и принадлежат классу, а не объекту. Они выполняются в порядке наследования: сначала суперкласс, затем подкласс.

```java
class Parent {
    static String staticField = "Parent static field";
    static {
        System.out.println("Static init block Parent: " + staticField);
    }

    String instanceField = "Parent instance field";
    {
        System.out.println("Instance init block Parent");
    }

    public Parent() {
        System.out.println("Constructor Parent");
    }
}

class Child extends Parent {
    static String staticField2 = "Child static field";
    static {
        System.out.println("Static init block Child: " + staticField2);
    }

    String instanceField2 = "Child instance field";
    {
        System.out.println("Instance init block Child");
    }

    public Child() {
        System.out.println("Constructor Child");
    }
}

public class Main {
    public static void main(String[] args) {
        new Child();
    }
}
```

```
Static init block Parent: Parent static field
Static init block Child: Child static field
Instance init block Parent
Constructor Parent
Instance init block Child
Constructor Child
```

### Особенности

- Статические блоки выполняются только при первой загрузке класса (например, при вызове `new Child()` или обращении к статическому члену).
- Если класс уже загружен, статическая инициализация не повторяется.

## Подводные камни и ошибки

### Вызов переопределяемых методов из конструктора

Вызов методов, которые могут быть переопределены в подклассе, из конструктора суперкласса опасен, так как подкласс ещё не инициализирован.

#### Пример

```java
class Parent {
    int value = 42;

    public Parent() {
        System.out.println("value in Parent: " + value);
        print(); // Плохая практика!
    }

    void print() {
        System.out.println("Parent print()");
    }
}

class Child extends Parent {
    String message = "Hello";

    @Override
    void print() {
        System.out.println("Child print: message = " + message); // message = null!
    }
}

public class Main {
    public static void main(String[] args) {
        new Child();
    }
}
```

#### Вывод

```
value in Parent: 42
Child print: message = null
Instance init block Child
Constructor Child
```

#### Проблема

Поле `message` ещё не инициализировано (`null`), так как конструктор `Parent` вызывается до инициализации `Child`.

#### Решение

- Избегайте вызова переопределяемых методов в конструкторах.
- Используйте финальные или статические методы:
    
    ```java
    final void print() { ... } // Нельзя переопределить
    ```
    

### Отсутствие конструктора без параметров

Если у суперкласса нет конструктора по умолчанию, подкласс должен явно вызывать `super(args)`.

#### Пример

```java
class Parent {
    public Parent(int x) {}
}

class Child extends Parent {
    public Child() {
        super(10); // Обязательный вызов
    }
}
```

#### Ошибка

```java
class Child extends Parent {
    public Child() {} // Ошибка компиляции: нет super()
}
```

#### Решение

Всегда проверяйте конструкторы суперкласса и вызывайте их явно, если требуется.

### Финальные поля

Финальные поля должны быть инициализированы до завершения конструктора, что может усложнить инициализацию в подклассе.

#### Пример

```java
class Parent {
    final int value;

    public Parent() {
        value = 42;
    }
}

class Child extends Parent {
    final String message;

    public Child() {
        message = "Hello"; // Финальные поля инициализируются в конструкторе
    }
}
```

#### Ошибка

```java
class Child extends Parent {
    final String message; // Ошибка: message не инициализировано
}
```

#### Решение

Инициализируйте финальные поля в конструкторе или при объявлении.

## Сложные сценарии

### Многоуровневое наследование

При нескольких уровнях наследования инициализация проходит от самого верхнего суперкласса к подклассу.

#### Пример

```java
class Grandparent {
    { System.out.println("Grandparent init block"); }
    public Grandparent() { System.out.println("Grandparent constructor"); }
}

class Parent extends Grandparent {
    { System.out.println("Parent init block"); }
    public Parent() { System.out.println("Parent constructor"); }
}

class Child extends Parent {
    { System.out.println("Child init block"); }
    public Child() { System.out.println("Child constructor"); }
}
```

#### Вывод

```
Grandparent init block
Grandparent constructor
Parent init block
Parent constructor
Child init block
Child constructor
```

### Интерфейсы и абстрактные классы

Интерфейсы не имеют конструкторов, но абстрактные классы могут их содержать.

#### Пример

```java
abstract class AbstractParent {
    { System.out.println("AbstractParent init block"); }
    public AbstractParent() { System.out.println("AbstractParent constructor"); }
}

class Child extends AbstractParent {
    { System.out.println("Child init block"); }
    public Child() { System.out.println("Child constructor"); }
}
```

#### Вывод

```
AbstractParent init block
AbstractParent constructor
Child init block
Child constructor
```

### Records и Sealed Classes (Java 17+)

- **Records**: Компактные классы для неизменяемых данных, автоматически генерируют конструктор.
- **Sealed Classes**: Ограничивают наследование, но порядок инициализации остаётся тем же.

#### Пример с `record`

```java
record Parent(String name) {
    public Parent {
        System.out.println("Parent record constructor: " + name);
    }
}

class Child extends Parent {
    String extra;

    public Child(String name, String extra) {
        super(name);
        this.extra = extra;
        System.out.println("Child constructor: " + extra);
    }
}
```

#### Вывод

```
Parent record constructor: Name
Child constructor: Extra
```

#### Пример с `sealed`

```java
sealed class Parent permits Child {
    { System.out.println("Parent init block"); }
    public Parent() { System.out.println("Parent constructor"); }
}

final class Child extends Parent {
    { System.out.println("Child init block"); }
    public Child() { System.out.println("Child constructor"); }
}
```

#### Вывод

Аналогичен стандартному наследованию.

## JVM-реализация

### Загрузка классов

- JVM использует `ClassLoader` для загрузки `Parent` и `Child` в Metaspace.
- Статические поля и блоки инициализируются через `<clinit>` (class initializer).

### Создание объекта

- Инструкция `new` выделяет память в куче.
- Поля инициализируются значениями по умолчанию.
- Конструкторы вызываются через `invokespecial`.

### Байт-код (для `new Child()`)

```java
new Child
dup
invokespecial Parent.<init>
invokespecial Child.<init>
astore_1
```

- `new`: Выделяет память.
- `invokespecial`: Вызывает конструкторы от `Parent` к `Child`.

### Метод `<clinit>` (для статической инициализации)

```java
// Для Parent
<clinit>:
  ldc "Parent static field"
  putstatic Parent.staticField
  getstatic System.out
  ldc "Static init block Parent"
  invokevirtual PrintStream.println
  return
```

## Производительность и оптимизация

### Статическая инициализация

- Выполняется один раз при загрузке класса.
- Может замедлить старт приложения, если содержит тяжёлые операции (например, чтение файлов).

### Экземплярная инициализация

- Затраты зависят от количества полей, блоков и уровней наследования.
- Вызов конструкторов суперклассов увеличивает время создания объекта.

### Рекомендации

- Минимизируйте логику в статических блоках.
- Избегайте сложных вычислений в конструкторах.
- Используйте ленивую инициализацию для тяжёлых ресурсов:
    
    ```java
    class Parent {
        static class LazyHolder {
            static final Object RESOURCE = heavyInit();
        }
    }
    ```
    

## Лучшие практики

### Избегайте переопределяемых методов в конструкторах

```java
class Parent {
    final void print() { ... } // Безопасно
}
```

### Явно вызывайте `super()` при необходимости

```java
class Child extends Parent {
    public Child() {
        super(42); // Явный вызов
    }
}
```

### Используйте финальные поля для неизменяемых данных

```java
final String message = "Immutable";
```

### Минимизируйте статические блоки

- Переносите логику в методы или ленивую инициализацию.

### Используйте `record` для простых данных (Java 17+)

```java
record Parent(String name) {}
```

### Проверяйте иерархию наследования

- Используйте `sealed` классы для контроля подклассов.

### Тестируйте инициализацию

- Используйте JUnit для проверки порядка инициализации:
    
    ```java
    @Test
    void testChildInitialization() {
        ByteArrayOutputStream out = new ByteArrayOutputStream();
        System.setOut(new PrintStream(out));
        new Child();
        assertTrue(out.toString().contains("Parent constructor"));
    }
    ```
