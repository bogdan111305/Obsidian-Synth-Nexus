# Наследование в Java

> [!QUOTE] Суть
> **Наследование** — механизм ООП, при котором дочерний класс (`extends`) получает поля и методы родителя. Отношение **"is-a"**. Java поддерживает только одиночное наследование классов. `super()` должен быть **первой** строкой конструктора.

## Основы наследования

### Основные преимущества

- **Переиспользование кода**: Общие поля и методы определяются в базовом классе.
- **Расширение поведения**: Дочерние классы могут добавлять новые методы или переопределять существующие.
- **Полиморфизм**: Объекты дочерних классов могут использоваться через ссылки родительского типа.

**Пример**:

```java
public class Animal {
    protected String name;

    public Animal(String name) {
        this.name = name;
    }

    public void makeSound() {
        System.out.println("Какой-то звук");
    }
}

public class Dog extends Animal {
    public Dog(String name) {
        super(name);
    }

    public void bark() {
        System.out.println("Гав!");
    }
}
```

**Использование**:

```java
Dog dog = new Dog("Бобик");
dog.makeSound(); // Выведет: Какой-то звук (унаследованный метод)
dog.bark();      // Выведет: Гав! (собственный метод)
```

## 2. Синтаксис наследования

- Используется ключевое слово `extends` для указания родительского класса.
- Каждый класс в Java неявно наследует `java.lang.Object`, если не указан другой родитель.

**Пример**:

```java
public class Animal {
    protected String name;

    public Animal(String name) {
        this.name = name;
    }

    public void eat() {
        System.out.println(name + " ест");
    }
}

public class Cat extends Animal {
    public Cat(String name) {
        super(name);
    }

    public void meow() {
        System.out.println(name + ": Мяу!");
    }
}
```

**Использование**:

```java
Cat cat = new Cat("Мурка");
cat.eat();   // Выведет: Мурка ест
cat.meow();  // Выведет: Мурка: Мяу!
```

## 3. Переопределение методов (Overriding)

Переопределение методов — это механизм, при котором дочерний класс предоставляет собственную реализацию метода родительского класса, сохраняя его сигнатуру.

### 3.1. Основные правила переопределения

1. **Совпадение сигнатуры**:
    - Имя метода, параметры и их типы должны быть идентичны.
2. **Возвращаемый тип**:
    - Допускается ковариантный возвращаемый тип (подтип оригинального типа).
3. **Модификаторы доступа**:
    - Нельзя уменьшать видимость (например, `public` → `protected`).
4. **Исключения**:
    - Переопределённый метод не может бросать новые или более общие проверяемые исключения.
5. **Аннотация `@Override`**:
    - Рекомендуется для проверки корректности переопределения.
6. **Ограничения**:
    - `final` методы нельзя переопределять.
    - `static` методы не переопределяются, а **скрываются** (method hiding).

**Пример**:

```java
public class Animal {
    protected String name;

    public Animal(String name) {
        this.name = name;
    }

    public void makeSound() {
        System.out.println(name + ": Какой-то звук");
    }
}

public class Dog extends Animal {
    public Dog(String name) {
        super(name);
    }

    @Override
    public void makeSound() {
        System.out.println(name + ": Гав!");
    }
}
```

**Использование**:

```java
Animal animal = new Dog("Бобик");
animal.makeSound(); // Выведет: Бобик: Гав!
```

### 3.2. Ковариантный возвращаемый тип

- Дочерний класс может возвращать подтип оригинального типа.

**Пример**:

```java
public class Animal {
    public Animal getInstance() {
        return this;
    }
}

public class Dog extends Animal {
    @Override
    public Dog getInstance() { // Ковариантный тип
        return this;
    }
}
```

### 3.3. Method Hiding (скрытие статических методов)

> [!WARNING] Ловушка: статические методы не переопределяются
> `Parent p = new Child(); p.greet()` — вызовет **Parent.greet()**, а не Child. Статика привязана к типу ссылки, а не к объекту. Это ломает ожидаемый полиморфизм.

- Статические методы не переопределяются, а скрываются в дочернем классе.
- Вызов зависит от типа ссылки, а не объекта.

**Пример**:

```java
public class Parent {
    public static void greet() {
        System.out.println("Привет от Parent");
    }
}

public class Child extends Parent {
    public static void greet() { // Скрытие, не переопределение
        System.out.println("Привет от Child");
    }
}
```

**Использование**:

```java
Parent parent = new Child();
Parent.greet(); // Выведет: Привет от Parent
Child.greet();  // Выведет: Привет от Child
```

## 4. Конструкторы и наследование

- **Конструкторы не наследуются**: Дочерний класс должен определять свои конструкторы.
- **Вызов родительского конструктора**: Используется `super(...)` в начале конструктора дочернего класса.
- Если `super(...)` не указан, компилятор неявно вызывает конструктор родителя без параметров.

**Пример**:

```java
public class Animal {
    protected String name;

    public Animal(String name) {
        this.name = name;
        System.out.println("Конструктор Animal: " + name);
    }
}

public class Dog extends Animal {
    public Dog(String name) {
        super(name); // Вызов конструктора родителя
        System.out.println("Конструктор Dog: " + name);
    }
}
```

**Использование**:

```java
Dog dog = new Dog("Бобик");
// Выведет:
// Конструктор Animal: Бобик
// Конструктор Dog: Бобик
```

> [!WARNING] Ловушка: порядок вызова конструкторов
> Если у родителя **нет** конструктора без параметров, дочерний класс **обязан** явно вызвать `super(args)`. Иначе — ошибка компиляции. `super()` всегда должен быть **первой строкой** конструктора.

**Подводные камни**:

- Если родительский класс не имеет конструктора без параметров, дочерний класс обязан явно вызвать `super(...)`.
- Неправильный вызов `super` приводит к ошибке компиляции.

## 5. Ключевое слово `super`

- **`super.method()`**: Вызывает метод родительского класса.
- **`super(...)`**: Вызывает конструктор родительского класса.
- Используется для доступа к унаследованным членам или для дополнения их поведения.

**Пример**:

```java
public class Animal {
    protected String name;

    public Animal(String name) {
        this.name = name;
    }

    public void makeSound() {
        System.out.println(name + ": Какой-то звук");
    }
}

public class Dog extends Animal {
    public Dog(String name) {
        super(name);
    }

    @Override
    public void makeSound() {
        super.makeSound(); // Вызов родительского метода
        System.out.println(name + ": Гав!");
    }
}
```

**Использование**:

```java
Dog dog = new Dog("Бобик");
dog.makeSound();
// Выведет:
// Бобик: Какой-то звук
// Бобик: Гав!
```

## 6. Ограничения наследования

1. **Одиночное наследование**:
    
    - Java не поддерживает множественное наследование классов (класс может наследовать только один класс).
    - Решение: Используйте интерфейсы для реализации множественного поведения.
2. **Final-классы**:
    
    - Классы, помеченные `final`, нельзя наследовать:
        
        ```java
        public final class FinalClass {}
        // Ошибка: class SubClass extends FinalClass {}
        ```
        
3. **Final-методы**:
    
    - Методы, помеченные `final`, нельзя переопределять.
4. **Абстрактные классы**:
    
    - Нельзя создавать объекты абстрактных классов, но они могут быть унаследованы.

## 7. Внутренняя реализация в JVM

- **Наследование**:
    - Дочерний класс содержит ссылку на родительский класс в метаданных.
    - Поля и методы родителя включаются в таблицу виртуальных методов (vtable) дочернего класса.
- **Переопределение**:
    - Переопределённые методы заменяют записи в vtable.
    - Вызовы через `invokevirtual` разрешаются на основе фактического типа объекта.
- **Конструкторы**:
    - Компилируются в метод `<init>`, вызывающий `<init>` родителя через `invokespecial`.
- **Super**:
    - `super.method()` вызывает метод родителя через `invokevirtual` или `invokespecial`.
    - `super(...)` вызывает конструктор родителя через `invokespecial`.

**Пример байт-кода (упрощённо)**:

```java
public class Animal {
    public void makeSound() { ... }
}
public class Dog extends Animal {
    public void makeSound() { ... }
}
```

Байт-код для `Dog`:

```
.class public Dog
  .super Animal
  .method public makeSound()V
    getstatic java/lang/System/out Ljava/io/PrintStream;
    ldc "Гав!"
    invokevirtual java/io/PrintStream/println(Ljava/lang/String;)V
    return
  .end method
```

- **Производительность**:
    - Полиморфные вызовы: O(1) с JIT-оптимизациями (инлайн-кэширование).
    - Доступ к полям: O(1), но может быть медленнее при глубоком наследовании.
    - Создание объектов: Зависит от цепочки конструкторов.

## 8. Производительность

- **Создание объектов**:
    - Вызов конструкторов родительского и дочернего классов добавляет минимальные накладные расходы.
    - JIT-компилятор оптимизирует цепочки вызовов `super(...)`.
- **Вызов методов**:
    - Переопределённые методы: O(1) через vtable.
    - Скрытие статических методов: O(1), разрешается по типу ссылки.
- **Память**:
    - Дочерний класс хранит поля родителя и свои собственные.
    - Глубокие иерархии увеличивают размер объекта и сложность оптимизации.

**Рекомендация**:

- Минимизируйте глубину наследования (2–3 уровня).
- Используйте `final` для методов, не требующих переопределения, чтобы упростить JIT-оптимизации.

## 9. Многопоточность

Наследование само по себе не обеспечивает потокобезопасность, но требует внимания:

- **Поля**:
    - Унаследованные поля требуют синхронизации при доступе из разных потоков:
        
        ```java
        public class Animal {
            protected volatile String name; // Потокобезопасное поле
        }
        ```
        
- **Методы**:
    - Переопределённые методы должны быть потокобезопасными, если вызываются в многопоточной среде:
        
        ```java
        public class Dog extends Animal {
            @Override
            public synchronized void makeSound() {
                System.out.println(name + ": Гав!");
            }
        }
        ```
        
- **Конструкторы**:
    - Безопасны, если не публикуют объект до завершения инициализации.
    - **Подводный камень**: Публикация `this` в конструкторе может привести к доступу к неинициализированному объекту.

**Пример потокобезопасности**:

```java
public class Animal {
    protected volatile String name;

    public Animal(String name) {
        this.name = name;
    }
}

public class Dog extends Animal {
    private final Object lock = new Object();

    public Dog(String name) {
        super(name);
    }

    @Override
    public void makeSound() {
        synchronized (lock) {
            System.out.println(name + ": Гав!");
        }
    }
}
```

**Рекомендация**: Используйте `synchronized`, `volatile`, или потокобезопасные структуры (`ConcurrentHashMap`, `AtomicReference`) для защиты состояния.

## 10. Сравнение с альтернативами

### 10.1. Наследование vs Композиция

- **Наследование**:
    - Отношение "является" (is-a).
    - Жесткая связь между классами.
    - Пример: `Dog` является `Animal`.
- **Композиция**:
    - Отношение "имеет" (has-a).
    - Гибкость в изменении поведения.
    - Пример: `Car` имеет `Engine`.

**Пример композиции**:

```java
public class AnimalBehavior {
    public void makeSound(String name) {
        System.out.println(name + ": Какой-то звук");
    }
}

public class Dog {
    private final AnimalBehavior behavior = new AnimalBehavior();
    private final String name;

    public Dog(String name) {
        this.name = name;
    }

    public void makeSound() {
        behavior.makeSound(name);
        System.out.println(name + ": Гав!");
    }
}
```

**Рекомендация**: Предпочитайте композицию, если нет явного "is-a" отношения, чтобы избежать избыточной зависимости.

### 10.2. Наследование vs Интерфейсы

- **Наследование**:
    - Подходит для общего состояния и реализации.
    - Ограничено одиночным наследованием.
- **Интерфейсы**:
    - Определяют контракт без реализации (или с `default` методами).
    - Поддерживают множественное наследование.

**Пример с интерфейсом**:

```java
public interface Soundable {
    void makeSound();
}

public class Dog implements Soundable {
    private final String name;

    public Dog(String name) {
        this.name = name;
    }

    @Override
    public void makeSound() {
        System.out.println(name + ": Гав!");
    }
}
```

## 12. Подводные камни

1. **Глубокое наследование**:
    
    - Многоуровневые иерархии усложняют код и оптимизацию.
    - **Решение**: Ограничивайте глубину (2–3 уровня).
2. **Неправильное использование `super`**:
    
    - Вызов `super` не в начале конструктора приводит к ошибке компиляции.
    - **Решение**: Всегда вызывайте `super(...)` первой строкой.
3. **Скрытие статических методов**:
    
    - Вызов зависит от типа ссылки, а не объекта.
    - **Решение**: Используйте статические методы осторожно или заменяйте их на экземплярные.
4. **Множественное наследование**:
    
    - Отсутствие поддержки в Java может усложнить дизайн.
    - **Решение**: Используйте интерфейсы или композицию.
5. **Потокобезопасность**:
    
    - Унаследованные поля могут вызвать состояние гонки.
    - **Решение**: Синхронизируйте доступ или используйте неизменяемые поля.

## 13. Лучшие практики

1. **Минимизируйте наследование**:
    
    - Используйте композицию или интерфейсы, если возможно.
2. **Используйте `@Override`**:
    
    - Гарантирует корректность переопределения:
        
        ```java
        @Override
        public void makeSound() { ... }
        ```
        
3. **Применяйте `final` для защиты**:
    
    - Помечайте классы и методы `final`, если они не должны наследоваться или переопределяться.
4. **Используйте sealed-классы**:
    
    - Ограничивайте иерархию для предсказуемости:
        
        ```java
        public sealed class Animal permits Dog, Cat { ... }
        ```
        
5. **Обеспечивайте потокобезопасность**:
    
    - Синхронизируйте доступ к полям:
        
        ```java
        public class Dog extends Animal {
            private final Object lock = new Object();
            @Override
            public void makeSound() {
                synchronized (lock) { System.out.println(name + ": Гав!"); }
            }
        }
        ```
        
6. **Документируйте методы**:
    
    ```java
    /**
     * Издаёт звук животного.
     */
    public void makeSound();
    ```
    

## 14. Пример: Комплексное использование

```java
import java.util.*;
import java.util.concurrent.ConcurrentHashMap;
import java.util.logging.Logger;

// Интерфейс
public interface Eatable {
    void eat();
}

// Sealed-класс
public sealed class Animal implements Eatable permits Dog, Cat {
    protected final String name;
    private static final Logger LOGGER = Logger.getLogger(Animal.class.getName());
    private static int animalCount;

    static {
        animalCount = 0;
        LOGGER.info("Инициализация Animal: animalCount = 0");
    }

    public Animal(String name) {
        this.name = name;
        animalCount++;
    }

    public void makeSound() {
        System.out.println(name + ": Какой-то звук");
    }

    @Override
    public void eat() {
        System.out.println(name + " ест");
    }

    public static int getAnimalCount() {
        return animalCount;
    }
}

// Дочерний класс 1
public final class Dog extends Animal {
    private final Object lock = new Object();

    public Dog(String name) {
        super(name);
    }

    @Override
    public void makeSound() {
        synchronized (lock) {
            super.makeSound();
            System.out.println(name + ": Гав!");
        }
    }
}

// Дочерний класс 2
public final class Cat extends Animal {
    private final Object lock = new Object();

    public Cat(String name) {
        super(name);
    }

    @Override
    public void makeSound() {
        synchronized (lock) {
            System.out.println(name + ": Мяу!");
        }
    }

    @Override
    public void eat() {
        synchronized (lock) {
            System.out.println(name + ": Ест рыбу");
        }
    }
}

// Демонстрация
public class InheritanceExample {
    private static final Logger LOGGER = Logger.getLogger(InheritanceExample.class.getName());

    public static void main(String[] args) {
        // Полиморфизм
        List<Animal> animals = new ArrayList<>();
        animals.add(new Dog("Бобик"));
        animals.add(new Cat("Мурка"));

        // Перебор с pattern matching
        animals.forEach(animal -> {
            switch (animal) {
                case Dog dog -> {
                    LOGGER.info("Собака: " + dog.name);
                    dog.makeSound();
                    dog.eat();
                }
                case Cat cat -> {
                    LOGGER.info("Кот: " + cat.name);
                    cat.makeSound();
                    cat.eat();
                }
            }
        });

        // Многопоточность
        ConcurrentHashMap<String, Animal> animalMap = new ConcurrentHashMap<>();
        animalMap.put("dog", new Dog("Шарик"));
        new Thread(() -> animalMap.put("cat", new Cat("Барсик"))).start();

        // Потокобезопасный перебор
        new Thread(() -> {
            animalMap.forEach((key, animal) -> {
                LOGGER.info("Потокобезопасный вызов: " + key);
                animal.makeSound();
            });
        }).start();

        // Статическое поле
        LOGGER.info("Всего животных: " + Animal.getAnimalCount());
    }
}
```

```

Бобик ест
INFO: Кот: Мурка
Мурка: Мяу!
Мурка: Ест рыбу

```

## Senior Insights

### S1. invokespecial vs invokevirtual — почему super.method() работает

`invokevirtual` — динамический dispatch через vtable (фактический тип объекта). `invokespecial` — **статический** dispatch: всегда вызывает метод конкретного класса.

```java
class Animal {
    void makeSound() { System.out.println("Sound"); }
}
class Dog extends Animal {
    @Override
    void makeSound() {
        super.makeSound();  // ← invokespecial Animal.makeSound
        System.out.println("Bark");
    }
}
```

**Байткод Dog.makeSound():**
```
// super.makeSound() → invokespecial (не invokevirtual!)
invokespecial Animal.makeSound()V

// System.out.println("Bark") → invokevirtual
getstatic java/lang/System.out Ljava/io/PrintStream;
ldc "Bark"
invokevirtual PrintStream.println(Ljava/lang/String;)V
```

**Почему `invokespecial`, а не `invokevirtual`?**
Если бы `super.makeSound()` использовал `invokevirtual`, JVM снова нашла бы `Dog.makeSound()` в vtable → бесконечная рекурсия. `invokespecial` обходит vtable и вызывает метод напрямую по ссылке на класс.

**`invokespecial` используется для:**
```
1. super.method()         — вызов родительского метода
2. private методы         — не виртуальные (нет в vtable подклассов)
3. <init> конструкторы    — new Dog() → invokespecial Dog.<init>
```

### S2. Конструктор в байткоде: <init> + invokespecial

```java
class Animal {
    String name;
    Animal(String name) { this.name = name; }
}

class Dog extends Animal {
    int age;
    Dog(String name, int age) {
        super(name);  // → invokespecial Animal.<init>
        this.age = age;
    }
}
```

**Байткод Dog.<init>:**
```
// super(name) → вызов Animal.<init>
aload_0          // this
aload_1          // name
invokespecial Animal.<init>(Ljava/lang/String;)V

// this.age = age
aload_0          // this
iload_2          // age
putfield Dog.age I
return
```

`<init>` — это синтетическое имя конструктора в JVM. При `new Dog(...)` выделяется память → вызывается `Dog.<init>` → который вызывает `Animal.<init>` → который вызывает `Object.<init>`.

### S3. Ковариантный return type — компилятор генерирует bridge method

Ковариантный тип — компилятор добавляет синтетический bridge-метод для обратной совместимости с типостёртым кодом:

```java
class Animal {
    Animal getInstance() { return this; }
}
class Dog extends Animal {
    @Override
    Dog getInstance() { return this; }  // ковариантный тип
}
```

**Что генерирует javac в байткоде Dog:**
```
// Реальный метод:
public Dog getInstance()  → return this;

// Синтетический bridge (ACC_BRIDGE + ACC_SYNTHETIC):
public synthetic bridge Animal getInstance() {
    return this.getInstance();  // вызывает Dog.getInstance()
}
```

Bridge нужен потому что JVM различает методы по сигнатуре с возвращаемым типом. Клиент, вызывающий через ссылку `Animal a`, видит `Animal getInstance()` и получает bridge, который делегирует в `Dog getInstance()`.

Проверить: `javap -v -p Dog.class` → увидишь два метода с `ACC_BRIDGE`.

### S4. Смертельная ловушка: вызов виртуального метода из конструктора

```java
class Animal {
    Animal() {
        makeSound(); // ← вызывает переопределённый метод подкласса!
    }
    void makeSound() { System.out.println("..."); }
}

class Dog extends Animal {
    String sound = "Bark"; // инициализируется ПОСЛЕ super()!

    Dog() {
        super(); // вызывается ПЕРВЫМ
    }

    @Override
    void makeSound() {
        System.out.println(sound); // sound == null здесь!
    }
}

new Dog(); // Выводит: null  ← СЮРПРИЗ
```

**Порядок выполнения:**
```
1. new Dog() → выделить память (все поля = default: null/0/false)
2. invokespecial Dog.<init>
3.   → super() → Animal.<init>
4.       → makeSound() → invokevirtual → находит Dog.makeSound!
5.       → Dog.makeSound: sound == null (ещё не инициализирован!)
6.   → this.sound = "Bark"  ← слишком поздно
```

> [!WARNING] Правило: никогда не вызывай виртуальные методы из конструктора
> Если метод может быть переопределён — его вызов в конструкторе родителя увидит объект подкласса в **частично инициализированном** состоянии. Все поля подкласса ещё `null`/`0`. Это одна из самых коварных ловушек в Java.

**Решение:** использовать `private` или `final` методы в конструкторе — они не виртуальные → JVM использует `invokespecial` → точно вызовет нужный метод.

### S5. sealed + pattern matching → JIT devirtualization

```java
sealed class Shape permits Circle, Square, Triangle {}

// JIT знает: Shape — только Circle, Square, или Triangle
// → деоптимизация при добавлении нового типа невозможна (sealed!)
// → JIT может инлайнить все 3 ветки и убрать vtable dispatch

double area(Shape s) {
    return switch (s) {           // exhaustive — компилятор проверяет
        case Circle c    -> Math.PI * c.r() * c.r();
        case Square sq   -> sq.side() * sq.side();
        case Triangle t  -> t.base() * t.height() / 2.0;
    };
}
```

С `sealed` JIT видит **закрытое множество** реализаций → может девиртуализировать вызов `area()` и инлайнить все ветки. С обычным `abstract` JIT делает то же самое, но только пока не появится новый подтип → после деоптимизация.
