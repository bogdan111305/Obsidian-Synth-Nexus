# Производительность коллекций в Java

> [!QUOTE] Суть
> Выбор коллекции = выбор алгоритма. `ArrayList` выигрывает у `LinkedList` в большинстве сценариев за счёт **cache locality** (непрерывная память). `HashMap` O(1) amortized, но деградирует до O(n) при плохом `hashCode()`. `TreeMap`/`TreeSet` — O(log n) всегда.

Производительность коллекций критически важна для эффективности Java-приложений. Эта статья рассматривает сложность операций, внутренние механизмы и практические рекомендации по оптимизации работы с коллекциями.

#### ArrayList
```java
// Внутренняя структура
transient Object[] elementData;
private int size;

// Амортизированная сложность добавления
public boolean add(E e) {
    ensureCapacityInternal(size + 1);
    elementData[size++] = e;
    return true;
}

// Расширение массива
private void grow(int minCapacity) {
    int oldCapacity = elementData.length;
    int newCapacity = oldCapacity + (oldCapacity >> 1); // +50%
    elementData = Arrays.copyOf(elementData, newCapacity);
}
```

```java
    Node<E> next;
    Node<E> prev;

// Быстрое добавление в начало/конец

    linkFirst(e);

}
```

#### HashMap
```java

transient Node<K,V>[] table;
transient int size;

// Хэширование и разрешение коллизий
static final int hash(Object key) {
    int h;
```

```java
public class PerformanceMetrics {
    private static final Map<String, Long> operationTimes = new HashMap<>();
    
    public static void measureTime(String operation, Runnable task) {
        long startTime = System.nanoTime();
        task.run();
        long endTime = System.nanoTime();
        
        long duration = endTime - startTime;
        operationTimes.merge(operation, duration, Long::sum);
        
        System.out.printf("%s: %d ns%n", operation, duration);

        operationTimes.forEach((operation, totalTime) -> {

                operation, totalTime, totalTime / 1000);
        });
    }
}
```

```java
// Алгоритм: Поиск дубликатов
// ❌ Плохо - O(n²)
public boolean hasDuplicates(List<String> list) {
    for (int i = 0; i < list.size(); i++) {
        for (int j = i + 1; j < list.size(); j++) {
            if (list.get(i).equals(list.get(j))) {
                return true;
            }
        }
    }
    return false;
}

// ✅ Хорошо - O(n)
public boolean hasDuplicates(List<String> list) {
    Set<String> seen = new HashSet<>();
    for (String item : list) {
        if (!seen.add(item)) {
            return true;
        }
    }
    return false;
}
```

## Senior Insights

### Cache Locality: почему ArrayList быстрее LinkedList на практике

```java
// BAD: LinkedList — каждый узел отдельный heap объект
LinkedList<Integer> linked = new LinkedList<>();
for (int i = 0; i < 1_000_000; i++) linked.add(i);

// Итерация: каждый следующий элемент → НОВЫЙ heap объект
// CPU cache line = 64 байт, Integer объект ~16 байт
// LinkedList: узлы разбросаны в heap → cache miss на КАЖДОМ шаге!
// L1 cache miss: ~4 cycles, L2: ~12, L3: ~40, RAM: ~200 cycles

for (Integer x : linked) {  // ~5x МЕДЛЕННЕЕ из-за cache misses!
    sum += x;
}

// SENIOR WAY: ArrayList — элементы рядом в памяти
ArrayList<Integer> array = new ArrayList<>(1_000_000);
for (int i = 0; i < 1_000_000; i++) array.add(i);

// Итерация: Object[] внутри ArrayList → последовательный доступ
// CPU prefetcher предсказывает следующие адреса → L1 cache hits!
for (Integer x : array) {   // cache-friendly
    sum += x;
}

// ЕЩЁ ЛУЧШЕ: int[] напрямую (нет boxing!)
int[] primitive = new int[1_000_000]; // 4MB vs ~24MB для ArrayList<Integer>
// Итерация: pure sequential memory → SIMD vectorization JIT!

// Вывод: LinkedList быстрее ArrayList ТОЛЬКО для:
// - Частые вставки/удаления В НАЧАЛО (O(1) vs O(n))
// - И вы держите Iterator в этой позиции
// В остальных случаях — ArrayList или ArrayDeque
```

### @Contended для volatile полей в высоконагруженных структурах

```java
// Проблема false sharing: несколько volatile полей в одной cache line
class BadConcurrentCounter {
    volatile long reads = 0;   // байты 0-7
    volatile long writes = 0;  // байты 8-15
    // Обе переменные в ОДНОЙ cache line (64 байт)!

    // Thread-1 пишет reads → инвалидирует cache line
    // Thread-2 хочет reads writes → cache miss!
    // При 16 ядрах: ConcurrentHashMap теряет 80% производительности из-за этого!
}

// SENIOR WAY: @Contended добавляет padding (128 байт = 2 cache lines)
import jdk.internal.vm.annotation.Contended;

class GoodConcurrentCounter {
    @Contended volatile long reads = 0;   // своя cache line + padding
    @Contended volatile long writes = 0;  // своя cache line + padding

    // Требует запуска с: -XX:-RestrictContended
    // Или через --add-opens java.base/jdk.internal.vm.annotation=ALL-UNNAMED
}

// Реальный пример: LongAdder (JDK источник):
// @Contended static final class Cell {
//     volatile long value;
// }
// LongAdder.add() пишет в разные Cell объекты для разных потоков
// Каждый Cell на своей cache line → нет false sharing → линейная масштабируемость!

// LongAdder vs AtomicLong при 16 потоках:
// AtomicLong.incrementAndGet(): ~100ns (contention → spin)
// LongAdder.increment():         ~10ns  (каждый поток в свою ячейку)
```

### HashMap внутренности: Treeification и loadFactor

```java
// HashMap в Java 8+: при коллизиях использует дерево вместо списка
// Порог для дерева: TREEIFY_THRESHOLD = 8 узлов в одном bucket
// Порог обратно в список: UNTREEIFY_THRESHOLD = 6

// Почему это важно:
// Плохой hashCode(): все элементы в один bucket → O(n) → O(log n) с деревом
// Хороший hashCode(): равномерное распределение → O(1)

// BAD: hashCode всегда 1
class BadKey {
    int id;
    @Override public int hashCode() { return 1; } // ВСЕ в один bucket!
    @Override public boolean equals(Object o) { return ((BadKey)o).id == id; }
}
// HashMap<BadKey, String> с 100 элементами → linked list из 100 → O(n) lookup!

// SENIOR WAY: качественный hashCode
class GoodKey {
    int id;
    String name;

    @Override
    public int hashCode() {
        // Objects.hash использует 31 (prime) для распределения битов
        return Objects.hash(id, name);
        // Или вручную (эффективнее, нет varargs):
        // int result = 31 * id + (name != null ? name.hashCode() : 0);
    }
}

// LoadFactor = 0.75 (default) — компромисс space vs time
// При заполнении > 75% capacity → rehash (O(n) операция!)
// Если знаете размер заранее — инициализируйте правильно:
Map<String, Data> map = new HashMap<>(expectedSize * 4 / 3 + 1);
// Избегает первого rehash!
```

### Senior Interview Q&A

**Q1: Почему ConcurrentHashMap быстрее `Collections.synchronizedMap(HashMap)` при чтении?**

> `Collections.synchronizedMap()` оборачивает **все** операции в `synchronized(mutex)` — при любом `get()` захватывается lock на весь map. При N читающих потоках все ждут очереди. `ConcurrentHashMap` (Java 8+): чтение (`get()`) — **вообще без lock** (volatile reads). Запись (`put()`) — lock только на конкретный bucket (bin). При 64 bucket-ах и 64 пишущих потоках — каждый держит свой независимый lock. Итог: `ConcurrentHashMap.get()` = `volatile` read = почти бесплатно. При 16 читающих потоках разница может быть 16x.

**Q2: Как ArrayList resize происходит и как его избежать?**

> При добавлении элемента в полный ArrayList: (1) создаётся новый массив размером `oldCapacity * 1.5` (точнее: `oldCapacity + oldCapacity >> 1`); (2) `Arrays.copyOf()` — memmove всего массива в новый; (3) старый массив становится мусором для GC. Это O(n) операция! При добавлении 1M элементов без presize: ~log1.5(1M) ≈ 34 resize операции, суммарно скопировано ~2M элементов. Избежание: `new ArrayList<>(expectedSize)` или `ensureCapacity(n)`. Для bulk добавлений: `Collections.addAll()` или `addAll()` с Collection — оптимизировано для одного resize.

**Q3: Что такое EnumMap и когда он предпочтительнее HashMap?**

> `EnumMap<K extends Enum<K>, V>` внутри хранит простой `Object[]` размером `Enum.values().length`. Нет hash вычислений, нет collision handling — `ordinal()` enum напрямую является индексом массива. Производительность: `get()`/`put()` = O(1) без hash overhead, итерация = простой array sweep (cache-friendly). Память: нет Entry объектов, только один массив. Когда использовать: ключи — Enum константы (статусы, типы, дни недели), любые enum-keyed lookup таблицы. Пример: `Map<OrderStatus, List<Order>> ordersByStatus = new EnumMap<>(OrderStatus.class)` — значительно эффективнее `HashMap<OrderStatus, ...>`.
