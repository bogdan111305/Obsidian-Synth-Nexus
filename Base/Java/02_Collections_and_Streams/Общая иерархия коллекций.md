# Общая иерархия коллекций Java

> Два корневых интерфейса: **Collection** (List, Set, Queue/Deque) и **Map** (ключ-значение). Java 21 добавил **SequencedCollection** для единообразного доступа к первому/последнему элементу. Большинство реализаций — не потокобезопасны.
> На интервью: чем Deque лучше Stack, почему Hashtable/Vector устарели, когда ConcurrentHashMap vs synchronizedMap, как `ConcurrentModificationException` связан с `modCount`.

## Иерархия Collection

```
Iterable<E>
└── Collection<E>
    ├── List<E>                     ← упорядочен, дублики разрешены
    │   ├── ArrayList<E>            ← динамический массив, O(1) get
    │   ├── LinkedList<E>           ← двусвязный список (также Deque)
    │   ├── Vector<E>  [legacy]     ← синхронизированный ArrayList
    │   └── Stack<E>   [legacy]     ← LIFO поверх Vector
    ├── Set<E>                      ← уникальные элементы
    │   ├── HashSet<E>              ← HashMap под капотом, нет порядка
    │   │   └── LinkedHashSet<E>    ← insertion order
    │   └── SortedSet<E>
    │       └── NavigableSet<E>
    │           └── TreeSet<E>      ← красно-чёрное дерево, sorted
    └── Queue<E>                    ← обычно FIFO
        ├── PriorityQueue<E>        ← min-heap (binary heap)
        ├── BlockingQueue<E>        ← потокобезопасная (java.util.concurrent)
        └── Deque<E>                ← двусторонняя очередь
            ├── ArrayDeque<E>       ← кольцевой буфер, рекомендован для stack/queue
            └── LinkedList<E>       ← (также List)

Java 21 — новые интерфейсы:
SequencedCollection<E> ← getFirst/getLast/reversed (List, Deque, LinkedHashSet)
SequencedSet<E>        ← (LinkedHashSet, TreeSet)
SequencedMap<K,V>      ← (LinkedHashMap, TreeMap, SortedMap)
```

## Иерархия Map

```
Map<K,V>
├── HashMap<K,V>                 ← нет порядка, O(1) amortized
│   └── LinkedHashMap<K,V>       ← insertion/access order
├── SortedMap<K,V>
│   └── NavigableMap<K,V>
│       └── TreeMap<K,V>         ← красно-чёрное дерево, sorted keys
├── EnumMap<K,V>                 ← только enum ключи, массив под капотом
├── IdentityHashMap<K,V>         ← сравнение ключей через ==, не equals
├── WeakHashMap<K,V>             ← weak references на ключи
└── Hashtable<K,V>  [legacy]     ← полностью synchronized, не используй
    └── Properties               ← для .properties файлов
```

## Concurrent коллекции (java.util.concurrent)

```
ConcurrentHashMap<K,V>           ← lock-striping, get() без lock (Java 8+)
CopyOnWriteArrayList<E>          ← copy-on-write, итерация без lock
CopyOnWriteArraySet<E>           ← на основе CopyOnWriteArrayList
ConcurrentLinkedQueue<E>         ← lock-free очередь (CAS)
ConcurrentLinkedDeque<E>         ← lock-free двусторонняя
BlockingQueue<E>:
  LinkedBlockingQueue<E>          ← bounded/unbounded, lock per head/tail
  ArrayBlockingQueue<E>           ← bounded, single lock
  PriorityBlockingQueue<E>        ← приоритетная, unbounded
  SynchronousQueue<E>             ← передача напрямую без буфера
  LinkedTransferQueue<E>          ← производитель ждёт потребителя
```

## Ключевые реализации — когда что

| Задача | Рекомендация | Почему |
|---|---|---|
| Список общего назначения | `ArrayList` | Cache-friendly, O(1) get |
| Стек / LIFO | `ArrayDeque` | Не Stack (legacy Vector) |
| Очередь / FIFO | `ArrayDeque` | Не LinkedList (cache misses) |
| Приоритетная очередь | `PriorityQueue` | Min-heap |
| Уникальные, быстрый поиск | `HashSet` | O(1) contains |
| Уникальные + insertion order | `LinkedHashSet` | |
| Уникальные + sorted | `TreeSet` | O(log n) |
| Ключ-значение, быстрый поиск | `HashMap` | O(1) amortized |
| Ключ-значение + insertion order | `LinkedHashMap` | |
| Ключ-значение + sorted | `TreeMap` | O(log n) |
| Ключи — enum | `EnumMap` | O(1), нет hash, массив |
| Многопоточность | `ConcurrentHashMap` | |
| Редкие записи, много читателей | `CopyOnWriteArrayList` | |

## ConcurrentModificationException — механика

```java
// modCount — счётчик структурных изменений в AbstractList и HashMap
// Создаётся Iterator: expectedModCount = modCount
// При каждом next(): if (modCount != expectedModCount) throw CME

List<String> list = new ArrayList<>(List.of("a", "b", "c"));
for (String s : list) {           // for-each создаёт Iterator
    if (s.equals("b")) list.remove(s); // list.modCount++ → CME!
}

// Правильно — через Iterator:
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("b")) it.remove(); // iterator.modCount sync
}

// Или removeIf (Java 8+):
list.removeIf(s -> s.equals("b")); // атомарно, нет CME
```

**CME — fail-fast, не гарантия.** На `ConcurrentHashMap` и `CopyOnWriteArrayList` не бросается.

## Null в коллекциях

| Коллекция | null ключ | null значение |
|---|---|---|
| `HashMap` | 1 null ключ | ✓ |
| `LinkedHashMap` | 1 null ключ | ✓ |
| `Hashtable` | ✗ | ✗ |
| `TreeMap` | ✗ (NPE) | ✓ |
| `ConcurrentHashMap` | ✗ | ✗ |
| `HashSet` | 1 null | — |
| `TreeSet` | ✗ (NPE) | — |
| `List.of()`, `Set.of()` | — | ✗ (NPE) |

## Подводные камни

- **`Stack` — наследник `Vector`** — `Stack.push()`/`pop()` + все методы `Vector` (get(index), add(index, ...)). Нарушает инкапсуляцию стека. Всегда: `ArrayDeque` для LIFO.
- **`LinkedList` как Queue/Deque** — теоретически подходит, практически медленнее `ArrayDeque` из-за выделения Node на каждый элемент и cache misses.
- **`Collections.synchronizedList()` без внешней синхронизации итерации** — сам `synchronized` обёртывает отдельные методы, но `for (x : list)` создаёт Iterator вне `synchronized` блока → гонка. Нужно: `synchronized(list) { for (...) }`.
- **`TreeMap`/`TreeSet` с `null`** — `NullPointerException` при `put(null, v)` или `add(null)`, т.к. при сравнении вызывается `Comparable.compareTo(null)`.
- **`IdentityHashMap` случайно** — если передать `IdentityHashMap` туда где ожидается обычный `Map`, логика `equals`-based поиска сломается. Используй только намеренно (canonicalization, object graph traversal).
