# Производительность коллекций в Java

> Выбор коллекции = выбор алгоритма. `ArrayList` выигрывает у `LinkedList` в большинстве сценариев за счёт **cache locality** (непрерывная память). `HashMap` — O(1) amortized, деградирует до O(log n) при коллизиях (treeification). `TreeMap`/`TreeSet` — O(log n) гарантированно. `EnumMap` — O(1) без hash-вычислений.
> На интервью: почему ArrayList быстрее LinkedList на итерации, HashMap treeification, loadFactor и инициализация, ConcurrentHashMap vs synchronizedMap, @Contended и false sharing.

## Cache Locality: почему ArrayList быстрее LinkedList

```java
// LinkedList: каждый узел — отдельный heap-объект (Node: prev + item + next = ~48 байт)
// Элементы разбросаны по heap → cache miss на каждом шаге
// L1 cache miss: ~4 cycles, L3: ~40, RAM: ~200 cycles

LinkedList<Integer> linked = new LinkedList<>(data);
for (Integer x : linked) { sum += x; } // ~5x медленнее из-за cache misses

// ArrayList: Object[] внутри → последовательная память
// CPU prefetcher предсказывает адреса → L1 cache hits
ArrayList<Integer> array = new ArrayList<>(data);
for (Integer x : array) { sum += x; }  // cache-friendly

// Ещё лучше — int[] напрямую (нет boxing, JIT vectorizes):
int[] primitive = new int[1_000_000]; // 4 MB vs ~24 MB для ArrayList<Integer>
```

**Когда LinkedList выигрывает:** только вставка/удаление в начало при наличии Iterator в позиции. Для очередей — лучше `ArrayDeque`.

## HashMap внутренности

**Структура (Java 8+):** `Node<K,V>[]` (bucket array) → при коллизии 8+ узлов → `TreeNode` (красно-чёрное дерево).

```java
// TREEIFY_THRESHOLD = 8  (list → tree)
// UNTREEIFY_THRESHOLD = 6 (tree → list при shrink)
// Цель: защита от hash-flood атаки / плохого hashCode()
// O(n) список → O(log n) дерево при коллизиях

// Плохой hashCode — worst case:
class BadKey {
    @Override public int hashCode() { return 1; } // всё в один bucket!
}
// 100 элементов → один bucket из 100 узлов → до treeification: O(n) lookup

// Хороший hashCode:
class GoodKey {
    int id; String name;
    @Override
    public int hashCode() {
        return Objects.hash(id, name); // prime 31, равномерное распределение
    }
}
```

**Инициализация с известным размером:**
```java
// LoadFactor = 0.75 (default): при заполнении 75% → rehash (O(n)!)
// Для N элементов без rehash нужна capacity = N / 0.75 + 1:
Map<String, Data> map = new HashMap<>((int)(expectedSize / 0.75) + 1);

// Или Guava:
Map<String, Data> map = Maps.newHashMapWithExpectedSize(expectedSize);
```

**HashMap vs EnumMap:**
```java
// EnumMap<K extends Enum<K>, V>: внутри просто Object[] размером Enum.values().length
// ordinal() = индекс массива, нет hash вычислений, нет Entry объектов

Map<OrderStatus, List<Order>> ordersByStatus = new EnumMap<>(OrderStatus.class);
// get()/put() = O(1) без hash overhead, итерация = cache-friendly array sweep
// Используй всегда когда ключи — enum константы
```

## @Contended и False Sharing

```java
// False sharing: несколько volatile полей в одной cache line (64 байт)
class BadCounter {
    volatile long reads = 0;   // байты 0-7 — cache line 0
    volatile long writes = 0;  // байты 8-15 — ТА ЖЕ cache line!
    // Thread-1 инкрементирует reads → инвалидирует всю cache line
    // Thread-2 читает writes → cache miss! Хотя writes не менялся
}

// @Contended добавляет padding (128 байт) до и после поля
import jdk.internal.vm.annotation.Contended;

class GoodCounter {
    @Contended volatile long reads = 0;   // своя cache line
    @Contended volatile long writes = 0;  // своя cache line
    // Требует: -XX:-RestrictContended
}

// Именно так работает LongAdder:
// @Contended Cell { volatile long value; }
// Разные потоки пишут в разные Cell объекты → нет false sharing
// LongAdder при 16 потоках: ~10 ns/op vs AtomicLong: ~100 ns/op
```

## ConcurrentHashMap vs synchronizedMap

```java
// Collections.synchronizedMap(): lock на ВЕСЬ map при любой операции
List<String> sync = Collections.synchronizedList(new ArrayList<>());
// 16 читающих потоков → все ждут очереди → 1x throughput (!) несмотря на 16 ядер

// ConcurrentHashMap (Java 8+):
// get() — вообще без lock (volatile read)
// put() — lock только на конкретный bucket
// computeIfAbsent() — lock только на затронутый bucket
Map<String, Data> chm = new ConcurrentHashMap<>();

// При 16 потоках: ConcurrentHashMap.get() в 16x быстрее synchronizedMap
// size() — не точное при concurrent writes (используй mappingCount() для long)
```

## Выбор коллекции — шпаргалка

```
Нужен порядок + быстрый доступ по индексу → ArrayList
Нужна очередь/стек → ArrayDeque (НЕ LinkedList, НЕ Stack)
Нужны уникальные элементы → HashSet (нет порядка) / LinkedHashSet (insertion order) / TreeSet (sorted)
Нужен Map → HashMap (нет порядка) / LinkedHashMap (insertion order) / TreeMap (sorted)
Ключи — enum → EnumMap
Многопоточность, много читателей → ConcurrentHashMap
Многопоточность, редкие записи → CopyOnWriteArrayList (copy при каждом write!)
Приоритетная очередь → PriorityQueue (min-heap)
```

## Подводные камни

- **ArrayList вместо ArrayDeque для стека/очереди** — `Stack` — legacy (extends `Vector`), `LinkedList` в роли очереди медленнее из-за cache misses. Всегда: `ArrayDeque` для LIFO/FIFO.
- **HashMap без initial capacity при bulk-загрузке** — загрузка 10000 элементов без указания capacity → ~12 rehash операций, суммарно скопировано ~20000 элементов. Инициализируй правильно.
- **CopyOnWriteArrayList при частых записях** — каждый `add()`/`remove()` создаёт полную копию массива (O(n))! Подходит только когда чтений >> записей (кэш observers/listeners).
- **`synchronized(concurrentHashMap)` снаружи** — теряет всю пользу от ConcurrentHashMap. Для атомарных составных операций используй `compute()`, `merge()`, `computeIfAbsent()`.
- **`size()` у ConcurrentHashMap** — возвращает приблизительное значение при concurrent writes. Для точного подсчёта: `mappingCount()` (возвращает `long`).
- **LinkedHashMap с accessOrder=true как LRU** — `new LinkedHashMap<>(capacity, 0.75f, true)` + override `removeEldestEntry()`. Но thread-safe только при внешней синхронизации. Для production LRU: Caffeine.
