Интерфейсы в Java — это основа слабой связанности (loose coupling). Если раньше они были лишь строгими контрактами, то с эволюцией языка превратились в мощный инструмент с логикой, защитой и глубокой оптимизацией на уровне JVM.

---
## Эволюция: От контракта к компоненту

Интерфейсы прошли путь от «просто набора заголовков» до полноценных строительных блоков.

|**Версия**|**Новинки**|**Зачем?**|
|---|---|---|
|**< 8**|Только `abstract` методы|Чистый контракт.|
|**Java 8**|`default` и `static` методы|Обратная совместимость (добавление методов в старые интерфейсы без поломки реализаций).|
|**Java 9**|`private` методы|Инкапсуляция общей логики внутри `default` методов.|
|**Java 17**|`sealed` интерфейсы|Контроль иерархии и помощь JIT-компилятору.|

## Конфликты `default`-методов (Diamond Problem)

Когда класс реализует два интерфейса с одинаковыми `default`-методами, возникает конфликт. Java решает его по строгим правилам:

1. **Класс всегда прав:** Если метод реализован в суперклассе, он важнее любого `default`.
    
2. **Специфичность:** Если `B extends A`, то `B.hello()` победит `A.hello()`.
    
3. **Явный выбор:** Если иерархии независимы, компилятор заставит вас переопределить метод вручную.

```java
@Override
public void hello() {
    A.super.hello(); // Обращаемся к конкретному «родителю»
}
```

## Функциональные интерфейсы (SAM)

Интерфейс с **ровно одним** абстрактным методом — это входной билет в мир лямбд.

- **Аннотация `@FunctionalInterface`:** Не обязательна, но крайне полезна — она заставит компилятор выдать ошибку, если кто-то случайно добавит в интерфейс второй абстрактный метод.
    
- **Стандартная библиотека:** Знать `Function`, `Predicate`, `Consumer` и `Supplier` нужно наизусть. Это «алфавит» современного Java-кода.

## Sealed-интерфейсы: Закрытые миры

`sealed` — это не только про запрет наследования, это про **исчерпаемость (Exhaustiveness)**.

```java
public sealed interface Shape permits Circle, Rectangle {}
```

Когда вы используете `sealed` в `switch` (Java 17+), компилятор проверяет, обработали ли вы все варианты. Если да — блок `default` не нужен. Это делает код безопаснее: добавили новый `permits` — старый `switch` перестал компилироваться, напоминая о доработке.

## Под капотом: itable vs vtable

Это то, что отличает Senior от остальных. Как JVM находит метод?

- **vtable (Virtual Table):** Используется для `invokevirtual` (вызов через класс). У каждого метода в классе есть фиксированный индекс. Поиск — это $O(1)$.

- **itable (Interface Table):** Используется для `invokeinterface`. Так как класс может реализовать много интерфейсов, индексы методов в `itable` не фиксированы. Поиск превращается в сканирование таблицы.

## JIT и магия Inline Caching

Чтобы вызовы через интерфейс не тормозили, JIT-компилятор использует **Inline Caching**:

1. **Мономорфный вызов:** Если в конкретной точке кода всегда приходит один и тот же тип объекта (например, всегда `ArrayList`), JIT выбрасывает поиск по `itable` и делает прямой вызов. Это летает.

2. **Биморфный вызов:** Два типа (например, `ArrayList` и `LinkedList`). JIT строит быструю проверку `if-else`.

3. **Мегаморфный вызов:** 3 и более реализаций в одной точке. Кэш сбрасывается, и JVM каждый раз честно ищет метод в `itable`. Это **медленно**.

> **Senior Tip:** `sealed` интерфейсы помогают JIT через **CHA (Class Hierarchy Analysis)**. Компилятор заранее знает все возможные реализации и может агрессивнее инлайнить код, даже если формально вызов идет через интерфейс.

## Вопросы на интервью (Interview Prep)

- **Чем `invokeinterface` отличается от `invokevirtual`?** (Ответ: Фиксированные индексы в vtable против динамического поиска в itable).

- **Что такое effectively final для лямбд?** (Ответ: Переменная, которая не помечена final, но не меняется после инициализации).

- **Можно ли создать объект интерфейса?** (Ответ: Нет, только через анонимный класс или лямбду, которые по сути являются реализациями).

- **Почему `static` методы интерфейса не наследуются?** (Ответ: Чтобы избежать конфликтов имен при множественном наследовании интерфейсов).

## Подводные камни (Pitfalls)

1. **Константы в интерфейсах:** Плохая практика (Constant Interface Antipattern). Засоряет пространство имен подклассов. Используйте `enum` или `final` классы.

2. **Мегаморфные вызовы в циклах:** Если у вас огромный список `List<Shape>`, где перемешаны 10 разных фигур, вызовы `shape.draw()` будут медленными из-за промахов кэша JIT.

3. **Default-методы и состояние:** Помните, что `default`-метод может обращаться только к другим методам интерфейса. Он не может иметь доступа к приватным полям будущего класса-реализации.
