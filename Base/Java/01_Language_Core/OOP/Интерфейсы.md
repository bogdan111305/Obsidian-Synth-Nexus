## Эволюция интерфейсов

| Версия | Что добавилось |
|---|---|
| < Java 8 | Только `abstract` методы и `public static final` поля |
| Java 8 | `default` методы, `static` методы |
| Java 9 | `private` методы (для вынесения общей логики из default) |
| Java 17 | `sealed` интерфейсы |

```java
public interface Lifecycle {
    int MAX_RETRIES = 3; // неявно public static final

    void start();        // неявно public abstract
    void stop();

    default void restart() {
        stop();
        start();
    }

    static Lifecycle noOp() {   // фабричный метод
        return new Lifecycle() {
            public void start() {}
            public void stop() {}
        };
    }

    private void log(String msg) { // выносим общую логику default-методов
        System.out.println("[Lifecycle] " + msg);
    }
}
```

## Множественное наследование и конфликты default-методов

```java
interface A {
    default void hello() { System.out.println("A"); }
}
interface B {
    default void hello() { System.out.println("B"); }
}

class C implements A, B {
    @Override
    public void hello() {
        A.super.hello(); // явный выбор — единственный способ
        B.super.hello();
    }
}
```

**Правила приоритета (Diamond problem):**
1. Класс всегда выигрывает у интерфейса
2. Более специфичный интерфейс выигрывает у менее специфичного
3. Если нет однозначного победителя — compile error, нужно явное переопределение

```java
interface Base {
    default void hello() { System.out.println("Base"); }
}
interface Derived extends Base {
    default void hello() { System.out.println("Derived"); }
}
class Impl implements Base, Derived {
    // NO compile error: Derived специфичнее Base → Derived.hello() выигрывает
}
```

## Sealed интерфейсы (Java 17+)

```java
public sealed interface Shape permits Circle, Rectangle, Triangle {}

public final class Circle    implements Shape { ... }
public final class Rectangle implements Shape { ... }
public non-sealed class Triangle implements Shape { ... } // открыт для дальнейшего наследования
```

- `final` — нельзя наследовать дальше
- `sealed` — можно наследовать только указанным классам
- `non-sealed` — открывает иерархию заново

**Exhaustiveness в switch:** компилятор знает все `permits` → не нужен `default`:

```java
double area(Shape s) {
    return switch (s) {
        case Circle c    -> Math.PI * c.r() * c.r();
        case Rectangle r -> r.w() * r.h();
        case Triangle t  -> t.base() * t.height() / 2;
        // default не нужен — sealed иерархия исчерпана
    };
}
```

## Функциональные интерфейсы

Интерфейс ровно с одним абстрактным методом (SAM — Single Abstract Method). `default`, `static`, `private` методы не считаются.

```java
@FunctionalInterface
interface Transformer<T, R> {
    R transform(T input);      // единственный абстрактный

    default Transformer<T, R> andLog() { // не нарушает SAM
        return input -> {
            R result = transform(input);
            System.out.println(input + " → " + result);
            return result;
        };
    }
}

Transformer<String, Integer> len = String::length; // method reference
Transformer<String, Integer> lenLog = len.andLog();
lenLog.transform("hello"); // "hello → 5"
```

Стандартные функциональные интерфейсы (`java.util.function`):
- `Function<T,R>` — `apply(T) → R`
- `Consumer<T>` — `accept(T)` (без возврата)
- `Supplier<T>` — `get() → T` (без аргументов)
- `Predicate<T>` — `test(T) → boolean`
- `BiFunction<T,U,R>`, `UnaryOperator<T>`, `BinaryOperator<T>` и др.

## Интерфейс vs Абстрактный класс

|                      | Интерфейс                                   | Абстрактный класс            |
| -------------------- | ------------------------------------------- | ---------------------------- |
| Состояние            | Нет (только `static final`)                 | Да (поля экземпляра)         |
| Конструктор          | Нет                                         | Да                           |
| Наследование         | Множественное (`implements`)                | Одиночное (`extends`)        |
| Модификаторы методов | `public`/`private`                          | `public`/`protected`/package |
| Когда использовать   | Контракт, возможность нескольких реализаций | Общая реализация + состояние |

**Правило:** если нужно состояние — абстрактный класс. Если только контракт — интерфейс.

## JVM: invokeinterface vs invokevirtual

> Это один из самых частых senior-вопросов про JVM.

**`invokevirtual`** (вызов метода через класс):
- Метод имеет **фиксированный индекс в vtable** — известен на этапе компиляции
- Lookup: `vtable[N]` → O(1) — прямой jump

**`invokeinterface`** (вызов метода через интерфейс):
- Один класс может реализовывать много интерфейсов с разными индексами
- JVM строит **itable** (interface table) — поиск по паре (interface, method)
- Без оптимизации: O(n) по числу реализованных интерфейсов

### Inline Caching — как JIT оптимизирует invokeinterface

```java
void process(Processor p, String data) {
    p.process(data); // invokeinterface
}

// После JIT-компиляции (мономорфный — один тип):
// if (p.getClass() == FileProcessor.class) {
//     FileProcessor.process(data);  // direct call, может инлайниться
// } else {
//     slowPath(p, data);
// }
```

| Состояние | Что происходит |
|---|---|
| **Мономорфный** (1 тип) | Inline cache: прямой вызов, JIT инлайнит |
| **Биморфный** (2 типа) | Два cache entry, оба инлайнятся |
| **Мегаморфный** (3+ типов) | Inline cache отключается → itable lookup → нет инлайнинга |

```java
// МЕГАМОРФНЫЙ САЙТ — потеря производительности:
List<Processor> list = List.of(
    new FileProcessor(), new NetworkProcessor(), new CacheProcessor(),
    new DatabaseProcessor(), new KafkaProcessor()  // 5 типов → megamorphic
);
for (Processor p : list) {
    p.process(data); // каждый вызов — через itable, без инлайнинга
}
```

### Sealed + JIT = девиртуализация

```java
sealed interface Processor permits FileProcessor, NetworkProcessor {}

// JIT знает все реализации через Class Hierarchy Analysis (CHA)
// + sealed PermittedSubclasses метаданные
// → может девиртуализировать даже через invokeinterface!
switch (processor) {
    case FileProcessor fp    -> fp.process(data);   // прямой вызов
    case NetworkProcessor np -> np.process(data);   // прямой вызов
}
```

**Диагностика:** `-XX:+PrintInlining` — если `invokeinterface` не помечен `inline`, это мегаморфный сайт.

## Подводные камни

- **Конфликт default-методов** — без явного переопределения класс не скомпилируется. Даже если реализации одинаковы — нужен явный override.
- **Мегаморфный call site** — 3+ реализаций интерфейса в одном месте вызова отключают inline caching. В горячем коде это критично.
- **`implements` vs `extends` для интерфейсов** — интерфейс `extends` другой интерфейс (не `implements`). Класс — `implements`.
- **Константы создают coupling** — хранить константы в интерфейсе (Constant Interface Antipattern) — антипаттерн. Используй `enum` или `final class`.
- **`static` метод интерфейса не наследуется** — `Child.staticMethod()` не скомпилируется, если метод объявлен в `Parent`-интерфейсе. Только через имя интерфейса.
- **`sealed` без `permits` в том же файле** — все `permits`-классы должны быть в том же пакете (без модульной системы) или модуле.
- **`@FunctionalInterface` не обязателен** — без аннотации SAM-интерфейс всё равно работает с лямбдами. Аннотация — только защита компилятора.
