# Инициализация объектов с учётом наследования

> Порядок инициализации при `new Child()`: **1) static поля/блоки родителя → 2) static поля/блоки потомка → 3) экземплярные поля/блоки родителя → 4) конструктор родителя → 5) экземплярные поля/блоки потомка → 6) конструктор потомка**. Static инициализация — **один раз** при загрузке класса.
> На интервью: порядок вывода для цепочки классов, почему виртуальный метод в конструкторе опасен, circular dependency при static инициализации.

## Полный порядок инициализации

```java
class GrandParent {
    static { System.out.println("1. static GrandParent"); }
    { System.out.println("3. instance GrandParent"); }
    GrandParent() { System.out.println("4. ctor GrandParent"); }
}

class Parent extends GrandParent {
    static { System.out.println("2. static Parent"); }
    { System.out.println("5. instance Parent"); }
    Parent() { System.out.println("6. ctor Parent"); }
}

class Child extends Parent {
    // static блоки уже выполнились (загрузка)
    { System.out.println("7. instance Child"); }
    Child() { System.out.println("8. ctor Child"); }
}

new Child();
// 1. static GrandParent
// 2. static Parent
// 3. instance GrandParent
// 4. ctor GrandParent
// 5. instance Parent
// 6. ctor Parent
// 7. instance Child
// 8. ctor Child
```

**Ключевые правила:**
- Static блоки и поля — при **загрузке класса** (один раз), по иерархии от родителя к потомку
- Instance блоки компилятор вставляет **в начало каждого конструктора** (перед явным кодом)
- `super()` — **всегда первое** в конструкторе; если не написать, компилятор добавляет неявно

## Байткод: что делает компилятор с instance initializer

```java
class Foo {
    int x = 10;         // instance field initializer
    { y = 20; }         // instance initializer block
    int y;

    Foo() {
        // Компилятор вставляет сюда:
        // invokespecial Object.<init>   (super())
        // putfield Foo.x = 10          (field init)
        // putfield Foo.y = 20          (init block)
        System.out.println("ctor");
    }
}
// Instance initializer blocks — синтаксический сахар.
// В байткоде они просто вставляются в начало каждого конструктора после super().
```

## Ловушка: виртуальный метод в конструкторе

```java
class Parent {
    Parent() { show(); } // вызывает переопределённый метод!
    void show() { System.out.println("Parent show"); }
}

class Child extends Parent {
    String msg = "Hello"; // инициализируется ПОСЛЕ Parent()

    Child() { super(); }  // super() вызывается неявно первым

    @Override
    void show() {
        System.out.println(msg); // msg == null здесь!
    }
}

new Child(); // Выводит: null — не "Hello"!
```

**Порядок:** `new Child()` → выделить память (msg=null) → `Child.<init>` → `Parent.<init>` → `show()` → JVM находит `Child.show()` → `msg` ещё null → `this.msg = "Hello"` слишком поздно.

**Решение:** не вызывай виртуальные методы из конструктора. Если нужно — используй `private` или `final`.

## Static инициализация и Circular dependency

```java
class A {
    static int x = B.y + 1;  // читает B.y при инициализации A
}
class B {
    static int y = A.x + 1;  // читает A.x при инициализации B
}
// Если загрузить A.class → начнёт инициализировать A → нужен B → начнёт B
// → нужен A → A ещё инициализируется → вернёт 0 (default) → B.y = 0+1 = 1
// → A.x = 1+1 = 2 → результат зависит от порядка загрузки!
```

Circular static dependency → неопределённые значения в зависимости от порядка загрузки классов. JVM не бросает исключение — просто даёт default значение.

## Подводные камни

- **Static блоки и исключения** — если static блок бросает `RuntimeException`, класс помечается как «failed to initialize». Последующие попытки загрузить класс → `NoClassDefFoundError` (не `ExceptionInInitializerError`).
- **Singleton через enum** — единственный способ гарантировать однократную инициализацию в многопоточной среде без явной синхронизации: JVM гарантирует thread-safe инициализацию static полей.
- **Null-поля при super() вызове метода** — самый частый источник NPE при наследовании. Всегда проверяй, не вызывает ли родитель виртуальный метод в конструкторе.
