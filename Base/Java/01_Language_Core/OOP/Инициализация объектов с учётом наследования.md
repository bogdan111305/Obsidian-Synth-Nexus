## Великий порядок: Кто за кем?

Когда вы пишете `new Child()`, запускается каскадный процесс. Главное правило: **сначала статика, потом экземпляры; сначала предки, потом потомки.**

### Полный цикл (6 шагов):

1. **Static-инициализация родителя:** поля и блоки `static` (выполняется один раз при загрузке класса).
    
2. **Static-инициализация потомка:** поля и блоки `static`.
    
3. **Instance-поля и блоки родителя:** в порядке объявления.
    
4. **Конструктор родителя:** тело конструктора.
    
5. **Instance-поля и блоки потомка:** в порядке объявления.
    
6. **Конструктор потомка:** тело конструктора.

## Байткод: Куда исчезают блоки инициализации?

Многие думают, что блоки инициализации `{ ... }` — это какая-то особая магия. На самом деле это **синтаксический сахар**.

Компилятор собирает все инициализаторы полей и блоки инициализации и «вклеивает» их в начало **каждого** конструктора сразу после вызова `super()`.

```java
// Исходный код
class Foo {
    int x = 10;
    { System.out.println("Block"); }
    Foo() { System.out.println("Ctor"); }
}

// Что видит JVM (упрощенно)
Foo() {
    super();                // Всегда первый
    this.x = 10;            // Прилетело из объявления поля
    System.out.println("Block"); // Прилетело из блока
    System.out.println("Ctor");  // Тело конструктора
}
```

## Главная ловушка: Виртуальные методы в конструкторе

Это «выстрел в ногу» номер один в Java. Если вызвать переопределяемый (non-final) метод в конструкторе родителя, он вызовет реализацию в потомке, **когда потомок еще не инициализирован**.

```java
class Parent {
    Parent() { 
        printMe(); // ОПАСНО!
    }
    void printMe() { System.out.println("Parent"); }
}

class Child extends Parent {
    String message = "Hello";

    @Override
    void printMe() {
        System.out.println(message.length()); // BAM! message еще null
    }
}

new Child(); // Результат: NullPointerException
```

**Почему так?** Конструктор `Parent` выполняется на шаге №4, а поле `message` в `Child` инициализируется только на шаге №5.

> **Правило:** Никогда не вызывайте в конструкторах методы, которые могут быть переопределены. Делайте их `private` или `final`.

## Циклическая static-зависимость

Статические поля инициализируются при загрузке класса. Если класс `A` обращается к `B`, а `B` к `A` во время инициализации, JVM не упадет, но вы получите дефолтные значения (0 или null) там, где их не ждали.

```java
class A { static int val = B.getVal(); }
class B { static int getVal() { return A.val + 1; } }
```

Если первым загрузится `A`, то `B.getVal()` увидит `A.val` как `0`, потому что `A` еще в процессе инициализации.

## Вопросы на интервью (Self-check)

1. **Почему `super()` должен быть первой строкой в конструкторе?**
    
    - _Ответ:_ Чтобы гарантировать, что состояние родителя сконструировано до того, как начнет работать логика потомка.
    
2. **Что произойдет, если статический блок инициализации бросит исключение?**
    
    - _Ответ:_ Вылетит `ExceptionInInitializerError`. После этого класс станет «битым», и любая попытка обратиться к нему вызовет `NoClassDefFoundError`.
    
3. **Инициализируются ли статические поля при создании каждого объекта?**
    
    - _Ответ:_ Нет, только один раз при первой загрузке класса в JVM (ClassLoader).

## Подводные камни (Pitfalls)

- **Order matters:** Внутри одного класса порядок полей и блоков важен. Если блок стоит выше поля, к которому он обращается, вы получите ошибку "Illegal forward reference".
    
- **Enum Singleton:** Инициализация `enum` потокобезопасна именно благодаря правилам статической инициализации JVM. Это самый дешевый способ сделать честный Singleton.
    
- **Shadowing (Затенение):** Если у родителя и потомка есть поля с одинаковым именем, они **не переопределяются**, а существуют параллельно. Конструктор родителя будет работать со своим полем, конструктор потомка — со своим.
