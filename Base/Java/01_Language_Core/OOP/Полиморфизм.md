## Два типа полиморфизма

В Java полиморфизм проявляется в двух формах, которые различаются моментом принятия решения о том, какой код выполнять.

### Статический полиморфизм (Overloading / Перегрузка)

Это наличие в классе методов с одним именем, но разными сигнатурами (типами или количеством аргументов).

- **Связывание**: Происходит на этапе компиляции (Compile-time).
    
- **Ловушка**: Компилятор выбирает метод, основываясь исключительно на **типе ссылки**, а не на фактическом типе объекта в куче. Например, если методу передать ссылку типа `Object`, содержащую строку, вызовется перегрузка для `Object`, а не для `String`.
    
- **Суть**: Это не является истинным динамическим полиморфизмом, а представляет собой выбор одного из нескольких методов с похожими именами.

### Динамический полиморфизм (Overriding / Переопределение)

Это создание в подклассе метода с той же сигнатурой, что и в родительском классе.

- **Связывание**: Происходит во время выполнения (Runtime).

- **Принцип**: JVM выбирает реализацию метода, основываясь на **фактическом типе объекта**.

- **Инструкция**: Используется байт-код `invokevirtual` или `invokeinterface`.

## Инструкции вызова: invokespecial vs invokevirtual

JVM использует специализированные инструкции для управления вызовами методов.

- **`invokevirtual`**: Используется для большинства вызовов методов экземпляра. Реализует динамический поиск через таблицу виртуальных методов (vtable).

- **`invokespecial`**: Используется для вызова конструкторов (`<init>`), private методов и методов суперкласса через `super.method()`.
    - **Почему `super` — это `invokespecial`?** Если бы вызов `super.makeSound()` использовал `invokevirtual`, JVM снова нашла бы переопределенную версию в текущем классе, что привело бы к бесконечной рекурсии. `invokespecial` обходит vtable и вызывает метод конкретного родительского класса напрямую.

- **`invokeinterface`**: Вызов методов через интерфейс. Работает медленнее, так как требует поиска в таблице интерфейсов (itable).

- **`invokestatic`**: Вызов статических методов. Связывается статически, так как статические методы не могут быть переопределены (они «скрываются» — method hiding).

## Механизм vtable и itable

Под капотом JVM управляет полиморфизмом через структуры данных в Metaspace.

### vtable (Virtual Method Table)

Каждый класс имеет массив указателей на методы — vtable.

- **Структура**: При наследовании подкласс копирует vtable родителя. Если метод переопределен, указатель в соответствующем «слоте» заменяется на адрес новой реализации.

- **Скорость**: Поиск метода имеет сложность **O(1)**, так как индекс конкретного метода (например, `toString`) в vtable одинаков для всей иерархии классов.

### itable (Interface Table)

Используется для вызовов через интерфейсы. Так как класс может реализовать множество интерфейсов в произвольном порядке, JVM не может зафиксировать смещение метода интерфейса заранее. Это делает `invokeinterface` потенциально сложнее и медленнее, чем `invokevirtual`.

## Оптимизации JIT: Inline Caching

Постоянный поиск в vtable накладен, поэтому JIT-компилятор использует технику «встраиваемого кэширования» (Inline Caching):

1. **Monomorphic (Мономорфный вызов)**: Если в конкретном месте кода всегда вызывается метод одного и того же типа объекта, JIT убирает поиск в vtable и инлайнит (вставляет) код метода напрямую.

2. **Bimorphic (Биморфный вызов)**: Если встречаются два типа объектов, JIT вставляет проверку `if (type == A) ... else if (type == B) ...`.

3. **Megamorphic (Мегаморфный вызов)**: Если типов 3 и более, оптимизация отключается, и JVM выполняет честный (и медленный) поиск через vtable. Однотипные коллекции данных помогают JIT поддерживать мономорфный статус вызовов.

## Продвинутые аспекты и ловушки

### Смертельная ловушка: Виртуальный метод в конструкторе

Вызов переопределяемого метода внутри конструктора родительского класса — опасная практика.

- **Проблема**: Когда выполняется конструктор родителя, поля наследника еще не инициализированы и содержат значения по умолчанию (`null`, `0`).
    
- **Результат**: Виртуальный вызов через `invokevirtual` найдет метод наследника, который попытается работать с неинициализированными полями, что приведет к `NullPointerException` или некорректному поведению.
    
- **Решение**: В конструкторах следует вызывать только `private` или `final` методы, так как они используют `invokespecial` и не являются виртуальными.
### Bridge-методы (Мосты)

При использовании **ковариантных типов возвращаемого значения** (когда метод подкласса возвращает более специфичный тип, чем в родителе), компилятор генерирует синтетические методы-мосты (bridge methods). Они нужны для обеспечения полиморфного вызова со стороны кода, работающего с типом родителя.

### Роль Sealed-классов

Введенные в Java 17 `sealed` классы позволяют ограничить иерархию наследников.

- Это дает JIT-компилятору преимущество: он заранее знает **все** возможные реализации и может агрессивнее девиртуализировать вызовы и выполнять инлайнинг, не опасаясь внезапного появления нового подтипа в runtime.
### Record и Инкапсуляция

`record` является идеальным способом реализации иммутабельных данных с правильной инкапсуляцией. В отличие от обычных классов, они не поддерживают наследование и всегда являются `final`, что упрощает работу JIT-оптимизатора за счет отсутствия виртуальности полей.
