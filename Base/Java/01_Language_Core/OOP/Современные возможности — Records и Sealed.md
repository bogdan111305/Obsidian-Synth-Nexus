## Records: Иммутабельные носители данных (Java 16+)

**Record** — это компактный класс, чья цель — прозрачное хранение данных. Это реализация концепции «Product Type» из теории типов.

### Что под капотом?

Для `public record Point(int x, int y) {}` компилятор генерирует:

- **Поля:** `private final int x;` и `private final int y;`.

- **Аксессоры:** `x()` и `y()` (вместо привычных `getX()`).

- **Конструктор:** Canonical Constructor, принимающий все аргументы.

- **Методы:** `equals()`, `hashCode()` и `toString()`. Они используют `invokedynamic` для оптимизации производительности на уровне JVM.

### Продвинутые фишки:

- **Компактный конструктор:** Идеален для валидации. Параметры не указываются, присваивание происходит автоматически в конце.
    
```java
public record Wrapper(List<String> items) {
	public Wrapper {
		items = List.copyOf(items); // Защитное копирование
	}
}
```
    
- **Безопасная десериализация:** В отличие от обычных классов, Records всегда проходят через конструктор, что исключает создание объектов с невалидным состоянием через «магию» байт-кода.

## Sealed Classes: Запечатанные иерархии (Java 17+)

**Sealed-типы** позволяют ограничить круг наследников. Это реализация «Sum Type».

### Синтаксис и правила

```java
public sealed interface Shape permits Circle, Rectangle, Triangle {}
```

Каждый наследник **обязан** явно указать свой статус:

1. **final**: Наследование запрещено (Records неявно final).

2. **sealed**: Продолжает закрытую иерархию.

3. **non-sealed**: Открывает ветку для произвольного наследования.

> **JIT-Оптимизация (CHA):** Благодаря четкому списку наследников, JIT-компилятор (через Class Hierarchy Analysis) может применять агрессивный инлайнинг, превращая медленные полиморфные вызовы в быстрые прямые переходы.

## Switch Expressions и Pattern Matching (Java 14–21)

Это клей, который объединяет Records и Sealed Classes в мощный механизм обработки бизнес-логики.

### Switch Expressions (Java 14)

Превращает `switch` из инструкции в выражение, возвращающее значение.

- **Стрелочный синтаксис (`->`):** Устраняет риск забыть `break` (fall-through невозможен).

- **Ключевое слово `yield`:** Используется для возврата значения из многострочного блока кода.

- **Exhaustiveness:** Компилятор требует обработать все варианты.

### Pattern Matching instanceof (Java 16)

Совмещает проверку типа и приведение (downcasting).

```java
if (obj instanceof String s && s.length() > 5) {
    System.out.println(s.toUpperCase()); // s уже типа String
}
```

### Pattern Matching in switch (Java 21)

Позволяет использовать типы и логические проверки прямо в `case`.

```java
static String describe(Object obj) {
    return switch (obj) {
        case Integer i when i < 0 -> "Отрицательное число"; // Guard Pattern
        case Integer i            -> "Число: " + i;         // Общий случай
        case String s             -> "Строка: " + s;
        case null                 -> "Объект равен null";    // Явная обработка null
        default                   -> "Неизвестный тип";
    };
}
```

## Деструктуризация и современные расширения

### Record Patterns (Java 21)

Позволяет «разобрать» объект на части прямо в условии.

```java
record Point(int x, int y) {}
record Circle(Point center, double radius) {}

if (obj instanceof Circle(Point(var x, var y), var r)) {
    System.out.println("Круг в координатах " + x + ":" + y);
}
```

### Unnamed Patterns (Java 22)

Использование символа подчеркивания `_` для игнорирования ненужных компонентов.

```java
case Circle(_, var r) -> Math.PI * r * r; // Нам не важен центр круга
```

### Primitive Types in Patterns (Java 25)

Позволяет использовать примитивы в паттернах без лишнего автобоксинга, обеспечивая высокую производительность.

```java
if (obj instanceof int i) { /* ... */ }
```

## Подводные камни (Pitfalls)

1. **Доминирование паттернов (Domination):** Специфичные условия должны быть выше общих. `case Integer i` выше `case Integer i when i > 0` вызовет ошибку компиляции.

2. **Проблема null:** Switch по умолчанию бросает `NPE`. Если вы ожидаете `null`, обязательно добавьте `case null`.

3. **Хрупкость Sealed:** Если вы добавите новый класс в `permits`, все `switch`-выражения по этой иерархии перестанут компилироваться. Это **фича**, гарантирующая, что вы не забудете обновить логику.

4. **Мутабельность внутри Record:** Record гарантирует неизменность ссылок, но не содержимого. `record MyRec(List<String> list) {}` все еще позволяет менять содержимое списка. Используйте `List.copyOf()`.

## Вопросы на интервью

### Junior/Middle

- **В чем разница между Switch Expression и Switch Statement?** (Expression возвращает значение, использует `->` и `yield`, требует обработки всех случаев).
    
- **Что такое pattern variable и какова её область видимости?** (Это переменная в `instanceof` или `case`, видимая только там, где проверка истинна).
### Senior

- **Как Sealed Classes помогают JIT-компилятору?** (Через Class Hierarchy Analysis для девиртуализации вызовов).
    
- **Почему Record Patterns не работают с null по умолчанию?** (Паттерн требует деструктуризации существующего объекта; для `null` нужен отдельный `case null`).
    
- **Зачем нужен `non-sealed`?** (Это способ оставить «лазейку» для расширения в строго закрытой иерархии).
