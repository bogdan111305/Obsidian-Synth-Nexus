# Наследование в Java: От Основ до Глубин JVM

**Наследование** — это механизм, позволяющий создать новый класс на основе существующего. Это реализует отношение **«is-a»** (является): `Dog` является `Animal`. В Java разрешено только **одиночное наследование** классов, чтобы избежать «алмаза смерти» (diamond problem), но разрешено множественное выполнение интерфейсов.

---

## 1. Основы и Правила Переопределения

Для наследования используется ключевое слово `extends`. Основная задача — повторное использование кода и создание полиморфных иерархий.

### Базовый пример

Java

```
public class Animal {
    protected String name; 

    public Animal(String name) { this.name = name; }

    public void makeSound() { System.out.println("Звук..."); }
}

public class Dog extends Animal {
    public Dog(String name) {
        super(name); // Вызов конструктора родителя — всегда ПЕРВАЯ строка
    }

    @Override
    public void makeSound() {
        super.makeSound(); // Вызов метода родителя
        System.out.println("Гав!");
    }
}
```

### Золотые правила переопределения (Overriding):

1. **Сигнатура:** Имя и аргументы должны полностью совпадать.
    
2. **Return Type:** Допускается **ковариантность** (можно вернуть подтип того, что возвращает родитель).
    
3. **Видимость:** Нельзя сужать (например, `public` → `protected` — ошибка). Расширять можно.
    
4. **Exceptions:** Нельзя бросать новые или более широкие **checked** исключения.
    
5. **Static методы:** Не переопределяются, а **скрываются** (Method Hiding).
    

> **Method Hiding vs Overriding:**
> 
> Если метод `static`, JVM выберет его по **типу ссылки**. Если метод обычный (виртуальный), JVM выберет его по **типу объекта** в куче.

## 2. Senior Insight: Байт-код и инструкции вызова

Понимание наследования на уровне Senior — это знание того, как работают инструкции `invokespecial` и `invokevirtual`.

### `invokespecial`: Прямой вызов

Используется для:

- Конструкторов (`super()`, `this()`).
    
- Приватных методов.
    
- Методов суперкласса (`super.method()`).
    

**Почему `super.method()` — это `invokespecial`?**

Если бы JVM использовала здесь `invokevirtual`, она бы пошла в `vtable` объекта, нашла там текущий (переопределенный) метод и вошла в бесконечную рекурсию. `invokespecial` принудительно вызывает реализацию конкретного класса, игнорируя динамический поиск.

### `invokevirtual`: Динамический поиск

Используется для обычных публичных/защищенных методов. Здесь работает **vtable**.

## 3. Bridge-методы: Мосты для ковариантности

Когда вы меняете тип возвращаемого значения в наследнике, для JVM это выглядит как нарушение контракта.

```java
class A { Object get() { return null; } }
class B extends A { String get() { return ""; } }
```

Чтобы код, работающий с типом `A`, не сломался, компилятор создает **синтетический bridge-метод** в классе `B`:

```java
// Сгенерировано компилятором
public synthetic bridge Object get() {
    return this.get(); // Вызывает версию, возвращающую String
}
```

(**Bridge methods** — костыли компилятора для ковариантности и дженериков.) Это гарантирует, что полиморфизм работает даже при изменении типов в сигнатуре.

## 4. Смертельная ловушка: Виртуальный метод в конструкторе

Это одна из самых популярных задач на глубокое понимание инициализации.

Java

```
class Parent {
    Parent() { printVar(); }
    void printVar() { System.out.println("Parent"); }
}

class Child extends Parent {
    String name = "Java";
    @Override
    void printVar() { System.out.println(name); }
}

new Child(); // Выведет: null
```

### Почему так происходит?

1. Выделяется память под `Child`, все поля инициализируются нулями/null.
    
2. Вызывается конструктор `Child`.
    
3. Первой строкой идет `super()`.
    
4. В конструкторе `Parent` вызывается `printVar()`.
    
5. Так как это виртуальный метод (`invokevirtual`), JVM лезет в `vtable` и находит версию `Child`.
    
6. Метод `Child.printVar()` пытается вывести поле `name`, но **инициализация полей Child еще не произошла** (она идет после возврата из `super()`).
    
7. Результат: **null**.
    

**Правило:** В конструкторах вызывайте только `final` или `private` методы — они не виртуальны и безопасны.

## 5. Sealed классы и оптимизация JIT

Современная Java (17+) вводит `sealed` классы, которые ограничивают список наследников:

Java

```
public sealed class Shape permits Circle, Square {}
```

### Как это помогает JIT-компилятору?

Обычно JIT осторожничает с **инлайнингом** (встраиванием кода), так как в Runtime может загрузиться новый класс-наследник и изменить поведение.

С `sealed` классами JIT точно знает **все** возможные реализации. Это позволяет ему:

1. Полностью убрать поиск в `vtable` (девиртуализация).
    
2. Агрессивно инлайнить код всех ветвей `switch` или иерархии.
    
3. Превратить сложный полиморфный вызов в простой набор ассемблерных команд.

## Итог для интервью

- **Наследование** — это `extends`, одиночное, `is-a`.
    
- **super()** — всегда первая строка конструктора.
    
- **vtable** — механизм для `invokevirtual`, обеспечивает $O(1)$ поиск.
    
- **invokespecial** — для конструкторов, приватных методов и `super`.
    
- **Bridge methods** — костыли компилятора для ковариантности и дженериков.
    
- **Виртуальные методы в конструкторе** — зло, приводящее к работе с неинициализированным состоянием.
