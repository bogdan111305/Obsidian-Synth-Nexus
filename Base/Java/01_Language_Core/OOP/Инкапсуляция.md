**Инкапсуляция** — это не просто создание приватных полей. Это проектирование системы так, чтобы объект всегда находился в валидном состоянии, а его внутренняя логика была скрыта за защищенным «черным ящиком».

## Инкапсуляция vs Абстракция

На интервью эти понятия часто просят сравнить:

- **Абстракция** — это процесс выделения значимых характеристик объекта и скрытие _сложности_ (что именно объект делает). Мы реализуем её через интерфейсы.

- **Инкапсуляция** — это скрытие _данных_ и реализации (как объект это делает). Она обеспечивает целостность (инвариант) данных.

## Модификаторы доступа: Базовые правила и тонкости

Java предоставляет 4 уровня доступа. Основной принцип — **POLD (Principle of Least Disclosure)**: всегда выбирайте самый строгий модификатор.

|**Модификатор**|**В классе**|**В пакете**|**В подклассах**|**Везде**|
|---|---|---|---|---|
|`private`|✅|❌|❌|❌|
|`package-private` (default)|✅|✅|❌|❌|
|`protected`|✅|✅|✅|❌|
|`public`|✅|✅|✅|✅|
### Ограничения для классов верхнего уровня

Классы верхнего уровня (не вложенные) могут быть только **public** или **package-private**. Вы не можете объявить такой класс как `protected` или `private`, так как это противоречит логике организации кода: класс, который никто не видит, бесполезен. Эти модификаторы разрешены только для вложенных (Nested) классов.
### Ловушка модификатора `protected`

Доступ к `protected` члену из другого пакета возможен **только через наследование и только через ссылку типа подкласса**.

```java
// Пакет com.base
public class Base { protected int value = 42; }

// Пакет com.derived
public class Derived extends Base {
    void method() {
        System.out.println(this.value); // OK: через наследование
        Base b = new Base();
        // System.out.println(b.value); // ОШИБКА: доступ из другого пакета через ссылку суперкласса запрещен
    }
}
```

## Пакеты и Импорты

Пакеты группируют связанные классы. Важно помнить:

- **Wildcard Import (`*`):** Команда `import java.util.*` импортирует все классы только из текущего пакета. Она **не включает подпакеты**. Например, `java.util.concurrent.*` нужно импортировать отдельно, так как с точки зрения JVM это совершенно другой пакет.
    
- **Default package:** Использование классов без объявления `package` — плохая практика. Классы в «дефолтном» пакете **нельзя импортировать** в обычные именованные пакеты.
    
- **Статический импорт:** Используется для констант и утилит (`import static java.lang.Math.*`). Это лучше, чем "Constant Interface Antipattern" (хранение констант в интерфейсах), который засоряет пространство имен.

## 4. Современный уровень доступа: JPMS (Java 9+)

Модульная система добавила уровень видимости поверх пакетов. Теперь `public` перестал означать «доступен всем».

- **exports:** Пакет виден другим модулям при компиляции и выполнении. Если `public` класс находится в неэкспортированном пакете, он будет невидим снаружи модуля.
    
- **opens:** Разрешает доступ к пакету через **Reflection API** в рантайме. Без этого вызова `setAccessible(true)` на приватных полях выбросит `InaccessibleObjectException`.

## 5. Защита состояния: Defensive Copying и Мутабельность

Инкапсуляция бесполезна, если вы отдаете прямой доступ к изменяемым полям.

### Проблема мутабельных объектов и массивов

Если геттер возвращает ссылку на массив или мутабельный список, инкапсуляция нарушена:

Java

```
private String[] roles;
public String[] getRoles() { return roles; } // ОШИБКА: внешний код может сделать getRoles()[0] = "ADMIN"
```

**Решение:** Всегда возвращайте копию через `Arrays.copyOf(roles, roles.length)` или используйте иммутабельные коллекции.

### Collections.unmodifiableList vs List.copyOf

- **`unmodifiableList`:** Это всего лишь «вид» (view) на оригинал. Если оригинал изменится внутри класса, «вид» тоже отразит изменения.
    
- **`List.copyOf`:** Создает полностью независимый «снимок» (snapshot) данных. Это гарантирует, что внешние изменения или изменения внутри класса не затронут возвращенную копию.

## 6. Инкапсуляция в Records

`record` (Java 14+) — лучший способ для DTO. Они по умолчанию иммутабельны (`private final` поля).

**Главный бонус для безопасности — Десериализация:**

В отличие от обычных классов, которые при десериализации могут обходить конструктор (игнорируя валидацию), `record` **всегда** проходит через канонический конструктор. Это гарантирует, что данные будут проверены даже при восстановлении из байтов.

## 7. Senior Insights: Как ломается инкапсуляция

1. **Reflection API:** Обходит `private` через `setAccessible(true)`. Защищается с помощью JPMS (`opens`).
    
2. **Внутренние классы:** Имеют доступ к `private` полям внешнего класса. JVM реализует это через скрытые «bridge» методы (например, `access$000`), что является техническим компромиссом в защите.
    
3. **Байт-код:** Проверка доступа (`getfield`/`putfield`) происходит на этапе **Verification** при загрузке класса в JVM. Если правила нарушены, выбрасывается `IllegalAccessError`.

## Best Practices (Подводные камни)

- **Избегайте «Сеттеромании»:** Не генерируйте сеттеры для каждого поля. Инкапсуляция — это про управление состоянием. Вместо `setBalance(balance + 100)` создайте метод `deposit(100)`, где будет встроена логика валидации и логирования.
    
- **Приоритет иммутабельности:** Чем меньше в классе мутабельных полей, тем проще поддерживать инкапсуляцию. Используйте `final` и классы из `java.time` вместо `java.util.Date`.
    
- **Минимальный доступ:** Начинайте с `private` и расширяйте доступ только тогда, когда это действительно необходимо.

## Байт-код: Как это видит JVM

Когда JVM загружает класс, этап **Verifier** проверяет корректность доступа.

- Каждое поле в `class`-файле имеет флаги доступа (**Access Flags**).
    
- Инструкции `getfield` и `putfield` проверяются на соответствие этим флагам.
    
- Если код пытается обратиться к `private` полю другого класса, JVM выбрасывает `IllegalAccessError`.
