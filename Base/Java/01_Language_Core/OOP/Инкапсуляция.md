# Инкапсуляция в Java

> **Инкапсуляция** — скрытие внутреннего состояния объекта с доступом только через публичный интерфейс. Поля — `private`, доступ — через методы с валидацией. `record` — наиболее правильный способ инкапсуляции для DTO. Основной риск: утечка через возврат изменяемых объектов.
> На интервью: defensive copying, record vs POJO, как рефлексия ломает инкапсуляцию, JPMS как защита.

## Модификаторы доступа

| Модификатор | Класс | Пакет | Подкласс | Все |
|---|:---:|:---:|:---:|:---:|
| `private` | ✓ | ✗ | ✗ | ✗ |
| `package-private` (default) | ✓ | ✓ | ✗ | ✗ |
| `protected` | ✓ | ✓ | ✓ | ✗ |
| `public` | ✓ | ✓ | ✓ | ✓ |

```java
// Правило: минимально необходимый доступ
public class BankAccount {
    private double balance;                        // только этот класс
    private final List<String> log = new ArrayList<>();

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("amount must be positive");
        balance += amount;
        logAction("deposit", amount);
    }

    public double getBalance() { return balance; }

    private void logAction(String type, double amount) { // скрытая логика
        log.add("[%s] %.2f".formatted(type, amount));
    }

    public List<String> getLog() {
        return Collections.unmodifiableList(log); // defensive copy!
    }
}
```

## Defensive Copying — утечка через геттер

```java
// ❌ Утечка инкапсуляции:
public class Period {
    private final Date start;
    private final Date end;

    public Period(Date start, Date end) {
        this.start = start;  // ПРОБЛЕМА: Date мутабельный
        this.end = end;      // вызывающий может изменить start снаружи
    }

    public Date getEnd() { return end; }  // ПРОБЛЕМА: внешний код может: p.getEnd().setTime(0)
}

// ✓ Правильно — defensive copy:
public class Period {
    private final Date start;
    private final Date end;

    public Period(Date start, Date end) {
        this.start = new Date(start.getTime()); // copy в конструкторе
        this.end   = new Date(end.getTime());
        if (this.start.after(this.end)) throw new IllegalArgumentException();
    }

    public Date getEnd() { return new Date(end.getTime()); } // copy в геттере
}

// ✓ Современный подход — используй иммутабельные типы вместо Date:
public record Period(Instant start, Instant end) {
    public Period {
        if (start.isAfter(end)) throw new IllegalArgumentException();
    }
    // Instant — иммутабельный, defensive copy не нужен
}
```

**Defensive copy нужен для:** `Date`, `byte[]`, `int[]`, изменяемых коллекций (`ArrayList`, `HashMap`).

## record — правильная инкапсуляция данных

```java
// record: компилятор генерирует private final поля + геттеры + equals/hashCode/toString
record Point(int x, int y) {}

// Point p = new Point(1, 2);
// p.x()    → 1  (геттер без "get" префикса)
// p.equals(new Point(1, 2)) → true
// p.toString() → "Point[x=1, y=2]"

// Compact constructor для валидации:
record Range(int min, int max) {
    Range {
        if (min > max) throw new IllegalArgumentException("min > max");
    }
}

// Кастомный геттер:
record Name(String firstName, String lastName) {
    public String fullName() { return firstName + " " + lastName; }
}
```

**record нельзя:** наследовать (extends), добавлять изменяемые поля, объявить non-final.
**record можно:** implements интерфейсы, static поля/методы, кастомные методы.

## Рефлексия ломает инкапсуляцию

```java
// Reflection обходит private:
Field field = BankAccount.class.getDeclaredField("balance");
field.setAccessible(true);          // обход access control
field.set(account, 1_000_000.0);    // инкапсуляция нарушена

// Java 9+ JPMS ограничивает это:
// module-info.java:
module com.myapp {
    // Без opens — getDeclaredField().setAccessible(true) → InaccessibleObjectException
    opens com.myapp.model to com.fasterxml.jackson.databind; // только для Jackson
}
```

**Почему JVM разрешала setAccessible:** legacy совместимость (Spring, Hibernate использовали рефлексию для DI и маппинга). JPMS (Java 9+) дает возможность запретить это per-module.

## JVM: контроль доступа на уровне байткода

```
// getfield/putfield — JVM проверяет флаги доступа при verifier'е
// private field:
putfield BankAccount.balance D    // → IllegalAccessError если вызывается из другого класса
```

JVM верифицирует bytecode при загрузке класса — `private` поля проверяются там, не в runtime. `setAccessible(true)` отключает эту проверку через native метод.

## Подводные камни

- **Геттер возвращает изменяемый объект** — `getList()` → внешний код вызывает `add()` → нарушение инкапсуляции. Возвращай `Collections.unmodifiableList()` или `List.copyOf()`.
- **`Collections.unmodifiableList` — не полная защита** — view оригинала: если оригинал изменить, unmodifiable view отразит изменение. Для полной защиты — `List.copyOf()`.
- **Мутабельный массив в поле** — `getArray()` возвращает ссылку → внешний код изменяет. Возвращай `Arrays.copyOf(arr, arr.length)`.
- **`transient` и сериализация** — `transient` скрывает поле от стандартной сериализации, но кастомный `readObject`/`writeObject` или reflection может его достать.
- **Сеттер для всего** — если каждое поле имеет сеттер, инкапсуляция иллюзорна. Предпочитай иммутабельные объекты: `final` поля + конструктор + `record`.
