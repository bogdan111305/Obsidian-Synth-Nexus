**Инкапсуляция** — это механизм скрытия внутренней реализации класса и объединения данных (полей) с методами их обработки в один защищенный «черный ящик».

Главная цель — обеспечить **целостность (инвариант)** данных. Внешний мир не должен иметь возможности привести объект в невалидное состояние напрямую.
## Фундамент: Модификаторы доступа

Java предоставляет 4 уровня доступа. Основной принцип проектирования — **POLD (Principle of Least Disclosure)**: всегда выбирайте самый строгий модификатор.

|**Модификатор**|**Внутри класса**|**В пакете**|**В подклассе**|**Везде**|
|---|---|---|---|---|
|`private`|✓|✗|✗|✗|
|`package-private` (default)|✓|✓|✗|✗|
|`protected`|✓|✓|✓|✗|
|`public`|✓|✓|✓|✓|

> **Важно:** `protected` дает доступ подклассам, даже если они находятся в других пакетах. Это часто используется в библиотечных фреймворках для создания точек расширения.

## Инкапсуляция vs Абстракция

На интервью эти понятия часто просят сравнить.

- **Абстракция** — это процесс выделения значимых характеристик объекта и скрытие _сложности_ (что объект делает). Мы реализуем её через интерфейсы и абстрактные классы.

- **Инкапсуляция** — это скрытие _данных_ и реализации (как объект это делает внутри себя).

## Защита состояния: Defensive Copying

Просто сделать поля `private` недостаточно, если вы работаете с мутабельными (изменяемыми) объектами (Date, List, массивы).

### Ошибка: Утечка ссылки

Если геттер возвращает прямую ссылку на `private` поле, внешний код может изменить его состояние:

```
public List<String> getTags() { return this.tags; } // Внешний код: getTags().clear()
```

### Решение: Оборонительное копирование

1. **В конструкторе:** Создавайте копию входящего объекта.
    
2. **В геттере:** Возвращайте либо копию, либо иммутабельную обертку.
    

Java

```
public class Product {
    private final List<String> tags;

    public Product(List<String> tags) {
        this.tags = List.copyOf(tags); // Создаем неизменяемую копию (Java 10+)
    }

    public List<String> getTags() {
        return Collections.unmodifiableList(tags); // Возвращаем view (вид)
    }
}
```

_Примечание:_ `Collections.unmodifiableList` — это лишь «вид». Если изменить оригинал внутри класса, «вид» тоже изменится. Для полной изоляции в геттере также лучше использовать `List.copyOf`.

## 4. Современная Инкапсуляция: Records

`record` (Java 14+) — это лучший способ инкапсуляции для DTO (объектов передачи данных).

- Поля автоматически `private final`.
    
- Геттеры именуются без `get` (просто `name()`).
    
- **Компактный конструктор** позволяет валидировать данные до их присвоения.
    

Java

```
public record User(String email, int age) {
    public User { // Компактный конструктор
        if (!email.contains("@")) throw new IllegalArgumentException();
    }
}
```

## 5. Senior Insights: Как ломается инкапсуляция

Профессионал должен знать, как обойти защиту и как от этого защититься.

### А. Reflection API

Рефлексия позволяет менять флаги доступа в рантайме:

Java

```
Field field = MyClass.class.getDeclaredField("secret");
field.setAccessible(true); // Инкапсуляция пробита
```

**Защита:** С Java 9+ модульная система (**JPMS**) блокирует это. Если пакет не помечен как `opens`, попытка `setAccessible` вызовет `InaccessibleObjectException`.

### Б. Внутренние классы (Inner Classes)

Внутренние классы имеют полный доступ к `private` полям внешнего класса.

**Подвох в байт-коде:** JVM на уровне инструкций не поддерживает доступ между разными классами. Поэтому компилятор генерирует синтетические методы (bridge methods), такие как `access$000()`, в родительском классе. Это микро-дыра в инкапсуляции.

### В. Сериализация: «Черный ход»

Стандартная сериализация Java может восстановить объект, обходя конструктор и его валидацию.

**Защита:** 1. Использовать `readObject` для ручной проверки данных после десериализации.

2. Использовать `record` — они всегда десериализуются через канонический конструктор, что гарантирует прохождение валидации.

## 6. Байт-код: Как это видит JVM

Когда JVM загружает класс, этап **Verifier** проверяет корректность доступа.

- Каждое поле в `class`-файле имеет флаги доступа (**Access Flags**).
    
- Инструкции `getfield` и `putfield` проверяются на соответствие этим флагам.
    
- Если код пытается обратиться к `private` полю другого класса, JVM выбрасывает `IllegalAccessError`.

## Вопросы на интервью (Self-check)

1. **Чем `Collections.unmodifiableList` отличается от `List.copyOf`?** (Первый — это вид на оригинал, второй — независимая копия).
    
2. **Почему нельзя просто возвращать `Date` из геттера?** (Потому что `Date` мутабелен, и его можно изменить снаружи).
    
3. **Как JPMS (Java 9+) усилила инкапсуляцию?** (Запретила рефлексивный доступ к внутренним пакетам модулей по умолчанию).
    
4. **Может ли конструктор нарушить инкапсуляцию?** (Да, если он сохраняет ссылку на переданный мутабельный объект без копирования).
    
5. **Что такое «инкапсуляция на уровне модулей»?** (Механизм, позволяющий сделать класс `public` внутри модуля, но невидимым для кода снаружи модуля).

## Подводные камни и Best Practices

- **Избегайте «Сеттеромании»:** Не генерируйте сеттеры автоматически. Объект должен менять состояние через методы с бизнес-смыслом (например, `confirmOrder()` вместо `setStatus(CONFIRMED)`).
    
- **Приоритет иммутабельности:** Если объект неизменяем (как `String`), ему не нужно оборонительное копирование. Это самый надежный способ инкапсуляции.
    
- **Валидация:** Всегда проверяйте входящие данные в конструкторе или сеттере. Инкапсуляция без валидации — это просто переименование переменных.
