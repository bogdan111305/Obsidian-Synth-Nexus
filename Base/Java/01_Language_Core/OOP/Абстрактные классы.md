Абстрактный класс в Java — это «чертеж чертежа». Он не предназначен для создания объектов сам по себе, но служит фундаментом для целого семейства классов. Если интерфейс — это **контракт** (что объект умеет делать), то абстрактный класс — это **сущность** (чем объект является и какую часть работы он уже берет на себя).

---

## 1. Abstract Class vs Interface: Вечное противостояние

С появлением `default` методов в Java 8 и `private` методов в Java 9 границы размылись, но фундаментальное отличие осталось — **состояние**.

|**Характеристика**|**Абстрактный класс**|**Интерфейс**|
|---|---|---|
|**Состояние (поля)**|Да, любые (private, protected, и т.д.)|Только `public static final` (константы)|
|**Конструктор**|**Да**, вызывается через `super()`|Нет|
|**Наследование**|Только один (`extends`)|Множественное (`implements`)|
|**Методы**|Любые: abstract, concrete, final, static|`abstract`, `default`, `static`, `private`|
|**Модификаторы**|Любые (protected, package-private)|Только `public` (по умолчанию) и `private`|

> **Когда что выбирать?**
> 
> - **Интерфейс:** Если нужно определить общую роль для совершенно разных классов (например, `Serializable`, `Cloneable`).
>    
> - **Абстрактный класс:** Если есть тесная родственная связь и общая база кода/полей (например, `BaseService` или `AbstractEntity`).

## Паттерн Template Method (Шаблонный метод)

Это «killer feature» абстрактных классов. Вы определяете скелет алгоритма в `final` методе, а конкретные шаги оставляете подклассам.

```java
public abstract class DataProcessor {
    protected final String source;

    public DataProcessor(String source) { this.source = source; }

    // Скелет алгоритма — менять нельзя!
    public final void process() {
        validate();
        var data = readData();
        transform(data);
        save(data);
    }

    protected abstract List<String> readData();
    protected abstract void transform(List<String> data);

    protected void validate() { /* дефолтная логика */ }
    private void save(List<String> data) { /* скрытая логика */ }
}
```

## Байткод: ACC_ABSTRACT

На уровне JVM абстрактность — это просто флаг.

Если вы посмотрите на класс через `javap -v`, вы увидите:

- **Флаг `ACC_ABSTRACT` на классе:** запрещает инструкцию `new`. Попытка создать экземпляр через рефлексию вызовет `InstantiationException`.

- **Флаг `ACC_ABSTRACT` на методе:** говорит JVM, что у метода нет тела (Code attribute), и его нужно искать в таблице виртуальных методов (vtable) подкласса.

## Современная Java: Sealed Abstract Classes (Java 17+)

Раньше любой мог наследоваться от вашего абстрактного класса. `sealed` позволяет ограничить круг «избранных».

```java
public sealed abstract class Shape permits Circle, Square {
    public abstract double area();
}
```

**Зачем это нужно?**

1. **Безопасность:** Вы точно знаете все реализации.

2. **Switch Expressions:** Компилятор проверит, все ли варианты вы обработали (Exhaustiveness check).

3. **Производительность:** JIT-компилятор может применить **девиртуализацию**, так как он видит всю иерархию целиком и может заменить виртуальный вызов прямым переходом.

## Вопросы на интервью (Interview Prep)

1. **Может ли быть абстрактный класс без абстрактных методов?**
    
    - _Ответ:_ Да. Это используется, чтобы просто запретить создание экземпляров класса.
        
2. **Может ли абстрактный метод быть private?**
    
    - _Ответ:_ Нет. Смысл абстрактного метода в том, чтобы его переопределили, а `private` методы не видны в подклассах.
        
3. **Зачем нужен конструктор в абстрактном классе, если нельзя сделать `new`?**
    
    - _Ответ:_ Для инициализации общих полей, которые будут использоваться подклассами. Вызывается через `super()`.
        
4. **В чем разница между абстрактным классом и интерфейсом в плане памяти?**
    
    - _Ответ:_ Поля абстрактного класса — это часть каждого объекта подкласса (в Heap). Поля интерфейса — это статические константы (в Metaspace).

## Подводные камни (Pitfalls)

- **Нарушение хрупкого базового класса (Fragile Base Class):** Если вы добавите новый абстрактный метод в старый класс, вы сломаете **все** подклассы во всем проекте.
    
- **Виртуальные методы в конструкторе:** Если конструктор абстрактного класса вызывает абстрактный метод (который реализован в потомке), вы получите вызов на «недостроенном» объекте. Поля потомка еще будут `null` или `0`.
    
- **Глубокое наследование:** Если у вас `AbstractA extends AbstractB extends AbstractC`, пора остановиться. Глубокие иерархии — это кошмар для поддержки. Используйте композицию.
