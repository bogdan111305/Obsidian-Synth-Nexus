# Абстрактные классы в Java

> **Абстрактный класс** — нельзя создать (`new`), но можно наследовать. Содержит абстрактные методы (без тела) и конкретные методы. Отличие от интерфейса: может иметь состояние (поля), конструкторы, `protected` методы. Используй когда есть **общее состояние + частичная реализация** (Template Method Pattern).
> На интервью: abstract vs interface (таблица), `ACC_ABSTRACT` в байткоде, Template Method Pattern, sealed abstract class.

## Abstract class vs Interface

| | Абстрактный класс | Интерфейс |
|---|---|---|
| Состояние (поля) | Да, любые | Только `public static final` |
| Конструктор | Да | Нет |
| Методы | abstract + concrete | abstract, default (Java 8+), static, private (Java 9+) |
| Наследование | Только один (`extends`) | Несколько (`implements`) |
| Модификаторы методов | public/protected/package | public (до Java 9), private (Java 9+) |
| Когда использовать | Общее состояние + частичная реализация | Контракт без состояния, множественное наследование |

```java
// Классический пример: Template Method Pattern
public abstract class DataProcessor {
    protected final String source;

    public DataProcessor(String source) {
        this.source = source;
    }

    // Template method — алгоритм фиксирован, шаги переопределяются
    public final void process() {
        validate();
        List<String> data = readData();
        transform(data);
        writeResult(data);
    }

    protected abstract List<String> readData();   // обязательно реализовать
    protected abstract void transform(List<String> data);

    protected void validate() { /* дефолтная реализация, можно переопределить */ }
    private void writeResult(List<String> data) { /* нельзя переопределить */ }
}

public class CsvProcessor extends DataProcessor {
    public CsvProcessor(String path) { super(path); }

    @Override
    protected List<String> readData() { return Files.readAllLines(Path.of(source)); }

    @Override
    protected void transform(List<String> data) { /* CSV-специфичная логика */ }
}
```

## Байткод: ACC_ABSTRACT

```
// javap -v DataProcessor.class:
.class public abstract DataProcessor       // ACC_ABSTRACT на классе
  .method protected abstract readData()Ljava/util/List;  // метод без тела
  .method public final process()V          // final — нельзя переопределить
```

`ACC_ABSTRACT` на классе → JVM запрещает `new DataProcessor()` → `InstantiationError`.
`ACC_ABSTRACT` на методе → метод без тела → JVM требует реализацию в конкретном подклассе.

## sealed abstract — контроль иерархии

```java
// Java 17+: sealed + abstract — закрытое множество реализаций
public sealed abstract class Shape permits Circle, Rectangle, Triangle {
    public abstract double area();
    public abstract double perimeter();

    // switch exhaustive — компилятор проверяет все подтипы
    public String describe() {
        return switch (this) {
            case Circle c    -> "Circle r=%.1f".formatted(c.radius());
            case Rectangle r -> "Rect %dx%d".formatted(r.w(), r.h());
            case Triangle t  -> "Triangle";
        };
    }
}

public final class Circle extends Shape {
    private final double radius;
    public Circle(double radius) { this.radius = radius; }
    public double radius() { return radius; }
    @Override public double area() { return Math.PI * radius * radius; }
    @Override public double perimeter() { return 2 * Math.PI * radius; }
}
```

**Преимущество sealed:** JIT видит закрытое множество реализаций → девиртуализация switch-ветвей → нет мегаморфного dispatch.

## Когда abstract class, когда interface

```java
// ✓ Abstract class: общее состояние + конструктор + protected методы
abstract class BaseRepository<T> {
    protected final DataSource dataSource; // общее состояние

    protected BaseRepository(DataSource ds) { this.dataSource = ds; }

    protected abstract T findById(long id);  // варьируется по типу
    protected Connection getConnection() { return dataSource.getConnection(); } // общий
}

// ✓ Interface: контракт без состояния, множественные реализации
interface Repository<T> {
    T findById(long id);
    void save(T entity);
    void delete(long id);
}

// ✓ Оба: abstract class реализует interface частично
abstract class AbstractJpaRepository<T> implements Repository<T> {
    @Override
    public void delete(long id) { /* общая реализация через JPA */ }
    // findById и save — abstract, реализуют конкретные классы
}
```

## Подводные камни

- **Добавление нового abstract метода в базовый класс** — ломает все существующие подклассы (ошибка компиляции). С интерфейсами это решается `default` методом. В abstract class — только добавляй concrete методы с дефолтной реализацией.
- **`protected` методы как контракт** — подклассы могут переопределить `protected` метод, использующийся в Template Method, и сломать алгоритм. Используй `final` для шагов алгоритма, которые не должны меняться.
- **Вызов виртуального метода в конструкторе** — `abstract` конструктор вызывает метод, переопределённый в подклассе → поля подкласса ещё не инициализированы → NPE или некорректное поведение. Подробнее: [[Наследование]].
- **Глубокие иерархии abstract классов** — усложняют понимание кода. Предпочитай композицию (Strategy Pattern) вместо глубокого наследования.
