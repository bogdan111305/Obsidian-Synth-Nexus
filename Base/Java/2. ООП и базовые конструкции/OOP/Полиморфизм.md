## Основы полиморфизма

### Концепция полиморфизма

Полиморфизм — ключевая концепция ООП, позволяющая объектам разных классов обрабатываться через единый интерфейс или суперкласс. Это обеспечивает гибкость, расширяемость и повторное использование кода. В Java полиморфизм делится на статический (компиляционный) и динамический (выполнения).

### Преимущества полиморфизма

- **Гибкость**: Один интерфейс для разных реализаций
- **Расширяемость**: Легко добавлять новые типы без изменения существующего кода
- **Повторное использование**: Общий код работает с разными типами
- **Упрощение**: Снижение сложности системы

---

## Виды полиморфизма

### Статический полиморфизм (перегрузка методов)

Статический полиморфизм (compile-time polymorphism) достигается через **перегрузку методов** (method overloading), когда один метод имеет одно имя, но разные сигнатуры (параметры).

**Особенности**:
- Выбор метода происходит на этапе компиляции.
- Зависит от типов и количества аргументов.
- Не связано с наследованием или интерфейсами.

**Пример**:

```java
class Calculator {
    int add(int a, int b) {
        return a + b;
    }

    double add(double a, double b) {
        return a + b;
    }

    int add(int a, int b, int c) {
        return a + b + c;
    }
}

public class TestOverloading {
    public static void main(String[] args) {
        Calculator calc = new Calculator();
        System.out.println(calc.add(1, 2));      // 3 (int)
        System.out.println(calc.add(1.5, 2.5));  // 4.0 (double)
        System.out.println(calc.add(1, 2, 3));   // 6 (int)
    }
}
```

**Байт-код (упрощённый)**:

```java
invokevirtual Calculator.add(II)I  // Вызов add(int, int)
invokevirtual Calculator.add(DD)D  // Вызов add(double, double)
invokevirtual Calculator.add(III)I // Вызов add(int, int, int)
```

### Динамический полиморфизм (переопределение методов)

Динамический полиморфизм (runtime polymorphism) достигается через **переопределение методов** (method overriding) в подклассах или классах, реализующих интерфейсы.

**Особенности**:
- Выбор метода происходит во время выполнения (динамическое связывание).
- Используется наследование или интерфейсы.
- Реализация зависит от фактического типа объекта.

**Пример**:

```java
abstract class Animal {
    abstract void sound();
    void sleep() {
        System.out.println("Sleeping...");
    }
}

class Dog extends Animal {
    @Override
    void sound() {
        System.out.println("Bark");
    }
}

class Cat extends Animal {
    @Override
    void sound() {
        System.out.println("Meow");
    }
}

public class TestDynamicPolymorphism {
    public static void main(String[] args) {
        Animal a1 = new Dog();
        Animal a2 = new Cat();
        a1.sound(); // Bark
        a2.sound(); // Meow
        a1.sleep(); // Sleeping...
    }
}
```

**Байт-код (упрощённый)**:

```java
invokevirtual Animal.sound()V  // Динамическое связывание
invokevirtual Animal.sleep()V  // Вызов метода суперкласса
```

## Абстрактные классы и полиморфизм

### Концепция абстрактных классов

Абстрактные классы служат базой для подклассов, определяя общий интерфейс и частичную реализацию.

**Особенности**:
- Могут содержать абстрактные методы (без реализации) и методы с реализацией.
- Ссылка на абстрактный класс может указывать на объект подкласса.
- Подклассы переопределяют абстрактные методы.

**Пример**:

```java
abstract class Shape {
    abstract double area();
    void describe() {
        System.out.println("This is a shape");
    }
}

class Circle extends Shape {
    double radius;
    Circle(double radius) { this.radius = radius; }
    @Override
    double area() { return Math.PI * radius * radius; }
}

class Rectangle extends Shape {
    double width, height;
    Rectangle(double width, double height) { this.width = width; this.height = height; }
    @Override
    double area() { return width * height; }
}

public class TestShapes {
    public static void main(String[] args) {
        Shape s1 = new Circle(5);
        Shape s2 = new Rectangle(4, 6);
        System.out.println(s1.area()); // 78.539...
        System.out.println(s2.area()); // 24.0
        s1.describe(); // This is a shape
    }
}
```

## Интерфейсы и полиморфизм

### Концепция интерфейсов

Интерфейсы определяют контракт поведения, который классы реализуют. Ссылка интерфейсного типа может указывать на любой объект, реализующий интерфейс.

**Особенности**:
- Поддерживают множественное наследование поведения.
- С Java 8 включают `default` и `static` методы.
- С Java 9 поддерживают `private` методы.

**Пример**:

```java
interface Animal {
    void sound();
    default void sleep() {
        System.out.println("Sleeping...");
    }
    static void info() {
        System.out.println("Animal interface");
    }
}

class Dog implements Animal {
    public void sound() {
        System.out.println("Bark");
    }
}

class Cat implements Animal {
    public void sound() {
        System.out.println("Meow");
    }
}

public class TestInterface {
    public static void main(String[] args) {
        Animal a1 = new Dog();
        Animal a2 = new Cat();
        a1.sound(); // Bark
        a2.sound(); // Meow
        a1.sleep(); // Sleeping...
        Animal.info(); // Animal interface
    }
}
```

## JVM-реализация

### Механизмы полиморфизма

- **Статический полиморфизм**:
    - Перегрузка методов разрешается компилятором через сигнатуру (например, `invokevirtual Calculator.add(II)I`).
    - Не требует runtime-проверок.
- **Динамический полиморфизм**:
    - JVM использует **таблицу виртуальных методов (vtable)** для выбора реализации метода.
    - Инструкция `invokevirtual` определяет метод по фактическому типу объекта.
- **Байт-код (для `Animal.sound()`)**:
    
    ```java
    invokevirtual Animal.sound()V  // Динамически выбирает Dog.sound или Cat.sound
    ```
    

**Процесс**:

1. JVM загружает класс объекта в Metaspace.
2. При вызове метода проверяется vtable объекта.
3. Выбирается адрес переопределённого метода.

### Оптимизации JVM

- **Inline caching**: Кэширование адресов методов для быстрого доступа
- **Monomorphic dispatch**: Оптимизация для случаев с одним типом
- **Megamorphic dispatch**: Обработка множественных типов

## Практическое применение

### Коллекции и фреймворки

- **Коллекции**:
    
    ```java
    List<String> list = new ArrayList<>(); // Полиморфизм: List → ArrayList
    list.add("Hello");
    ```
    
- **Java EE / Jakarta EE**:
    - JAX-RS: Интерфейсы для REST API (`@Path`, `@GET`).
    - CDI: Полиморфное внедрение зависимостей (`@Inject`).
- **Spring**:
    - Интерфейсы (`@Service`, `@Repository`) для полиморфных компонентов.
- **Игровые движки**:
    - Полиморфизм для обработки объектов (например, `Entity` → `Player`, `Enemy`).

### Паттерны проектирования

- **Strategy**: Разные алгоритмы через общий интерфейс
- **Factory**: Создание объектов через общий интерфейс
- **Observer**: Уведомления через интерфейсы
- **Command**: Выполнение команд через полиморфизм

## Производительность и оптимизация

### Анализ производительности

- **Статический полиморфизм**:
    - O(1), так как выбор метода происходит на этапе компиляции.
- **Динамический полиморфизм**:
    - O(1) в среднем, но требует поиска в vtable.
    - JVM оптимизирует через **inline caching** и **monomorphic dispatch**.
- **Рекомендации**:
    - Используйте `final` для методов, не требующих переопределения, чтобы избежать vtable.
    - Минимизируйте глубокие иерархии наследования.

### Оптимизация

- **Inline-оптимизация**: JVM может встроить методы в точках вызова
- **VTable оптимизация**: Оптимизация таблицы виртуальных методов
- **Escape analysis**: Анализ выхода объектов из области видимости

## Типичные проблемы и решения

### Неправильное использование instanceof

**Проблема**:

```java
if (obj instanceof Dog) {
    Dog d = (Dog) obj; // Лишний код
}
```

**Решение**: Используйте pattern matching:

```java
if (obj instanceof Dog d) {
    d.sound();
}
```

### Перегрузка с неоднозначностью

**Проблема**:

```java
class Test {
    void method(Number n) {}
    void method(Integer i) {}
}
Test t = new Test();
t.method(42); // Неоднозначность без явного приведения
```

**Решение**: Явное приведение типов или изменение сигнатуры.

### Переопределение с неверной сигнатурой

**Проблема**:

```java
class Parent {
    void method() {}
}
class Child extends Parent {
    void method(int x) {} // Перегрузка, а не переопределение
}
```

**Решение**: Используйте `@Override`:

```java
@Override
void method() {} // Ошибка компиляции, если сигнатура неверна
```

### Default-методы в интерфейсах

**Проблема**: Конфликты при реализации нескольких интерфейсов:

```java
interface A { default void method() { System.out.println("A"); } }
interface B { default void method() { System.out.println("B"); } }
class C implements A, B {
    public void method() { A.super.method(); } // Явный выбор
}
```

## Сравнение подходов

### Абстрактные классы vs. Интерфейсы

|Критерий|Абстрактный класс|Интерфейс|
|---|---|---|
|Наследование|Один суперкласс|Множественная реализация|
|Методы|Абстрактные + методы с реализацией|Абстрактные, `default`, `static`, `private` (Java 9+)|
|Поля|Поля с состоянием|Только константы (`public static final`)|
|Конструкторы|Поддерживаются|Отсутствуют|
|Применение|Общая реализация + частичное поведение|Контракт поведения, множественное наследование|
|Гибкость|Ограничена одним суперклассом|Высокая за счёт множественной реализации|

### Типы полиморфизма

|Тип полиморфизма|Описание|Пример применения|
|---|---|---|
|Статический|Перегрузка методов, выбор на этапе компиляции|`Calculator.add(int, int)`, `Calculator.add(double, double)`|
|Динамический|Переопределение методов, выбор во время выполнения|`Animal.sound()` → `Dog.sound()`, `Cat.sound()`|

## Лучшие практики

### Основные принципы

1. **Используйте интерфейсы для гибкости**:
    
    ```java
    interface Movable { void move(); }
    class Car implements Movable { public void move() { ... } }
    ```
    
2. **Применяйте `sealed` классы для контроля**:
    
    ```java
    sealed interface Shape permits Circle, Rectangle { ... }
    ```
    
3. **Используйте pattern matching**:
    
    ```java
    if (shape instanceof Circle c) { System.out.println(c.area()); }
    ```
    
4. **Избегайте глубоких иерархий**:
    - Предпочитайте композицию над наследованием.
5. **Проверяйте переопределение**:
    
    ```java
    @Override
    void sound() { ... }
    ```
    
6. **Тестируйте полиморфизм**:
    
    ```java
    @Test
    void testPolymorphism() {
        Animal a = new Dog();
        assertEquals("Bark", captureOutput(a::sound));
    }
    ```
    

### Дополнительные рекомендации

- **Принцип подстановки Лисков**: Дочерние классы должны быть заменимы родительскими
- **Инкапсуляция**: Скрывайте детали реализации
- **Тестирование**: Пишите тесты для полиморфных компонентов
- **Документация**: Документируйте контракты интерфейсов
