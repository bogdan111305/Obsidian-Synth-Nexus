# Компоненты в Spring

**Дата создания:** 2022-12-07  
**Теги:** #book #ЧистаяАрхитектура #spring #components

## Обзор

Компоненты в Spring — это классы, которые помечены специальными аннотациями и автоматически регистрируются в Spring контейнере как бины. Это основа для создания модульной и тестируемой архитектуры.

## Основные аннотации компонентов

### @Component
Базовая аннотация для создания бинов.

```java
@Component
public class UserService {
    // Бизнес-логика
}
```

### @Service
Специализация `@Component` для сервисного слоя.

```java
@Service
public class OrderService {
    @Autowired
    private OrderRepository repository;
    
    public Order findById(Long id) {
        return repository.findById(id);
    }
}
```

### @Repository
Специализация `@Component` для слоя доступа к данным.

```java
@Repository
public class UserRepository {
    // Методы для работы с данными
}
```

### @Controller
Специализация `@Component` для веб-контроллеров.

```java
@Controller
public class UserController {
    @GetMapping("/users")
    public String getUsers() {
        return "users";
    }
}
```

### @RestController
Комбинирует `@Controller` и `@ResponseBody`.

```java
@RestController
@RequestMapping("/api/users")
public class UserRestController {
    @GetMapping("/{id}")
    public User getUser(@PathVariable Long id) {
        return userService.findById(id);
    }
}
```

## Принципы связности компонентов

### 3 принципа связности:

**REP** : Reuse/Release Equivalence Principle - принцип эквивалентности повторного использования и выпусков.  
**CCP** : Common Closure Principle - принцип согласованного изменения.  
**CRP** : Common Reuse Principle - принцип совместного повторного использования.

#### REP (Reuse/Release Equivalence Principle)

> Единица повторного использования есть единица выпуска

Классы и модули, объединяемые в компонент, должны выпускаться вместе.

**Пример в Spring:**
```java
@Component
public class EmailService {
    // Все методы для работы с email в одном компоненте
    public void sendEmail(String to, String subject, String body) { }
    public void sendTemplate(String to, String template) { }
    public void sendBulk(List<String> recipients, String subject, String body) { }
}
```

#### CCP (Common Closure Principle)

> В один компонент должны включаться классы, изменяющиеся по одним причинам и в одно время. В разные компоненты должны включаться классы, изменяющиеся в разное время и по разным причинам.

По сути это принцип единой ответственности, перефразированный для компонентов.

**Пример в Spring:**
```java
// Все классы, связанные с безопасностью
@Component
public class SecurityService { }

@Component
public class AuthenticationService { }

@Component
public class AuthorizationService { }
```

#### CRP (Common Reuse Principle)

> Не вынуждайте пользователей компонента зависеть от того, что им не требуется

По сути это принцип разделения интерфейсов для компонента.

**Пример в Spring:**
```java
// Разделение интерфейсов
public interface UserReadService {
    User findById(Long id);
    List<User> findAll();
}

public interface UserWriteService {
    User save(User user);
    void delete(Long id);
}

@Service
public class UserService implements UserReadService, UserWriteService {
    // Реализация
}
```

## Принцип цикличности зависимостей

Когда одна команда заканчивает с компонентом и он будет внедряться в систему, будет правильным, если у него будет как можно меньше зависимостей. Принцип говорит о том, что структура зависимостей должна представлять *ациклический направленный граф*.

### Решения проблем образования циклов

Когда один компонент нежелательно зависит от другого:

1. **Принцип инверсии зависимостей** - изменить направление зависимости путём добавления абстракции.

```java
// Плохо - циклическая зависимость
@Service
public class OrderService {
    @Autowired
    private UserService userService;
}

@Service
public class UserService {
    @Autowired
    private OrderService orderService;
}

// Хорошо - инверсия зависимостей
public interface OrderRepository {
    Order save(Order order);
}

@Service
public class OrderService {
    private final OrderRepository orderRepository;
    
    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }
}
```

2. **Создать третий компонент**, от которого будут зависеть оба компонента.

```java
// Общий интерфейс
public interface NotificationService {
    void sendNotification(String message);
}

@Service
public class EmailNotificationService implements NotificationService {
    @Override
    public void sendNotification(String message) {
        // Отправка email
    }
}

@Service
public class OrderService {
    private final NotificationService notificationService;
    
    public OrderService(NotificationService notificationService) {
        this.notificationService = notificationService;
    }
}

@Service
public class UserService {
    private final NotificationService notificationService;
    
    public UserService(NotificationService notificationService) {
        this.notificationService = notificationService;
    }
}
```

## Устойчивость зависимостей

Принцип устойчивых зависимостей - модули с трудом поддающиеся изменениям не будут зависеть от модулей, спроектированных для упрощения изменений.

Устойчивость - простота изменения компонента.

$$ I = \frac{F_{out}}{F_{in} + F_{out}} $$

Где:
- $F_{in}$ - количество входящих зависимостей
- $F_{out}$ - количество исходящих зависимостей
- $I$ - индекс неустойчивости (Instability)

### Примеры в Spring

#### Устойчивый компонент (мало зависимостей)
```java
@Service
public class EmailService {
    // Минимум зависимостей, максимум функциональности
    public void sendEmail(String to, String subject, String body) {
        // Реализация
    }
}
```

#### Неустойчивый компонент (много зависимостей)
```java
@Service
public class OrderProcessingService {
    @Autowired
    private OrderRepository orderRepository;
    @Autowired
    private UserService userService;
    @Autowired
    private PaymentService paymentService;
    @Autowired
    private NotificationService notificationService;
    @Autowired
    private InventoryService inventoryService;
    // Много зависимостей = неустойчивый компонент
}
```

## Лучшие практики для компонентов Spring

### 1. Принцип единственной ответственности
```java
// Хорошо - каждый компонент отвечает за одну область
@Service
public class UserService {
    // Только операции с пользователями
}

@Service
public class OrderService {
    // Только операции с заказами
}
```

### 2. Инверсия зависимостей
```java
// Хорошо - зависимость от абстракции
@Service
public class OrderService {
    private final OrderRepository orderRepository;
    
    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }
}
```

### 3. Разделение интерфейсов
```java
// Хорошо - разделение ответственности
public interface UserReadService {
    User findById(Long id);
}

public interface UserWriteService {
    User save(User user);
}

@Service
public class UserService implements UserReadService, UserWriteService {
    // Реализация
}
```

### 4. Использование профилей
```java
@Service
@Profile("dev")
public class DevUserService implements UserService {
    // Реализация для разработки
}

@Service
@Profile("prod")
public class ProdUserService implements UserService {
    // Реализация для продакшена
}
```
