# Наследование в Hibernate

Есть 2 общих возможных подхода к наследованию - это когда суперкласс не является отдельной сущностью Entity и когда является  

## Суперкласс не является entity

В таком случае нам достаточно всего-лишь указать аннотацию <i style="color:#f3ff00">@MappedSuperclass</i> над суперклассом. 
Теперь при записи, в таблице Person также будут поля *createdAt* и *createdBy*:

``` Java
@MappedSuperclass  
public abstract class AuditableEntity{  
    private Instant createdAt;  
    private String createdBy;  
}
```

``` Java
@Entity
public class Person extends AuditableEntity<Integer> {  
    @Id  
    @GeneratedValue(strategy = GenerationType.IDENTITY)  
    private Integer id;  
    private String name;  
}
```


## Суперкласс это entity

Тут есть 3 стратегии наследования, в зависимости от того, как они будут отображаться в таблицах:
![[image.png]]

Над сущностью которая будет наследоваться ставится аннотация <i style="color:#f3ff00">@Inheritance</i>, в аргументах которой указывается выбранная стратегия.


Для примера представим: 
*Person* - сущность которая будет наследоваться, абстрактна и не имеет собственной таблицы
*Programmer* - наследник
*Manager* - наследник

### Table Per Class
каждый из наследников будет создаваться в своей таблице, однако общие поля (из Users) будут дублироваться 

``` Java
@Entity   
@Inheritance(strategy = InheritanceType.TABLE_PER_CLASS)  
public abstract class Person extends AuditableEntity<Integer> {  
    @Id  
    @GeneratedValue(strategy = GenerationType.SEQUENCE)  
    private Integer id;  
    private String name;  
}

@Entity
public class Programmer extends Person{  
    private String language;  
}

@Entity  
public class Manager extends Person{  
    private String projectName;  
}
```

**Почему sequence?**  - потому что по сути у них будет id браться из общей последовательности - чтобы можно было приводить к сущности *Person*

`Person person = session.get(Person.class, 1)`

минусы: 
-   приходится дублировать общие поля
-   использование union all при получении объекта суперкласса    
-   вы выборе стратегии генерации id ограничиваемся Sequence, и он общий для всех таблиц
плюсы:
- если нужен какой-то конкретный наследник, то мы обращаемся только к нужной таблице

### Single Table

Общая таблица для всех наследников, которая содержит: 
- общие поля из супер класса
- поля каждого из наследников
- поле type, для понимания, какой класс необходимо создать при get

В реализации данной стратегии активно учавствует аннотации: 
<i style="color:#f3ff00">@DiscriminatorColumn</i> - как будет называться колонка с типом класса, ставится в суперклассе
<i style="color:#f3ff00">@DiscriminatorValue</i> - ставится в насследниках, как будет называться тип этого класса 

*эти аннотации optional, так как если их не будет hibernate сам придумает названия этим вещам*
``` Java
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)  
@DiscriminatorColumn(name = "type")  
public class Person extends AuditableEntity<Integer> {  
    @Id  
    @GeneratedValue(strategy = GenerationType.IDENTITY)  
    private Integer id;  
    private String name;  
}

@Entity
@DiscriminatorValue("manager")
public class Programmer extends Person{  
    private String language;  
}

@Entity 
@DiscriminatorValue("programmer")
public class Manager extends Person{  
    private String projectName;  
}
```
При select запросах в таблицу hibernate добавляет select по type.

минусы: 
- не можем давать constraint на поля
- при оращении к конкретному наследнику происходит поиск по всему множеству

плюсы: 
- нет union all - работает быстрее 
- можно использовать генерацию id IDENTITY
### Joined

При такой стратегии создаётся несколько таблиц:
- таблица сущности суперкласса
- таблицы наследников, где id - одновременно и PK и FK на общую таблицу users

 Аннотация <i style="color:#f3ff00">@PrimaryKeyJoinColumn(name = "id")</i> в наследниках, говорит о том как назвать колонку с ключом.

``` JAVA
@Entity   
@Inheritance(strategy = InheritanceType.JOINED)  
public abstract class Person extends AuditableEntity<Integer> {  
  
    @Id  
    @GeneratedValue(strategy = GenerationType.SEQUENCE)  
    private Integer id;  
    private String name;  
}

@Entity
@PrimaryKeyJoinColumn(name = "id")
public class Programmer extends Person{  
    private String language;  
}

@Entity  
@PrimaryKeyJoinColumn(name = "id")
public class Manager extends Person{  
    private String projectName;  
}
```

минусы: 
- худшая производительность из-за dml команд, все они создают join на все таблицы 

плюсы: 
- используются нормальные формы
