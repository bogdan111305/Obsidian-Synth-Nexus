# JDBC vs ORM: Сравнение подходов

> [!QUOTE] JDBC — низкоуровневый API: SQL пишется вручную, контроль максимальный. ORM (Hibernate, JPA) — абстракция над БД: классы ↔ таблицы, SQL генерируется автоматически.

## Таблица сравнения

| Критерий | JDBC | ORM (Hibernate, JPA) |
|----------|------|-----------------------|
| Производительность | Высокая (ручной контроль) | Высокая при кэше и batch |
| Объём кода | Много шаблонного кода | Декларативно, меньше кода |
| Гибкость SQL | Максимальная | Ограничена абстракцией |
| Транзакции | Ручное управление | Автоматизация |
| Кэширование | Нет | Есть (1st / 2nd level cache) |
| Миграции | Нет | Liquibase, Flyway |
| Поддержка БД | Любая | Зависит от реализации |
| Batch | Ручная настройка | Полуавтоматически |
| Lazy loading | Нет | Есть |
| Отладка SQL | Явно виден | Генерируется, нужен логгер |

## Плюсы и минусы

### JDBC
- Максимальный контроль над SQL
- Прозрачность: видно каждый запрос
- Минимум зависимостей и "магии"
- Много шаблонного кода (открытие/закрытие ресурсов, маппинг)
- Ручное управление транзакциями и соединениями
- Нет кэширования, нет миграций

### ORM
- Быстрая разработка, меньше кода
- Кэширование, ленивые коллекции, миграции
- Автоматическое управление транзакциями
- Сложность отладки (неочевидные SQL-запросы)
- Overhead при сложных запросах
- N+1 проблема при неаккуратном использовании

> [!WARNING] ORM не исключает знание SQL. Неочевидные запросы (N+1, декартово произведение) могут убить производительность. Всегда логируйте генерируемый SQL в режиме разработки.

## Когда выбирать JDBC, когда ORM

**Выбирайте JDBC:**
- Критичные к производительности участки (аналитика, bulk-операции)
- Интеграция с legacy-схемами БД
- Полный контроль над запросами обязателен
- Простые скрипты и утилиты

**Выбирайте ORM:**
- Сложные доменные модели с богатыми связями
- Быстрое прототипирование
- Нужны кэширование, lazy loading, миграции
- Команда уже знает Hibernate/JPA

> [!INFO] JDBC и ORM можно комбинировать в одном проекте: ORM для бизнес-логики, JDBC или `@Query` с нативным SQL для сложных отчётов и оптимизированных операций.

---

## Best practices

- Для сложных отчётов и аналитики — JDBC или нативные запросы
- Для CRUD и бизнес-объектов — ORM
- Логируйте SQL и профилируйте запросы в обоих подходах
- В Spring: `JdbcTemplate` (JDBC) и `JpaRepository` (ORM) легко сосуществуют
