![[Pasted image 20230302212440.png]]

Практически все миграционные фреймворки работают схожим образом.

2 таблицы: 
1. таблица для лока бд 
2. таблица для логов и мета информации о наших скриптах 

##### Как работает: 
 1. Сначала **выполняется pessimistick lock,** если таблица лок залочена, то не один другой процесс не сможет работать с бд. То есть по сути наш процесс блокирует бд от других.
 2. Дальше идёт работа с log таблицей: 
    - Ключ, по которому **определяется накатывать скрипт или** нет это *id*, *author* и *filename* 
      (если по таким полям ещё нету записи, то вставь таблицу)
    -  md-five-sum  -- чек сумма нашего скрипта, если скрипт, после накатывания изменился выбросится исключение  

## Основные понятия

![[Pasted image 20230302213412.png]]

**Changelog** - скрипт, который должен накатится на базу данных (может быть в разных форматах). 
Для определения форматов иногда необходимо указать комментарий (например SQL) 

**Changeset** - над set ставится комментарий с автором и id (может содержать и другие данные, которые зарезервированы).  
id уникален в рамках changelog-а. 
Каждый *changset выполняется в рамках одной транзакции* 

**Change** - непосредственно скрипт 

best_practies: *1 change в 1 changeset*
