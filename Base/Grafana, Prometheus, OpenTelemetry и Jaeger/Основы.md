> [!QUOTE] Observability (наблюдаемость)
> Три столпа: **Логи** (что произошло), **Метрики** (как работает система в числах), **Трейсы** (как запрос прошёл через сервисы). Вместе они дают полную картину состояния приложения.

## Логи

> [!INFO] Что такое логи
> Записи событий приложения: ошибки, предупреждения, информационные сообщения. Используются для диагностики и аудита.

### Настройка в Spring Boot

Spring Boot использует **Logback** по умолчанию.

`src/main/resources/application.properties`:
```properties
logging.level.root=INFO
logging.level.com.example=DEBUG
logging.file.name=app.log
```

Зависимость уже включена в `spring-boot-starter`. Использование в коде:

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class MyService {
    private static final Logger logger = LoggerFactory.getLogger(MyService.class);

    public void doSomething() {
        logger.info("Выполняется операция...");
        try {
            // логика
        } catch (Exception e) {
            logger.error("Ошибка: ", e);
        }
    }
}
```

### Механика

- **Уровни**: TRACE → DEBUG → INFO → WARN → ERROR. Все уровни ниже заданного фильтруются.
- **Конфиг**: через `application.properties` или `logback.xml`.
- **Назначение вывода**: консоль, файл, внешняя система (ELK Stack).
- **Асинхронность**: настраивается для снижения задержек в приложении.
- **Actuator**: Spring Boot экспортирует логи через `/actuator` в централизованные системы.

### Лучшие практики

- Используйте уровни осознанно: `DEBUG` — для разработки, `INFO`/`WARN`/`ERROR` — для продакшена.
- Применяйте структурированные логи (JSON) — проще парсить и искать.
- Храните логи централизованно (ELK Stack, Loki).

> [!WARNING] Избыточные логи
> Слишком подробное логирование на уровне INFO/DEBUG в продакшене увеличивает нагрузку на диск и снижает производительность. Контролируйте уровень через конфиг без перезапуска (Actuator позволяет менять уровень логирования на лету).

## Метрики: Prometheus + Grafana

> [!INFO] Что такое метрики
> Количественные данные о работе системы: время ответа, количество запросов, использование памяти. Prometheus собирает, Grafana визуализирует.

### Prometheus — установка и запуск

1. Скачать: [https://prometheus.io/download/](https://prometheus.io/download/)
2. Настроить `prometheus.yml`:
```yaml
scrape_configs:
  - job_name: 'spring-app'
    static_configs:
      - targets: ['localhost:8080']
```
3. Запустить: `./prometheus --config.file=prometheus.yml`

### Интеграция с Spring Boot

`build.gradle`:
```groovy
implementation 'io.micrometer:micrometer-registry-prometheus'
```

`application.properties`:
```properties
management.endpoints.web.exposure.include=prometheus
management.endpoint.prometheus.enabled=true
```

Метрики доступны по эндпоинту: `/actuator/prometheus`

### Grafana — установка и настройка

1. Скачать: [https://grafana.com/get](https://grafana.com/get)
2. Запустить: `./grafana-server`
3. Открыть: `http://localhost:3000`
4. Добавить Data Source: выбрать **Prometheus**, указать URL `http://localhost:9090`
5. Создать дашборд с PromQL-запросами, например:
```promql
rate(http_server_requests_seconds_count[5m])
```

### Механика метрик

- **Pull-модель**: Prometheus сам опрашивает эндпоинты каждые 15 секунд (настраивается).
- **Формат**: временные ряды (time series) с метками (labels).
- **Хранение**: локальная БД Prometheus, по умолчанию до 15 дней.
- **PromQL**: язык запросов для агрегации и фильтрации данных.
- **Встроенные метрики**: `http_server_requests`, `jvm_memory_used` и др. — через Spring Boot Actuator + Micrometer.
- **Алертинг**: Prometheus AlertManager + Grafana Alerting — уведомления в email, Slack, PagerDuty.

## Трейсинг: OpenTelemetry + Jaeger

> [!INFO] Что такое трейсинг
> Отслеживание пути запроса через всю цепочку сервисов. Помогает найти узкие места и локализовать ошибки в распределённых системах.

**Ключевые понятия:**
- **Трейс (trace)** — полный путь одного запроса от входа до ответа.
- **Спан (span)** — единица работы внутри трейса (например, один HTTP-вызов, один SQL-запрос).
- **Контекст (propagation context)** — передаётся через заголовки (`traceparent`) между сервисами для связывания спанов.

### OpenTelemetry — установка

`build.gradle`:
```groovy
implementation 'io.opentelemetry:opentelemetry-api:1.42.0'
implementation 'io.opentelemetry:opentelemetry-sdk:1.42.0'
```

Настройка экспортера в Jaeger:
```java
import io.opentelemetry.api.OpenTelemetry;
import io.opentelemetry.sdk.OpenTelemetrySdk;
import io.opentelemetry.exporters.jaeger.JaegerGrpcSpanExporter;

public class TelemetryConfig {
    public static OpenTelemetry init() {
        JaegerGrpcSpanExporter exporter = JaegerGrpcSpanExporter.builder()
            .setEndpoint("http://localhost:14250")
            .build();
        return OpenTelemetrySdk.builder()
            .setTracerProvider(SdkTracerProvider.builder()
                .addSpanProcessor(SimpleSpanProcessor.create(exporter))
                .build())
            .buildAndRegisterGlobal();
    }
}
```

Инициализация в `main`:
```java
public static void main(String[] args) {
    TelemetryConfig.init();
    SpringApplication.run(Application.class, args);
}
```

### Jaeger — установка

```bash
docker run -d -p 16686:16686 -p 14268:14268 jaegertracing/all-in-one:latest
```

UI: `http://localhost:16686`

### Механика трейсинга

- **Создание спанов**: OpenTelemetry инструментирует код и создаёт спаны с ID, временными метками, атрибутами.
- **Связывание**: через заголовок `traceparent` — спаны из разных сервисов объединяются в один трейс.
- **Экспорт**: асинхронная отправка в Jaeger через gRPC или HTTP.
- **Хранение**: Jaeger хранит данные в Cassandra или Elasticsearch.
- **Визуализация**: Jaeger UI показывает задержки, ошибки, вложенность вызовов.

### Лучшие практики

- Добавляйте корреляционный ID в каждый сервис.
- Используйте семантические атрибуты: `http.method`, `db.operation`, `messaging.system`.

## Архитектура стека

| Инструмент | Роль |
|---|---|
| **Prometheus** | Сбор метрик (pull-модель) |
| **Grafana** | Визуализация метрик через PromQL |
| **OpenTelemetry** | Генерация и экспорт трейсов и метрик |
| **Jaeger** | Хранение и отображение трейсов |

**Поток данных:**
1. Приложение → экспортирует метрики на `/actuator/prometheus` и трейсы в Jaeger.
2. Prometheus → scrape метрик каждые 15 сек.
3. Grafana → PromQL-запросы к Prometheus → дашборды.
4. Jaeger → сборка трейсов → анализ задержек.
5. Алертинг → Prometheus AlertManager / Grafana уведомляют при нарушении порогов.
