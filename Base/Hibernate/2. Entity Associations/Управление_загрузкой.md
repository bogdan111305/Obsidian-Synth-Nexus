## <a name="введение"></a>Введение в управление загрузкой

Управление загрузкой данных в Hibernate определяет, когда и как связанные данные загружаются из базы данных. Это критически важный аспект для оптимизации производительности приложения.

### Ключевые концепции

- **Стратегии загрузки**: EAGER (немедленная) и LAZY (ленивая)
- **Прокси-объекты**: Механизм для реализации ленивой загрузки
- **N+1 проблема**: Частая проблема производительности
- **JOIN FETCH**: Оптимизация запросов для загрузки связанных данных

---

## <a name="fetch-types"></a>Fetch Types - Типы выборки

Тип выборки (`fetch`) определяет, когда связанные данные загружаются из базы данных. Hibernate поддерживает два типа выборки: `EAGER` и `LAZY`.

### FetchType.EAGER - Немедленная загрузка

Данные загружаются сразу при загрузке основной сущности.

#### Характеристики EAGER загрузки:

- **Время загрузки**: Данные загружаются немедленно
- **Количество запросов**: Может увеличить количество запросов к БД
- **Использование памяти**: Больше данных в памяти одновременно
- **Производительность**: Может привести к проблемам при больших наборах данных

#### Пример EAGER загрузки:

```java
@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @Column(name = "username")
    private String username;
    
    @ManyToOne(fetch = FetchType.EAGER)
    @JoinColumn(name = "company_id")
    private Company company;
    
    // Геттеры и сеттеры
}
```

#### SQL-запрос при EAGER загрузке:

```sql
SELECT u.id, u.username, c.id, c.name 
FROM users u 
LEFT JOIN companies c ON u.company_id = c.id 
WHERE u.id = ?
```

### FetchType.LAZY - Ленивая загрузка

Данные загружаются только при первом обращении к связанному объекту.

#### Характеристики LAZY загрузки:

- **Время загрузки**: Данные загружаются по требованию
- **Количество запросов**: Меньше начальных запросов
- **Использование памяти**: Экономия памяти
- **Производительность**: Более эффективно для больших наборов данных

#### Пример LAZY загрузки:

```java
@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @Column(name = "username")
    private String username;
    
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "company_id")
    private Company company;
    
    // Геттеры и сеттеры
}
```

#### SQL-запросы при LAZY загрузке:

```sql
-- Первый запрос - загрузка пользователя
SELECT id, username, company_id FROM users WHERE id = ?

-- Второй запрос - загрузка компании при обращении
SELECT id, name FROM companies WHERE id = ?
```

### Значения по умолчанию

- **@OneToOne**: `FetchType.EAGER`
- **@ManyToOne**: `FetchType.EAGER`
- **@OneToMany**: `FetchType.LAZY`
- **@ManyToMany**: `FetchType.LAZY`

## <a name="optional"></a>Optional параметр

Параметр `optional` в аннотации `@ManyToOne` указывает, может ли связанная сущность быть `null`.

### optional = true (по умолчанию)

Связь может быть `null`, то есть пользователь может не принадлежать никакой компании.

```java
@ManyToOne(optional = true)
@JoinColumn(name = "company_id")
private Company company;
```

### optional = false

Связь обязательна, и Hibernate будет ожидать, что внешний ключ всегда указывает на существующую запись.

```java
@ManyToOne(optional = false)
@JoinColumn(name = "company_id", nullable = false)
private Company company;
```

### Обработка исключений

Если `optional = false`, попытка сохранить `User` с `company = null` вызовет исключение:

```java
try {
    User user = new User("john_doe", "john@example.com");
    user.setCompany(null); // Вызовет исключение
    em.persist(user);
} catch (ConstraintViolationException e) {
    System.err.println("Company is required for user");
}
```

## <a name="proxy"></a>Hibernate Proxy - Прокси-объекты

Hibernate использует прокси-объекты для реализации ленивой загрузки, что позволяет избежать ненужных запросов к базе данных.

### Принцип работы прокси

1. **Создание "заглушки"**: Hibernate создает прокси-объект без загрузки данных
2. **Перехват вызовов**: Прокси перехватывает вызовы методов
3. **Отложенная загрузка**: При первом обращении выполняется SQL-запрос
4. **Прозрачность**: Клиентский код не знает о прокси

### Типы прокси

#### Dynamic Proxy (через интерфейсы)

Hibernate создаёт прокси-объекты, которые выступают как "заглушки" для реальных сущностей. Эти прокси могут быть основаны на интерфейсах.

```java
// Если сущность реализует интерфейс
public interface Company {
    Long getId();
    String getName();
}

@Entity
public class CompanyImpl implements Company {
    // Реализация
}
```

#### Proxy через наследование (ByteBuddy)

Начиная с Hibernate 5.2, для создания прокси-объектов используется библиотека **ByteBuddy**. Она генерирует подкласс сущности, который перехватывает вызовы методов для выполнения ленивой загрузки.

```java
@Entity
public class Company {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String name;
    
    // Геттеры и сеттеры
}
```

### ByteBuddyInterceptor

`ByteBuddyInterceptor` — внутренний механизм Hibernate, который обрабатывает вызовы методов прокси-объекта. Он определяет, когда нужно загрузить данные из базы, и передаёт управление реальной сущности.

### Инициализация Proxy объекта

Прокси-объект остаётся неинициализированным, пока не будет вызван метод, требующий данных.

```java
User user = em.find(User.class, 1L);
// На этом этапе company - это прокси, данные не загружены

Company company = user.getCompany();
// При первом обращении к company выполняется SQL-запрос

String companyName = company.getName();
// Данные уже загружены, дополнительных запросов нет
```

## <a name="утилитный-класс"></a>Утилитный класс Hibernate

Утилитный класс `org.hibernate.Hibernate` предоставляет методы для работы с прокси-объектами.

### Hibernate.initialize(Object)

Принудительно инициализирует прокси или коллекцию, выполняя запрос к базе.

```java
User user = em.find(User.class, 1L);
Hibernate.initialize(user.getCompany()); // Принудительная инициализация

// Теперь company полностью загружен
System.out.println(user.getCompany().getName());
```

### Hibernate.isInitialized(Object)

Проверяет, инициализирован ли объект или коллекция.

```java
User user = em.find(User.class, 1L);
Company company = user.getCompany();

if (Hibernate.isInitialized(company)) {
    System.out.println("Company is already loaded");
} else {
    System.out.println("Company is a proxy");
    Hibernate.initialize(company);
}
```

### Полный пример использования

```java
import org.hibernate.Hibernate;

public class ProxyExample {
    public static void main(String[] args) {
        EntityManager em = // получение EntityManager
        
        User user = em.find(User.class, 1L);
        
        // Проверка статуса прокси
        Company company = user.getCompany();
        System.out.println("Is company initialized: " + Hibernate.isInitialized(company));
        
        // Принудительная инициализация
        Hibernate.initialize(company);
        System.out.println("Is company initialized: " + Hibernate.isInitialized(company));
        
        // Теперь можно безопасно использовать company
        System.out.println("Company name: " + company.getName());
    }
}
```

## <a name="примеры"></a>Практические примеры

### Пример 1: Сравнение EAGER и LAZY

```java
// EAGER загрузка
@Entity
public class User {
    @ManyToOne(fetch = FetchType.EAGER)
    @JoinColumn(name = "company_id")
    private Company company;
}

// LAZY загрузка
@Entity
public class User {
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "company_id")
    private Company company;
}

// Тестирование
public class FetchTypeTest {
    public static void main(String[] args) {
        EntityManager em = // получение EntityManager
        
        // EAGER - данные загружаются сразу
        User eagerUser = em.find(User.class, 1L);
        // SQL: SELECT u.*, c.* FROM users u LEFT JOIN companies c ON u.company_id = c.id WHERE u.id = 1
        
        // LAZY - данные загружаются при обращении
        User lazyUser = em.find(User.class, 2L);
        // SQL: SELECT u.* FROM users u WHERE u.id = 2
        
        Company company = lazyUser.getCompany(); // Дополнительный запрос
        // SQL: SELECT c.* FROM companies c WHERE c.id = ?
    }
}
```

### Пример 2: Обработка LazyInitializationException

```java
public class LazyInitializationExample {
    public static void main(String[] args) {
        EntityManager em = // получение EntityManager
        
        try {
            User user = em.find(User.class, 1L);
            em.close(); // Сессия закрыта
            
            // Это вызовет LazyInitializationException
            String companyName = user.getCompany().getName();
        } catch (LazyInitializationException e) {
            System.err.println("Cannot access lazy-loaded association outside session");
        }
    }
}
```

### Пример 3: Решение проблемы LazyInitializationException

```java
public class LazyInitializationSolution {
    public static void main(String[] args) {
        EntityManager em = // получение EntityManager
        
        // Вариант 1: Использование JOIN FETCH
        User user = em.createQuery(
            "SELECT u FROM User u JOIN FETCH u.company WHERE u.id = :id", User.class)
            .setParameter("id", 1L)
            .getSingleResult();
        
        em.close();
        
        // Теперь можно безопасно обращаться к company
        String companyName = user.getCompany().getName();
        
        // Вариант 2: Принудительная инициализация
        EntityManager em2 = // получение нового EntityManager
        
        User user2 = em2.find(User.class, 1L);
        Hibernate.initialize(user2.getCompany());
        em2.close();
        
        // Теперь можно безопасно обращаться к company
        String companyName2 = user2.getCompany().getName();
    }
}
```

### Пример 4: Оптимизация с помощью JOIN FETCH

```java
public class JoinFetchExample {
    public static void main(String[] args) {
        EntityManager em = // получение EntityManager
        
        // Проблемный запрос (N+1)
        List<User> users = em.createQuery("SELECT u FROM User u", User.class)
            .getResultList();
        
        // Для каждого пользователя будет отдельный запрос для загрузки компании
        for (User user : users) {
            System.out.println(user.getCompany().getName()); // N+1 проблема
        }
        
        // Оптимизированный запрос с JOIN FETCH
        List<User> optimizedUsers = em.createQuery(
            "SELECT u FROM User u JOIN FETCH u.company", User.class)
            .getResultList();
        
        // Все данные загружены одним запросом
        for (User user : optimizedUsers) {
            System.out.println(user.getCompany().getName()); // Нет дополнительных запросов
        }
    }
}
```

### Пример 5: Работа с коллекциями

```java
@Entity
public class Department {
    @OneToMany(mappedBy = "department", fetch = FetchType.LAZY)
    private List<Employee> employees = new ArrayList<>();
}

public class CollectionExample {
    public static void main(String[] args) {
        EntityManager em = // получение EntityManager
        
        Department dept = em.find(Department.class, 1L);
        
        // Проверка инициализации коллекции
        if (Hibernate.isInitialized(dept.getEmployees())) {
            System.out.println("Employees collection is loaded");
        } else {
            System.out.println("Employees collection is a proxy");
            Hibernate.initialize(dept.getEmployees());
        }
        
        // Теперь коллекция полностью загружена
        for (Employee emp : dept.getEmployees()) {
            System.out.println(emp.getName());
        }
    }
}
```

### 1. Выбор стратегии загрузки

```java
// Используйте LAZY для больших коллекций
@OneToMany(mappedBy = "department", fetch = FetchType.LAZY)
private List<Employee> employees;

// Используйте EAGER для часто используемых связей
@ManyToOne(fetch = FetchType.EAGER)
@JoinColumn(name = "company_id")
private Company company;
```

### 2. Использование JOIN FETCH

```java
// Избегайте N+1 проблем
List<User> users = em.createQuery(
    "SELECT u FROM User u JOIN FETCH u.company", User.class)
    .getResultList();
```

### 3. Обработка LazyInitializationException

```java
try {
    User user = em.find(User.class, 1L);
    em.close();
    
    // Это вызовет исключение
    String companyName = user.getCompany().getName();
} catch (LazyInitializationException e) {
    // Используйте JOIN FETCH или Hibernate.initialize()
    System.err.println("Use JOIN FETCH or Hibernate.initialize()");
}
```

### 4. Мониторинг SQL-запросов

```properties
# В persistence.xml или application.properties
hibernate.show_sql=true
hibernate.format_sql=true
hibernate.use_sql_comments=true
```

### 5. Использование Entity Graph

```java
@Entity
public class User {
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "company_id")
    private Company company;
}

// Использование Entity Graph
@EntityGraph(attributePaths = {"company"})
@Query("SELECT u FROM User u")
List<User> findAllWithCompany();
```

### 6. Batch Loading

```java
@Entity
public class Company {
    @OneToMany(mappedBy = "company")
    @BatchSize(size = 20)
    private List<User> users;
}
```

### 7. Проверка статуса прокси

```java
public void processUser(User user) {
    Company company = user.getCompany();
    
    if (!Hibernate.isInitialized(company)) {
        // Данные не загружены, загружаем принудительно
        Hibernate.initialize(company);
    }
    
    // Теперь можно безопасно использовать company
    System.out.println("Company: " + company.getName());
}
```

## Senior-insights: ByteBuddy proxy и LazyInitializationException

### Как ByteBuddy генерирует proxy для lazy loading

Начиная с Hibernate 5.2, для lazy-ассоциаций Hibernate использует **ByteBuddy** — библиотеку генерации байткода. При обращении к lazy-ассоциации Hibernate возвращает не реальный объект, а **динамически сгенерированный подкласс**.

```
User.company (FetchType.LAZY) → возвращается CompanyProxy extends Company
CompanyProxy:
  - поля: все пустые (null / 0)
  - методы: при вызове → если не инициализирован → SELECT → делегирование реальному объекту
  - поле __hibernateInitializer: ссылка на ByteBuddyInterceptor
```

`ByteBuddyInterceptor` перехватывает все вызовы методов и при первом обращении:
1. Проверяет, открыта ли сессия
2. Если открыта — выполняет `SELECT` и инициализирует proxy
3. Если закрыта — выбрасывает `LazyInitializationException`

### Почему entity.getClass() на proxy вернёт не реальный класс

```java
Company proxy = user.getCompany(); // FetchType.LAZY

// НЕПРАВИЛЬНО:
proxy.getClass();                     // → Company$HibernateProxy$xyz (подкласс!)
proxy.getClass() == Company.class;    // → false!

// ПРАВИЛЬНО:
Hibernate.getClass(proxy);            // → Company (реальный класс)
proxy instanceof Company;             // → true (proxy наследует Company)
```

> [!WARNING]
> **Никогда не используй `entity.getClass()` для определения типа** в коде с Hibernate proxy! Используй `instanceof` или `Hibernate.getClass()`.

```java
// Частая ошибка в equals():
@Override
public boolean equals(Object o) {
    if (getClass() != o.getClass()) return false; // СЛОМАЕТСЯ с proxy!
    // ...
}

// Правильно:
@Override
public boolean equals(Object o) {
    if (!(o instanceof Company other)) return false; // proxy наследует Company → работает
    // ...
}
```

### instanceof работает корректно

`instanceof` проверяет иерархию наследования. Поскольку ByteBuddy-proxy является **подклассом** реальной entity, `instanceof` всегда возвращает `true`:

```java
Company proxy = user.getCompany();
System.out.println(proxy instanceof Company); // true — proxy extends Company
```

### LazyInitializationException: причина и решения

**Причина**: попытка обратиться к lazy-ассоциации после закрытия сессии.

```java
User user = entityManager.find(User.class, 1L);
entityManager.close(); // сессия закрыта

user.getCompany().getName(); // → LazyInitializationException!
// ByteBuddyInterceptor пытается загрузить данные → сессия закрыта → исключение
```

**Решение 1: JOIN FETCH (рекомендуется)**

```java
// Загружаем всё нужное за один запрос
User user = entityManager.createQuery(
    "SELECT u FROM User u JOIN FETCH u.company WHERE u.id = :id", User.class)
    .setParameter("id", 1L)
    .getSingleResult();
// Теперь company уже загружена — proxy инициализирован
entityManager.close();
user.getCompany().getName(); // OK
```

**Решение 2: @Transactional на уровне сервиса (правильная архитектура)**

```java
@Service
public class UserService {

    @Transactional // сессия живёт на протяжении всего метода
    public UserDto getUserWithCompany(Long id) {
        User user = userRepository.findById(id).orElseThrow();
        // внутри @Transactional сессия открыта — lazy loading работает
        String companyName = user.getCompany().getName(); // OK
        return new UserDto(user.getId(), user.getEmail(), companyName);
        // после выхода из метода → сессия закрыта, DTO уже собран
    }
}
```

**Решение 3: Hibernate.initialize() (для точечных случаев)**

```java
@Transactional
public User getUserWithInitializedCompany(Long id) {
    User user = entityManager.find(User.class, id);
    Hibernate.initialize(user.getCompany()); // принудительная инициализация
    return user; // company уже загружена — можно использовать вне транзакции
}
```

> [!WARNING]
> **Open Session in View (OSIV)** — антипаттерн: держать сессию открытой на всё время HTTP-запроса. В Spring Boot включён по умолчанию (`spring.jpa.open-in-view=true`). Выключи его: `spring.jpa.open-in-view=false`. Проблемы OSIV: скрытые N+1 запросы в view-слое, долгие соединения с БД, сложность отладки.

### Hibernate.initialize() — когда применять

```java
// Применяй когда нужно форсировать загрузку конкретной lazy-коллекции внутри транзакции
@Transactional
public Department getDepartmentWithEmployees(Long id) {
    Department dept = entityManager.find(Department.class, id);
    Hibernate.initialize(dept.getEmployees()); // загрузить коллекцию сейчас
    return dept; // безопасно передать за пределы транзакции
}

// Условная инициализация
@Transactional
public void process(User user, boolean needsCompany) {
    if (needsCompany && !Hibernate.isInitialized(user.getCompany())) {
        Hibernate.initialize(user.getCompany());
    }
}
```
