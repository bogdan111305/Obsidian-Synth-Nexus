## <a name="введение"></a>Введение: зачем нужны best practices

Проблема N+1 выборок — одна из самых частых причин деградации производительности в Hibernate/JPA-проектах. Даже опытные разработчики могут допустить ошибку, если не придерживаются проверенных практик. Этот гайд — концентрат современных подходов, которые реально работают.

---

## <a name="общие"></a>Общие рекомендации
- **Используйте FetchType.LAZY по умолчанию для всех связей.**
- **Не используйте FetchType.EAGER без крайней необходимости.**
- **Загружайте связанные сущности явно через JOIN FETCH или Entity Graph.**
- **Контролируйте доступ к геттерам в циклах — не вызывайте их без предварительной загрузки.**
- **Проверяйте логи Hibernate на предмет N+1.**
- **Тестируйте количество SQL-запросов в интеграционных тестах.**

## <a name="по-типам"></a>Практики для разных типов связей

### <a name="one-to-one"></a>1. @OneToOne
- **Современный подход:**
  - Для bidirectional-связей с естественным ключом используйте LAZY, optional, @MapsId, @PrimaryKeyJoinColumn.
  - Для синтетического ключа — LAZY и JOIN FETCH.
  - Для ленивой загрузки используйте bytecode enhancement или proxies (Hibernate >= 5.3).
- **Пример:**
```java
@Entity
public class User {
    @Id
    private Long id;
    @OneToOne(mappedBy = "user", fetch = FetchType.LAZY, optional = true)
    private Profile profile;
}
@Entity
public class Profile {
    @Id
    private Long id;
    @OneToOne(fetch = FetchType.LAZY, optional = true)
    @MapsId
    @JoinColumn(name = "id")
    private User user;
}
```
- **Запрос:**
  - HQL: `SELECT u FROM User u LEFT JOIN FETCH u.profile`
- **Best practice:** не используйте EAGER, не вызывайте геттеры в цикле без предварительной загрузки.

### <a name="many-to-one"></a>2. @ManyToOne
- **Всегда используйте LAZY.**
- Для массовой загрузки используйте JOIN FETCH или Entity Graph.
- @BatchSize может помочь, но не решает проблему полностью.
- **Пример:**
```java
@Entity
public class Order {
    @ManyToOne(fetch = FetchType.LAZY)
    private User user;
}
```
- **Запрос:**
  - HQL: `SELECT o FROM Order o JOIN FETCH o.user`

### <a name="one-to-many"></a>3. @OneToMany и @ManyToMany
- **Используйте LAZY для коллекций.**
- Загружайте только одну коллекцию за раз через JOIN FETCH.
- Для нескольких коллекций — делайте отдельные запросы.
- Используйте Set вместо List для избежания дублей.
- **Пример:**
```java
@Entity
public class User {
    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    private Set<Order> orders;
}
```
- **Запрос:**
  - HQL: `SELECT u FROM User u JOIN FETCH u.orders`

## <a name="современные"></a>Современные техники: Entity Graph, bytecode enhancement, логирование, тестирование

### Entity Graph
- Позволяет гибко управлять загрузкой связей без изменения модели.
- **Пример:**
```java
EntityGraph<User> graph = em.createEntityGraph(User.class);
graph.addAttributeNodes("profile");
List<User> users = em.createQuery("SELECT u FROM User u", User.class)
    .setHint("javax.persistence.fetchgraph", graph)
    .getResultList();
```
- Используйте @NamedEntityGraph для повторно используемых сценариев.

### Bytecode enhancement
- Для корректной работы LAZY на @OneToOne/@ManyToOne используйте bytecode enhancement (Hibernate enhancer plugin).
- **Преимущество:** позволяет лениво загружать даже одиночные связи.

### Логирование и тестирование
- Включайте `spring.jpa.show-sql=true` и `hibernate.format_sql=true` для анализа SQL.
- Используйте Hibernate Statistics или DataDog для мониторинга количества запросов.
- В интеграционных тестах проверяйте, что для массовых выборок не возникает N+1.

## <a name="ошибки"></a>Типовые ошибки и подводные камни
- Использование EAGER по умолчанию.
- JOIN FETCH для нескольких коллекций — декартово произведение.
- @BatchSize не работает для одиночных связей.
- SUBSELECT не работает для @OneToOne/@ManyToOne.
- @FetchProfile не работает без явной активации.
- Не использовать LAZY для @OneToOne без bytecode enhancement.
- Вызов геттеров в цикле без предварительной загрузки.
- Использование устаревших аннотаций вместо современных техник (Entity Graph).

## <a name="чеклист"></a>Чек-лист для проекта
- [ ] Все связи по умолчанию LAZY
- [ ] Для массовой загрузки используйте JOIN FETCH или Entity Graph
- [ ] Для коллекций используйте Set и отдельные запросы для каждой коллекции
- [ ] Не используйте EAGER без крайней необходимости
- [ ] Для @OneToOne с естественным ключом используйте @MapsId и LAZY
- [ ] Для сложных графов — Entity Graph
- [ ] Не используйте JOIN FETCH для нескольких коллекций одновременно
- [ ] Проверяйте логи Hibernate на предмет N+1
- [ ] Используйте интеграционные тесты для контроля количества SQL-запросов
