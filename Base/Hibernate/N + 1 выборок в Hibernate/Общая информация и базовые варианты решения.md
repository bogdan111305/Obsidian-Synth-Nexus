## <a name="введение"></a>Введение: что такое N+1 и почему это важно

**Проблема N+1** — это ситуация, когда для загрузки N сущностей и их связанных данных Hibernate выполняет 1 основной запрос и N дополнительных (например, для каждой дочерней коллекции или связи). Это приводит к экспоненциальному росту количества SQL-запросов и резкому падению производительности.

**Пример:**
- Есть сущность User и у каждого User — Profile (OneToOne).
- Вы загружаете 100 пользователей: Hibernate делает 1 запрос на User и 100 отдельных запросов на Profile.

**Почему это плохо:**
- - Лишняя нагрузка на БД и сеть.
- - Медленный отклик приложения.
- - Трудности с масштабированием и поддержкой.

---

## <a name="диагностика"></a>Диагностика N+1: как заметить проблему

- В логах Hibernate видно множество однотипных SELECT-запросов.
- Время отклика резко увеличивается при росте количества сущностей.
- Профилировщик БД показывает сотни/тысячи коротких запросов вместо одного оптимального.

**Пример лога:**
```
select * from user
select * from profile where id = ?
select * from profile where id = ?
...
```

## <a name="типы-маппингов"></a>Типы маппингов и специфика N+1

### <a name="one-to-one"></a>1. @OneToOne (однонаправленные и двунаправленные)

#### Проблема
- **Двунаправленные связи** (`User.profile` и `Profile.user`) часто вызывают N+1, особенно при ленивой загрузке.
- Если ключ не синтетический (например, Profile использует тот же id, что и User), то можно настроить связь как optional и lazy, чтобы избежать лишних запросов и обеспечить корректную ленивую загрузку.

#### Современное решение (Hibernate 5.3+)
- Используйте `@OneToOne(fetch = FetchType.LAZY, optional = true)` и `@PrimaryKeyJoinColumn` для естественного ключа.
- Для bidirectional-связи с естественным ключом:
```java
@Entity
public class User {
    @Id
    private Long id;
    @OneToOne(mappedBy = "user", fetch = FetchType.LAZY, optional = true)
    private Profile profile;
}

@Entity
public class Profile {
    @Id
    private Long id;
    @OneToOne(fetch = FetchType.LAZY, optional = true)
    @MapsId
    @JoinColumn(name = "id")
    private User user;
}
```
- **Важно:** LAZY для @OneToOne работает только с bytecode enhancement или через proxies (Hibernate >= 5.3).
- Если ключ синтетический — используйте обычный LAZY и JOIN FETCH при необходимости.

#### Решение через запросы
- HQL: `SELECT u FROM User u LEFT JOIN FETCH u.profile`
- Criteria API: `root.fetch("profile", JoinType.LEFT)`
- Querydsl: `leftJoin(user.profile).fetchJoin()`

#### Преимущества
- Нет N+1, гибкая загрузка, поддержка bidirectional-связей без избыточных запросов.

---

### <a name="many-to-one"></a>2. @ManyToOne

#### Проблема
- При ленивой загрузке (LAZY) Hibernate делает отдельный запрос для каждой связанной сущности при обращении к геттеру.
- При жадной загрузке (EAGER) — может быть избыточная загрузка.

#### Современное решение
- Всегда используйте `fetch = FetchType.LAZY` для всех @ManyToOne.
- Для массовой загрузки связанных сущностей используйте JOIN FETCH или Entity Graph.
- @BatchSize может помочь, но не решает проблему полностью (уменьшает количество запросов, но не устраняет N+1 полностью).

#### Пример
```java
@Entity
public class Order {
    @ManyToOne(fetch = FetchType.LAZY)
    private User user;
}
```
- HQL: `SELECT o FROM Order o JOIN FETCH o.user`
- @BatchSize: `@BatchSize(size = 20)` на уровне класса User

---

### <a name="one-to-many"></a>3. @OneToMany и @ManyToMany

#### Проблема
- При загрузке коллекций (orders, tags и т.д.) Hibernate может делать отдельный запрос для каждой коллекции (N+1).
- JOIN FETCH для нескольких коллекций приводит к декартовому произведению.

#### Современное решение
- Используйте LAZY для коллекций.
- Загружайте только одну коллекцию за раз через JOIN FETCH.
- Для нескольких коллекций — делайте отдельные запросы (или используйте Entity Graph с осторожностью).
- Используйте Set вместо List для избежания дублей и декартова произведения.

#### Пример
```java
@Entity
public class User {
    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    private Set<Order> orders;
}
```
- HQL: `SELECT u FROM User u JOIN FETCH u.orders`
- Для второй коллекции — отдельный запрос.

## <a name="bidirectional"></a>Bidirectional-связи: современные best practices

- Для двунаправленных связей используйте LAZY и optional, если ключ не синтетический.
- Для естественного ключа используйте @MapsId и @PrimaryKeyJoinColumn.
- Для синтетического ключа — обычный LAZY и JOIN FETCH.
- Не используйте EAGER без крайней необходимости.
- Контролируйте доступ к связанным сущностям через сервисный слой, а не в цикле в шаблоне/контроллере.
- Для ленивой загрузки @OneToOne используйте bytecode enhancement или proxies (Hibernate >= 5.3).

---

## <a name="fetchtype"></a>FetchType.LAZY vs EAGER: что выбрать и почему

- **FetchType.LAZY** — всегда по умолчанию для всех связей.
- **FetchType.EAGER** — только для действительно необходимых случаев (например, reference-entity, которая всегда нужна).
- EAGER игнорируется в HQL, если не используется JOIN FETCH.
- LAZY требует контроля доступа к геттерам (не вызывать в цикле без предварительной загрузки, использовать сервисный слой для массовой загрузки).

## <a name="fetch-strategies"></a>JOIN FETCH, Entity Graph, @BatchSize, @Fetch, @FetchProfile: сравнение и примеры

### JOIN FETCH
- HQL: `SELECT u FROM User u JOIN FETCH u.profile`
- Criteria API: `root.fetch("profile", JoinType.LEFT)`
- Querydsl: `leftJoin(user.profile).fetchJoin()`
- Позволяет загрузить связанные сущности в одном запросе.

### Entity Graph
- Позволяет динамически указывать, какие связи загружать.
- Пример:
```java
EntityGraph<User> graph = em.createEntityGraph(User.class);
graph.addAttributeNodes("profile");
List<User> users = em.createQuery("SELECT u FROM User u", User.class)
    .setHint("javax.persistence.fetchgraph", graph)
    .getResultList();
```
- Более гибко, чем JOIN FETCH, особенно для сложных графов.

### @BatchSize
- Позволяет загружать пачки связанных сущностей через IN.
- Применим к коллекциям и @ManyToOne.
- Не решает проблему полностью, но уменьшает количество запросов.

### @Fetch
- Позволяет задать стратегию (JOIN, SELECT, SUBSELECT).
- SUBSELECT — для коллекций, загружает все коллекции одним подзапросом.
- Не рекомендуется для сложных случаев — лучше использовать JOIN FETCH или Entity Graph.

### @FetchProfile
- Позволяет определить профили загрузки для разных сценариев.
- Требует явного включения профиля.
- Не работает для HQL без активации.

## <a name="pitfalls"></a>Типовые ошибки и подводные камни

- Использование EAGER по умолчанию.
- JOIN FETCH для нескольких коллекций — декартово произведение (всегда избегайте этого!).
- @BatchSize не работает для одиночных связей.
- SUBSELECT не работает для @OneToOne/@ManyToOne.
- @FetchProfile не работает без явной активации.
- Не использовать LAZY для @OneToOne без bytecode enhancement.
- Вызов геттеров в цикле без предварительной загрузки.

---

## <a name="checklist"></a>Чек-лист для устранения N+1

- [ ] Все связи по умолчанию LAZY
- [ ] Для массовой загрузки используйте JOIN FETCH или Entity Graph
- [ ] Для коллекций используйте Set и отдельные запросы для каждой коллекции (или осторожно Entity Graph)
- [ ] Не используйте EAGER без крайней необходимости
- [ ] Для @OneToOne с естественным ключом используйте @MapsId и LAZY
- [ ] Для сложных графов — Entity Graph
- [ ] Не используйте JOIN FETCH для нескольких коллекций одновременно
- [ ] Проверяйте логи Hibernate на предмет N+1

## Bytecode enhancement: как сделать LAZY действительно ленивым

**Bytecode enhancement** — это механизм, при котором Hibernate модифицирует байткод ваших сущностей на этапе сборки или запуска, чтобы добавить поддержку настоящей ленивой загрузки для одиночных связей (`@OneToOne`, `@ManyToOne`). Без этого механизма LAZY для этих связей часто не работает как ожидается: Hibernate просто игнорирует LAZY и загружает связанные сущности немедленно.

### Зачем нужен bytecode enhancement?
- Позволяет реально лениво загружать одиночные связи (`@OneToOne`, `@ManyToOne`), а не только коллекции.
- Устраняет N+1 даже для сложных bidirectional-связей с LAZY.
- Позволяет использовать LAZY для @OneToOne с естественным ключом (через @MapsId и @PrimaryKeyJoinColumn).
- Делает поведение Hibernate более предсказуемым и эффективным.

### Как включить bytecode enhancement

**Gradle:**
```groovy
plugins {
    id 'org.hibernate.orm' version '6.4.4.Final'
}
hibernate {
    enhance {
        lazyInitialization true
        dirtyTracking true
        associationManagement true
    }
}
```

**Maven:**
```xml
<plugin>
    <groupId>org.hibernate.orm.tooling</groupId>
    <artifactId>hibernate-enhance-maven-plugin</artifactId>
    <version>6.4.4.Final</version>
    <executions>
        <execution>
            <goals>
                <goal>enhance</goal>
            </goals>
        </execution>
    </executions>
    <configuration>
        <lazyInitialization>true</lazyInitialization>
        <dirtyTracking>true</dirtyTracking>
        <associationManagement>true</associationManagement>
    </configuration>
</plugin>
```

- После включения enhancement Hibernate сможет создавать прокси даже для одиночных связей, и LAZY будет работать как задумано.
- Для Spring Boot: используйте [spring-boot-plugin-hibernate-enhance](https://github.com/hibernate/hibernate-enhance-maven-plugin) или настройте enhancement вручную.

### Когда bytecode enhancement обязателен
- Если вы хотите использовать LAZY для @OneToOne или @ManyToOne и быть уверенным, что Hibernate не проигнорирует настройку.
- Для production-проектов, где важна производительность и контроль над SQL.
- Для сложных bidirectional-связей с естественным ключом.

### Best practices
- Всегда включайте bytecode enhancement для всех production-проектов на Hibernate/JPA.
- Проверяйте, что LAZY реально работает: в логах не должно быть лишних SELECT при доступе к геттерам без инициализации.
- Не используйте EAGER без крайней необходимости.
- Тестируйте поведение LAZY в интеграционных тестах.
