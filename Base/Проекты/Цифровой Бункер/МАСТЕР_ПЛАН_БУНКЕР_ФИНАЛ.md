## 1. Принципы и архитектурные инварианты

Следующие принципы не могут быть нарушены ни одним техническим решением на любом этапе разработки:

**P1 — Zero-Trust by Default.**
Ни одна зона, сервис или сетевой сегмент не считается доверенным автоматически. Каждый запрос несёт явное доказательство identity: JWT (пользователь) или SVID-сертификат (сервис). Смежность в одной Docker-сети не означает доверия.

**P2 — Envelope Encryption везде.**
Сервер никогда не видит plaintext данных пользователя. Ключи существуют в открытом виде только в RAM клиентского устройства и изолированной Vault-зоне, только в момент операции, только в форме `byte[]`, и уничтожаются сразу после использования.

**P3 — Аппаратная привязка.**
Master Key и сессия привязаны к физическому устройству: Titan M2 (Android) или Secure Enclave (iOS/macOS). NFC-кольцо является обязательным вторым фактором. Компрометация пароля без физического кольца не даёт доступа.

**P4 — Код не содержит адресов.**
Все хосты, порты, ключи и секреты — в переменных окружения. Переход от локального Docker к продакшн-кластеру не требует правки кода, только `.env`.

**P5 — Сертификаты с первого дня.**
mTLS между сервисами включён в dev-режиме с самого начала. Это не усложнение, а требование, которое невозможно добавить «потом» без переписывания конфигураций.

## 3. Криптографический протокол Zero-Trust

### 3.1 Иерархия ключей — Envelope Encryption

```
Пароль (UTF-8) + Salt (32 bytes, с HMAC-верификацией)
        │
        ▼  Argon2id(m=65536 КБ, t=3, p=4, len=32)
  ┌─────────────────┐
  │  Master Key (MK)│  32 bytes
  │  Хранится ТОЛЬКО│  в Secure Enclave (Android Titan M2 / iOS Secure Enclave)
  │  По сети: НИКОГДА│
  └────────┬────────┘
           │  AES-256-GCM decrypt(ECK, MK)
           ▼
  ┌─────────────────┐
  │Category Key (CK)│  32 bytes
  │  В RAM: только  │  во время операции
  │  Vault Redis:   │  TTL=30s, TLS, bind=127.0.0.1
  │  После: fill(0) │  + VarHandle.fullFence()
  └────────┬────────┘
           │  AES-256-GCM decrypt(EDK, CK)
           ▼
  ┌─────────────────┐
  │  Data Key (DK)  │  32 bytes
  │  В RAM: только  │  во время потокового дешифрования
  │  После: fill(0) │  немедленно
  └────────┬────────┘
           │  Tink StreamingAead (AES-256-GCM-HKDF-4KB)
           ▼
        Plaintext файла или секрета
```

**Зачем три уровня ключей:**
- Ротация CK не требует перешифровки файлов — только перешифровывается EDK.
- Компрометация одного DK не компрометирует другие файлы.
- Сервер в любой момент может забыть CK (kill switch), не трогая данные.

### 3.2 Генерация Master Key и верификация соли

**Проблема без HMAC:** Salt скачивается с сервера. Атакующий, способный подменить ответ `/auth/salt/{username}`, подкладывает заранее известную соль и строит rainbow table для конкретного пользователя.

**Решение — Salt + HMAC:**

```
Регистрация (один раз):
  1. Сервер генерирует: salt = SecureRandom.nextBytes(32)
  2. Сервер вычисляет: salt_hmac = HMAC-SHA256(salt, SERVER_STATIC_KEY)
  3. Сервер сохраняет: {salt, salt_hmac} в PostgreSQL → users
  4. Клиент получает: {salt, salt_hmac}
  5. Клиент вычисляет MK = Argon2id(password, salt)
  6. Клиент сохраняет в Secure Enclave: {mk, expected_salt_fingerprint=SHA256(salt_hmac)}
     ← fingerprint закреплён на устройстве при регистрации (TOFU)

Каждый вход:
  1. GET /auth/salt/{username} → {salt, salt_hmac}
  2. Клиент проверяет: SHA256(salt_hmac) == expected_salt_fingerprint из Enclave
     Если не совпадает → отказ, предупреждение "соль подменена"
  3. Клиент вычисляет MK = Argon2id(password, verified_salt)
  4. MK используется для расшифровки ECK
```

**Параметры Argon2id (производительность vs безопасность):**
```
m = 65536 КБ (64 МБ оперативной памяти)
t = 3 итерации
p = 4 параллельных потока
len = 32 байта (256 бит)

На современном телефоне: ~300-500мс → приемлемо для UX
На GPU атакующего: 64МБ per instance → резко ограничивает параллельность
```

### 3.3 WebAuthn — NFC-кольцо как физический MFA

**Требования к кольцу:** FIDO2 CTAP2 (не FIDO U2F, не просто NFC-эмуляция). Обязательные capability: user presence, signature counter (signCount), поддержка ES256 (ECDSA P-256).

**Детальный поток аутентификации:**

```
──── РЕГИСТРАЦИЯ КОЛЬЦА (один раз на устройстве) ────

Client → POST /auth/webauthn/register/options
  body: {username}

Server → PublicKeyCredentialCreationOptions {
  challenge: random 32 bytes (хранится в сессии 90 секунд)
  rp: {id: "bunker.yourdomain.com", name: "Цифровой Бункер"}
  user: {id: user_uuid, name: username}
  pubKeyCredParams: [{type: "public-key", alg: -7}]  // ES256
  authenticatorSelection: {
    authenticatorAttachment: "cross-platform"  // внешний authenticator (кольцо)
    userVerification: "preferred"
    residentKey: "preferred"
  }
  timeout: 60000
}

Client → касается кольцом телефона (NFC)
Кольцо  → генерирует keypair (приватный хранится в кольце, не извлекаем)
Кольцо  → attestation = sign(clientDataJSON + authenticatorData, privkey)

Client → POST /auth/webauthn/register/finish
  body: {id: credential_id, response: {attestationObject, clientDataJSON}}

Server → webauthn4j verify attestation:
  1. Проверить challenge совпадает
  2. Проверить origin == "https://bunker.yourdomain.com"
  3. Проверить rpIdHash == SHA256("bunker.yourdomain.com")
  4. Сохранить: {credential_id, public_key_cose, aaguid, sign_count=0}

──── АУТЕНТИФИКАЦИЯ (каждый вход) ────

Client → POST /auth/webauthn/login/options
  body: {username}

Server → PublicKeyCredentialRequestOptions {
  challenge: random 32 bytes (новый для каждого запроса, TTL=90s)
  rpId: "bunker.yourdomain.com"
  allowCredentials: [{id: credential_id, type: "public-key", transports: ["nfc"]}]
  userVerification: "preferred"
}

Client → касается кольцом (NFC tap)
Кольцо  → signature = sign(authenticatorData + SHA256(clientDataJSON), privkey)
           authenticatorData содержит: rpIdHash + flags + signCount(++) + extensions

Client → POST /auth/webauthn/login/finish
  body: {id, response: {authenticatorData, clientDataJSON, signature}}

Server → webauthn4j verify assertion:
  1. Проверить signature публичным ключом из PostgreSQL
  2. Проверить challenge
  3. КРИТИЧНО: assert response.signCount > stored_signCount
     Если signCount <= stored → кольцо клонировано! → REJECT + audit log
  4. UPDATE webauthn_credentials SET sign_count = response.signCount
  5. → выдать JWT (ECDSA P-256, exp=15мин) + Refresh Token

Почему signCount важен: физическое кольцо монотонно увеличивает счётчик.
Клон кольца не знает текущего счётчика → всегда даёт меньшее значение.
```

**Flutter-реализация WebAuthn:**
```dart
// Используем платформенные API через пакет `passkeys`
// Android: CredentialManager API (Android 14+)
// iOS: ASAuthorizationController

import 'package:passkeys/passkeys.dart';

class WebAuthnService {
  final _authenticator = PasskeyAuthenticator();

  Future<void> registerRing(String username, Map<String, dynamic> options) async {
    final credential = await _authenticator.register(
      RegisterRequest(
        challenge: options['challenge'],
        rpId: options['rp']['id'],
        rpName: options['rp']['name'],
        userId: options['user']['id'],
        userName: username,
      ),
    );
    // Отправить credential на сервер
    await api.post('/auth/webauthn/register/finish', credential.toJson());
  }

  Future<String> authenticateWithRing(Map<String, dynamic> options) async {
    final assertion = await _authenticator.authenticate(
      AuthenticateRequest(
        challenge: options['challenge'],
        rpId: options['rpId'],
        allowedCredentialIds: options['allowCredentials']
            .map((c) => c['id'])
            .toList(),
      ),
    );
    // Получить JWT в ответе
    final response = await api.post('/auth/webauthn/login/finish', assertion.toJson());
    return response['jwt_token'];
  }
}
```

### 3.4 Инъекция Category Key в Vault Redis

> [!INFO] Архитектурное решение из аудита
> Redis находится ИСКЛЮЧИТЕЛЬНО на Vault-узле (internal_secure сеть).
> CK никогда не проходит через Sentinel и не хранится на публичном сервере.

**Полный алгоритм инъекции:**

```
──── КЛИЕНТ (Flutter) ────

1. Клиент расшифровывает ECK:
   byte[] ck = AES_GCM.decrypt(eck_bytes, mk);
   // mk — из Secure Enclave, ck — в RAM, не в String

2. Клиент генерирует ephemeral X25519 keypair:
   KeyPair ek = X25519.generateKeyPair();  // ek_priv, ek_pub

3. Клиент вычисляет shared secret:
   byte[] shared = X25519.computeSharedSecret(ek_priv, VAULT_STATIC_PUB_KEY);
   // VAULT_STATIC_PUB_KEY — закреплён в приложении при сборке

4. Клиент шифрует CK:
   byte[] enc_ck = AES_GCM.encrypt(ck, shared, session_id_as_aad);
   Arrays.fill(ck, (byte) 0);      // немедленно затереть plaintext CK
   Arrays.fill(shared, (byte) 0);  // затереть shared secret

5. Клиент отправляет (через TLS + gRPC mTLS):
   InjectCategoryKeyRequest {
     session_id: UUID,
     ek_pub: bytes,      // 32 bytes, X25519 public key
     enc_ck: bytes,      // зашифрованный CK
     enc_ck_iv: bytes,   // GCM nonce
     enc_ck_tag: bytes,  // GCM auth tag
   }

──── СЕРВЕР bunker-storage (Vault, Platform Thread) ────

6. Vault вычисляет shared secret:
   byte[] shared = X25519.computeSharedSecret(VAULT_STATIC_PRIV_KEY, request.ek_pub);

7. Vault расшифровывает CK:
   byte[] ck = AES_GCM.decrypt(request.enc_ck, shared, session_id_as_aad);
   Arrays.fill(shared, (byte) 0);  // затереть shared secret

8. Vault кладёт CK в локальный Redis (bind=127.0.0.1):
   redis.set("ck:" + session_id, ck, EX, 30);
   Arrays.fill(ck, (byte) 0);  // затереть локальную копию
   VarHandle.fullFence();

9. Vault возвращает клиенту: {status: READY, session_id}

──── ИСПОЛЬЗОВАНИЕ КЛЮЧА (Platform Thread) ────

10. При запросе файла/секрета:
    byte[] ck = redis.get("ck:" + session_id);
    if (ck == null) throw new KeyExpiredException("CK истёк, повторите инъекцию");
    try {
        byte[] dk = AES_GCM.decrypt(edk, ck);  // расшифровать Data Key
        return streamDecrypt(filePath, dk);      // потоковое дешифрование
    } finally {
        Arrays.fill(ck, (byte) 0);
        VarHandle.fullFence();
        redis.del("ck:" + session_id);  // немедленное удаление, не ждать TTL
    }
```

**Конфигурация Redis для CK-хранения:**
```
bind 127.0.0.1          # только loopback — нет сетевого доступа даже внутри Docker
port 0                  # отключить plaintext порт
tls-port 6380           # только TLS
requirepass <256-bit>   # обязательная аутентификация
save ""                 # отключить RDB — CK не должны быть на диске
appendonly no           # отключить AOF
maxmemory-policy noeviction  # CK не вытесняется в swap при нехватке памяти
rename-command KEYS     ""   # запрет сканирования всех ключей
rename-command DEBUG    ""   # запрет отладочных команд
rename-command CONFIG   ""   # запрет изменения конфигурации
rename-command FLUSHALL ""   # FLUSHDB разрешён только для Kill Switch (через Lua)
```

### 3.5 Управление памятью в Java — Production-паттерны

**Проблема 1: JIT Dead-Write Elimination**
JVM JIT может оптимизировать `Arrays.fill()` как «мёртвую запись» если после неё массив недостижим и не читается. В Java нет аналога `SecureZeroMemory()` с гарантией компилятора.

**Решение:** `VarHandle.fullFence()` создаёт барьер памяти, который JIT не убирает:
```java
Arrays.fill(secret, (byte) 0);
VarHandle.fullFence();  // memory fence: JIT не оптимизирует предыдущую запись
```

**Проблема 2: Virtual Threads (Project Loom) + heap serialization**
При parking виртуального потока JVM сериализует его стек на heap. Если стек содержит ссылку на `byte[] secret`, она попадает в heap до момента zeroing — классический heap disclosure через memory dump.

**Решение:** Все криптографические операции — только в Platform Threads:
```java
// Пул platform threads для всей криптографии
private static final ExecutorService CRYPTO_EXECUTOR =
    Executors.newFixedThreadPool(
        Runtime.getRuntime().availableProcessors(),
        Thread.ofPlatform().name("crypto-", 0).factory()
    );

// Правильный паттерн
public CompletableFuture<byte[]> decryptAsync(byte[] encData, UUID sessionId) {
    return CompletableFuture.supplyAsync(
        () -> decryptInPlatformThread(encData, sessionId),
        CRYPTO_EXECUTOR  // platform thread, не virtual
    );
}

private byte[] decryptInPlatformThread(byte[] encData, UUID sessionId) {
    byte[] ck = null;
    byte[] dk = null;
    try {
        ck = fetchAndDecryptCK(sessionId);    // из Redis
        dk = aesGcm.decrypt(edk, ck);         // расшифровать Data Key
        return tinkDecrypt(encData, dk);       // использовать
    } finally {
        if (ck != null) { Arrays.fill(ck, (byte) 0); }
        if (dk != null) { Arrays.fill(dk, (byte) 0); }
        VarHandle.fullFence();
        redis.del("ck:" + sessionId);
    }
}
```

**Проблема 3: String internment**
`String` в Java интернируется в пуле строк и не подлежит явной очистке. GC не гарантирует немедленного освобождения.
```java

Arrays.fill(passwordChars, '\0');                   // ✅

```

```java
    long size = Files.size(path);
    try (FileChannel ch = FileChannel.open(path, StandardOpenOption.WRITE)) {
        // Pass 1: zeros
        ch.write(ByteBuffer.wrap(new byte[(int) size]));
        ch.force(true);
        // Pass 2: ones (0xFF)
        ch.position(0);
        byte[] ones = new byte[(int) size];
        Arrays.fill(ones, (byte) 0xFF);
        ch.write(ByteBuffer.wrap(ones));
        ch.force(true);

        ch.write(ByteBuffer.wrap(rnd));
```

### 4.5 `bunker-chat` — Scala 3.4, ZIO 2.1

**Почему ZIO, а не Virtual Threads:**
- `ZIO STM` — Software Transactional Memory: атомарные транзакции над in-memory state без locks (статусы сообщений, presence). В Java аналога нет.
- `ZIO Fiber` — 1 fiber per WebSocket connection при M:N mapping на OS threads. Virtual Threads дают то же, но ZIO предоставляет строгую composability.
- `ZIO Scope` — детерминированное управление lifecycle ресурсов: гарантия что соединения закрываются даже при ошибках.
- `ZIO Schedule` — retry/backoff политики для Kafka producer declaratively.

**Основной WebSocket handler:**
```scala
def handleWebSocket(userId: UserId, ws: WebSocketChannel): ZIO[Scope, Throwable, Unit] =
  for
    _         <- ZIO.logInfo(s"WS connected: $userId")
    inbox     <- Queue.bounded[OutboundFrame](256)  // bounded: backpressure
    _         <- presenceService.markOnline(userId)

    // Fiber 1: входящие сообщения → валидация → Kafka
    inFiber   <- ws.receiveStream
                   .mapZIO { frame =>
                     for
                       msg <- parseAndValidate(frame, userId)
                       _   <- kafkaProducer.produce("chat-messages", msg.chatId, msg.encPayload)
                       _   <- auditLog.record(userId, "MESSAGE_SENT", msg.chatId)
                     yield ()
                   }
                   .runDrain
                   .onError(e => ZIO.logError(s"Inbound error: $e"))
                   .fork

    // Fiber 2: исходящие из inbox → клиент
    outFiber  <- ZStream.fromQueue(inbox)
                   .mapZIO(frame => ws.send(frame))
                   .runDrain
                   .fork

    // Fiber 3: Kafka consumer для этого пользователя → inbox
    kafkaFiber <- kafkaConsumer
                   .subscribeUser(userId)
                   .mapZIO(record => inbox.offer(OutboundFrame(record.value)))
                   .runDrain
                   .fork

    // Ждать завершения любого из fiber (клиент отключился)
    _         <- inFiber.join race outFiber.join race kafkaFiber.join
    _         <- presenceService.markOffline(userId)
    _         <- ZIO.logInfo(s"WS disconnected: $userId")
  yield ()
```

## 5. Экосистема мессенджера E2EE

### 5.1 Протокол X3DH (Extended Triple Diffie-Hellman)

X3DH заменяет хранение Chat Key в Infisical. Chat Key вычисляется независимо на каждом устройстве и **нигде не хранится на сервере** — только зашифрованные сообщения в Kafka/ClickHouse.

**Предварительная подготовка (Bob публикует prekey bundle):**
```
Bob генерирует и публикует в PostgreSQL (таблицы x3dh_prekeys, one_time_prekeys):

IK_B    = X25519 Identity Key (постоянный, 32 bytes pub)
SPK_B   = X25519 Signed PreKey (меняется каждые 7 дней, 32 bytes pub)
Sig_IK  = Ed25519.sign(IK_B_priv, SPK_B_pub)  ← подпись SPK ключом Identity Key
OPK_B[] = X25519 One-Time PreKeys (50 штук, расходуются и пополняются)
```

**Alice инициирует X3DH с Bob:**
```
1. Alice запрашивает bundle Bob'а:
   GET /api/chat/prekeys/{bob_user_id}
   → {IK_B_pub, SPK_B_pub, Sig_IK, OPK_B_pub (один, удаляется сервером)}

2. Alice верифицирует подпись:
   Ed25519.verify(IK_B_pub, SPK_B_pub, Sig_IK)
   Если verify == false → ABORT (атака на цепочку ключей)

3. Alice генерирует ephemeral keypair:
   EK_A = X25519.generateKeyPair()

4. Alice вычисляет 4 DH-операции:
   DH1 = X25519(IK_A_priv,  SPK_B_pub)   // Identity-to-Signed
   DH2 = X25519(EK_A_priv,  IK_B_pub)    // Ephemeral-to-Identity
   DH3 = X25519(EK_A_priv,  SPK_B_pub)   // Ephemeral-to-Signed
   DH4 = X25519(EK_A_priv,  OPK_B_pub)   // Ephemeral-to-OneTime

5. Alice вычисляет Master Secret:
   F   = 0xFF * 32  // 32 байта 0xFF (как в Signal Protocol)
   MS  = HKDF-SHA256(
           ikm  = F || DH1 || DH2 || DH3 || DH4,
           salt = zeros(32),
           info = "bunker-x3dh-v1",
           len  = 32
         )

6. Alice инициализирует Double Ratchet с MS как корневым ключом
   Chat Key = первый ключ Double Ratchet (меняется каждое сообщение!)

7. Alice отправляет первое сообщение с initial header:
   {
     sender_ik_pub: IK_A_pub,     // для Bob чтобы повторить DH
     sender_ek_pub: EK_A_pub,     // ephemeral ключ Alice
     opk_id: OPK_B_id,            // какой OPK использован
     ciphertext: encrypt(plaintext, chat_key_0)
   }

Bob получает первое сообщение:
1. Bob повторяет те же 4 DH зеркально (у него приватные ключи)
2. Получает идентичный MS
3. Инициализирует Double Ratchet с тем же MS
4. Расшифровывает ciphertext

После первого обмена: Double Ratchet обеспечивает Forward Secrecy.
Каждое сообщение использует новый ключ (ratchet step).
```

**Ротация Signed PreKey:**
```
Каждые 7 дней клиент:
1. Генерирует новую SPK_B
2. Подписывает Ed25519(IK_B_priv, new_SPK_B_pub)
3. PUT /api/chat/prekeys/signed → обновляет в PostgreSQL
4. Старая SPK удаляется через 24 часа (дать время завершить активные сессии)
```

**Пополнение One-Time PreKeys:**
```
Сервер присылает события когда OPK stock < 5:
  Kafka event: {type: "OPK_DEPLETED", user_id, remaining: N}

Клиент получает push → в фоне генерирует 50 новых OPK → PUT /api/chat/prekeys/onetime
```

### 5.2 Kafka как слепой курьер

```
Топики (создаются в Phase 0):

chat-messages      partitions=12, retention=7days
  key:   chatId (UUID)
  value: {
    sender_id_hmac: HMAC-SHA256(userId, SERVER_KEY),  // не userId напрямую!
    encrypted_payload: bytes,  // Double Ratchet ciphertext
    timestamp: epoch_millis
  }
  Партиционирование по chatId → все сообщения чата в одну партицию (порядок)

system-events      partitions=1, cleanup.policy=compact
  key:   event_type (string)
  value: {type, timestamp, metadata}
  compacted → SYSTEM_LOCKED состояние сохраняется между рестартами

notifications      partitions=4, retention=1day
  key:   recipient_user_id
  value: {device_token, chat_id, platform}

audit-events       partitions=4, retention=90days
  key:   user_id
  value: {event_type, timestamp, ip, metadata}
```

**КРИТИЧНО:** `bunker-chat` публикует сообщения и **не декодирует payload**. Сервер видит только зашифрованные байты. Логирование payload запрещено — это нарушает E2EE.

## 7. Полная SQL-схема PostgreSQL

```sql
-- ================================================================
-- DIGITAL BUNKER — PostgreSQL 16 Schema
-- Версия 1.0 | 2026-03-20
-- Применять: psql -f init-db.sql
-- ================================================================

CREATE EXTENSION IF NOT EXISTS "pgcrypto";   -- gen_random_uuid(), gen_random_bytes()

-- ================================================================
-- СЕКЦИЯ 1: ПОЛЬЗОВАТЕЛИ И АУТЕНТИФИКАЦИЯ
-- ================================================================

CREATE TABLE users (
    id              UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    username        TEXT        NOT NULL UNIQUE,

    -- Соль для Argon2id: 32 случайных байта
    -- Публичная в том смысле что отдаётся клиенту — но с защитой через HMAC
    argon2_salt     BYTEA       NOT NULL CHECK (length(argon2_salt) = 32),

    -- HMAC-SHA256(argon2_salt, SERVER_STATIC_KEY)
    -- Клиент верифицирует перед использованием соли
    -- SERVER_STATIC_KEY хранится в Vault Infisical/ENV, не в БД
    salt_hmac       BYTEA       NOT NULL CHECK (length(salt_hmac) = 32),

    is_locked       BOOLEAN     NOT NULL DEFAULT FALSE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_login_at   TIMESTAMPTZ,

    CONSTRAINT username_length CHECK (length(username) BETWEEN 3 AND 64),
    CONSTRAINT username_chars  CHECK (username ~ '^[a-zA-Z0-9_\-\.]+$')
);

COMMENT ON TABLE users IS 'Пользователи системы. Пароль не хранится — используется Argon2id на клиенте.';
COMMENT ON COLUMN users.argon2_salt IS 'Соль для Argon2id. Публичная, но верифицируется через salt_hmac.';
COMMENT ON COLUMN users.salt_hmac IS 'HMAC-SHA256(salt, SERVER_KEY). Клиент обязан проверить перед деривацией MK.';

-- ----------------------------------------------------------------
-- FIDO2 / WebAuthn credentials (NFC-кольцо, Passkey, Touch ID)
-- ----------------------------------------------------------------
CREATE TABLE webauthn_credentials (
    id               UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id          UUID    NOT NULL REFERENCES users(id) ON DELETE CASCADE,

    -- FIDO2 Credential ID: выдаётся аутентификатором, variable length
    credential_id    BYTEA   NOT NULL UNIQUE,

    -- COSE-encoded EC P-256 public key (CBOR формат из WebAuthn spec)
    public_key_cose  BYTEA   NOT NULL,

    -- Монотонный счётчик подписей (replay + clone detection)
    -- Если response.signCount <= stored → клонирование → REJECT
    sign_count       BIGINT  NOT NULL DEFAULT 0,

    -- Authenticator AAGUID: идентифицирует модель устройства
    -- Позволяет отследить "какое именно кольцо" зарегистрировано
    aaguid           BYTEA,

    device_label     TEXT,        -- "NFC Ring", "MacBook Touch ID", "YubiKey 5C"
    transports       TEXT[],      -- ["nfc"], ["internal"], ["usb"], ["hybrid"]

    -- backed_up=true означает ключ синхронизирован в облако → снижение безопасности
    -- Для NFC-кольца всегда FALSE
    backed_up        BOOLEAN NOT NULL DEFAULT FALSE,

    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_used_at     TIMESTAMPTZ,

    CONSTRAINT one_credential_per_user_device UNIQUE (user_id, aaguid)
);

COMMENT ON COLUMN webauthn_credentials.sign_count IS
    'Монотонный счётчик. При sign_count <= stored в БД — детектируется клонирование кольца.';

-- ----------------------------------------------------------------
-- JWT Refresh Tokens
-- ----------------------------------------------------------------
CREATE TABLE refresh_tokens (
    id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID        NOT NULL REFERENCES users(id) ON DELETE CASCADE,

    -- Храним SHA-256(raw_token), не raw_token
    -- Даже при компрометации БД — токены не скомпрометированы
    token_hash  BYTEA       NOT NULL UNIQUE CHECK (length(token_hash) = 32),

    device_fp   TEXT        NOT NULL,  -- fingerprint устройства для валидации
    expires_at  TIMESTAMPTZ NOT NULL,
    revoked     BOOLEAN     NOT NULL DEFAULT FALSE,
    revoked_at  TIMESTAMPTZ,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_refresh_tokens_user_active
    ON refresh_tokens(user_id, expires_at)
    WHERE revoked = FALSE;

-- ================================================================
-- СЕКЦИЯ 2: УПРАВЛЕНИЕ КЛЮЧАМИ (Envelope Encryption)
-- ================================================================

-- Категории хранилища: Documents, Passwords, Finance, Work, Private...
-- Каждая категория имеет свой Category Key (CK), зашифрованный Master Key
CREATE TABLE categories (
    id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID        NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name        TEXT        NOT NULL,

    -- ECK = Category Key, зашифрованный Master Key (AES-256-GCM)
    -- CK никогда не хранится в открытом виде — только ECK
    eck         BYTEA       NOT NULL,  -- ciphertext
    eck_iv      BYTEA       NOT NULL CHECK (length(eck_iv) = 12),    -- GCM nonce
    eck_tag     BYTEA       NOT NULL CHECK (length(eck_tag) = 16),   -- GCM auth tag

    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    UNIQUE(user_id, name),
    CONSTRAINT name_not_empty CHECK (length(trim(name)) > 0)
);

COMMENT ON TABLE categories IS
    'Категории хранилища. ECK = Category Key зашифрованный Master Key пользователя. Ротация CK = пересоздать ECK.';

-- ================================================================
-- СЕКЦИЯ 3: ХРАНИЛИЩЕ ФАЙЛОВ
-- ================================================================

CREATE TABLE file_metadata (
    id               UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id          UUID    NOT NULL REFERENCES users(id),
    category_id      UUID    NOT NULL REFERENCES categories(id),

    -- Имя файла зашифровано Category Key на клиенте перед отправкой
    -- Сервер не знает реального имени файла
    encrypted_name   BYTEA   NOT NULL,
    name_iv          BYTEA   NOT NULL CHECK (length(name_iv) = 12),
    name_tag         BYTEA   NOT NULL CHECK (length(name_tag) = 16),

    -- Физический путь на диске: /vault/files/{user_id}/{file_id}
    -- UUID-based, не выводится из имени файла
    storage_path     TEXT    NOT NULL,

    size_bytes       BIGINT  NOT NULL CHECK (size_bytes > 0),

    -- Зашифрованный MIME type (или NULL если не предоставлен)
    mime_type_enc    BYTEA,
    mime_iv          BYTEA,

    -- EDK = Data Encryption Key, зашифрованный Category Key (AES-256-GCM)
    -- Трёхуровневая иерархия: MK → CK → DK → файл
    -- Ротация CK: нужно пересоздать только EDK, не перешифровывать файл
    edk              BYTEA   NOT NULL,
    edk_iv           BYTEA   NOT NULL CHECK (length(edk_iv) = 12),
    edk_tag          BYTEA   NOT NULL CHECK (length(edk_tag) = 16),

    -- Tink StreamingAead метаданные (без самого ключа — только алгоритм/версия)
    tink_keyset_info BYTEA,

    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    -- Soft delete: файл помечается но не удаляется немедленно
    deleted_at       TIMESTAMPTZ,

    -- Подтверждение secure overwrite фоновым воркером (3 прохода)
    wiped_at         TIMESTAMPTZ,

    CONSTRAINT valid_delete CHECK (wiped_at IS NULL OR deleted_at IS NOT NULL)
);

CREATE INDEX idx_file_metadata_user_category
    ON file_metadata(user_id, category_id, created_at DESC)
    WHERE deleted_at IS NULL;

CREATE INDEX idx_file_metadata_pending_wipe
    ON file_metadata(deleted_at)
    WHERE deleted_at IS NOT NULL AND wiped_at IS NULL;

COMMENT ON COLUMN file_metadata.edk IS
    'Encrypted Data Key. Позволяет ротировать Category Key без перешифровки файла.';
COMMENT ON COLUMN file_metadata.wiped_at IS
    'Когда воркер выполнил secure overwrite (DoD 5220.22-M). NULL = ещё не затёрт.';

-- ================================================================
-- СЕКЦИЯ 4: МЕНЕДЖЕР ПАРОЛЕЙ
-- ================================================================

CREATE TABLE vault_secrets (
    id             UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id        UUID    NOT NULL REFERENCES users(id),
    category_id    UUID    NOT NULL REFERENCES categories(id),

    -- Зашифрованный JSON-blob структуры записи:
    -- {
    --   "title": "Gmail",
    --   "login": "user@gmail.com",
    --   "password": "...",
    --   "url": "https://gmail.com",
    --   "notes": "...",
    --   "totp_secret": "BASE32_SECRET",
    --   "custom_fields": [{"name": "...", "value": "..."}]
    -- }
    -- Шифруется Category Key через AES-256-GCM
    encrypted_blob BYTEA   NOT NULL,
    blob_iv        BYTEA   NOT NULL CHECK (length(blob_iv) = 12),
    blob_tag       BYTEA   NOT NULL CHECK (length(blob_tag) = 16),

    -- EDK для этой конкретной записи (индивидуальная ротация)
    edk            BYTEA,
    edk_iv         BYTEA,
    edk_tag        BYTEA,

    -- Для optimistic locking при конкурентном обновлении
    version        INT     NOT NULL DEFAULT 1,

    created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at     TIMESTAMPTZ,

    CONSTRAINT edk_complete CHECK (
        (edk IS NULL AND edk_iv IS NULL AND edk_tag IS NULL) OR
        (edk IS NOT NULL AND edk_iv IS NOT NULL AND edk_tag IS NOT NULL)
    )
);

CREATE INDEX idx_vault_secrets_user_category
    ON vault_secrets(user_id, category_id, created_at DESC)
    WHERE deleted_at IS NULL;

-- ================================================================
-- СЕКЦИЯ 5: МЕССЕНДЖЕР — X3DH (Signal Protocol)
-- ================================================================

-- Identity Key + Signed PreKey bundle (публичные части)
-- Приватные ключи хранятся ТОЛЬКО на устройстве (Secure Enclave / Titan M2)
CREATE TABLE x3dh_prekeys (
    id                  UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID    NOT NULL REFERENCES users(id) ON DELETE CASCADE,

    -- Identity Key: постоянный X25519 public key (32 bytes)
    -- Меняется только при полной смене устройства
    identity_key_pub    BYTEA   NOT NULL CHECK (length(identity_key_pub) = 32),

    -- Signed PreKey: X25519, ротируется каждые 7 дней
    signed_prekey_id    INT     NOT NULL,
    signed_prekey_pub   BYTEA   NOT NULL CHECK (length(signed_prekey_pub) = 32),

    -- Ed25519 подпись: sign(IK_priv, SPK_pub)
    -- Получатель верифицирует перед X3DH, чтобы доказать что SPK принадлежит IK
    signed_prekey_sig   BYTEA   NOT NULL CHECK (length(signed_prekey_sig) = 64),

    spk_created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    -- Один актуальный bundle per пользователь
    UNIQUE(user_id)
);

COMMENT ON TABLE x3dh_prekeys IS
    'X3DH prekey bundle. Только публичные ключи. Приватные — исключительно в Secure Enclave устройства.';

-- One-Time PreKeys: каждый используется РОВНО ОДИН РАЗ, затем DELETE
CREATE TABLE one_time_prekeys (
    id          UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID    NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    prekey_id   INT     NOT NULL,
    prekey_pub  BYTEA   NOT NULL CHECK (length(prekey_pub) = 32),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    UNIQUE(user_id, prekey_id)
);

CREATE INDEX idx_one_time_prekeys_user
    ON one_time_prekeys(user_id, prekey_id);

COMMENT ON TABLE one_time_prekeys IS
    'X3DH One-Time PreKeys. Сервер выдаёт по одному за запрос и немедленно DELETE. Клиент пополняет до 50 штук.';

-- Метаданные чатов (без содержимого — оно в ClickHouse зашифрованным)
CREATE TABLE chats (
    id          UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    is_group    BOOLEAN NOT NULL DEFAULT FALSE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE chat_members (
    chat_id     UUID NOT NULL REFERENCES chats(id)   ON DELETE CASCADE,
    user_id     UUID NOT NULL REFERENCES users(id)   ON DELETE CASCADE,
    role        TEXT NOT NULL DEFAULT 'member',  -- 'admin', 'member'
    joined_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (chat_id, user_id)
);

CREATE INDEX idx_chat_members_user ON chat_members(user_id);
```

**Flutter-клиент Phase 1 (базовый):**
```dart
// Только: получить соль → Argon2id → показать MK fingerprint
// WebAuthn: получить challenge → WebAuthn API → получить JWT
// Хранить JWT в SecureStorage

class AuthBloc extends Bloc<AuthEvent, AuthState> {
  Future<void> _onLogin(LoginEvent e, Emitter<AuthState> emit) async {
    // 1. Получить соль
    final saltResponse = await api.getSalt(e.username);

    // 2. Верифицировать HMAC (защита от подмены соли)
    final expectedFp = await secureStorage.read(key: 'salt_fp_${e.username}');
    final actualFp = sha256(saltResponse.saltHmac);
    if (expectedFp != null && expectedFp != actualFp) {
      emit(AuthState.error("Salt fingerprint mismatch — возможна атака!"));
      return;
    }

    // 3. Derive MK через Argon2id (в изолированном isolate)
    final mk = await compute(deriveKey, ArgonParams(e.password, saltResponse.salt));

    // 4. WebAuthn: запрос challenge + NFC tap кольцом
    final loginOptions = await api.getWebAuthnLoginOptions(e.username);
    final assertion = await webAuthn.authenticate(loginOptions);

    // 5. Отправить assertion → получить JWT
    final jwt = await api.webAuthnLoginFinish(assertion);
    await secureStorage.write(key: 'jwt', value: jwt);

    // 6. Сохранить MK в Secure Enclave (flutter_secure_storage → Keystore/Enclave)
    await secureStorage.write(key: 'mk', value: base64.encode(mk));

    // 7. Затереть mk в памяти Dart (лучшее что можно сделать в managed runtime)
    mk.fillRange(0, mk.length, 0);

    emit(AuthState.authenticated(jwt));
  }
}
```

**Deliverable Phase 1:**

```scala
object X3DHHandshake:
  def initiateWith(bobBundle: PreKeyBundle, aliceIk: KeyPair): ZIO[Any, Throwable, MasterSecret] =
    for
      _      <- ZIO.when(!Ed25519.verify(bobBundle.identityKeyPub,
                                          bobBundle.signedPreKeyPub,
                                          bobBundle.signedPreKeySig))(
                  ZIO.fail(new SecurityException("SPK signature verification failed")))
      ekPair <- ZIO.succeed(X25519.generateKeyPair())
      dh1    = X25519.computeSharedSecret(aliceIk.privateKey, bobBundle.signedPreKeyPub)
      dh2    = X25519.computeSharedSecret(ekPair.privateKey,  bobBundle.identityKeyPub)
      dh3    = X25519.computeSharedSecret(ekPair.privateKey,  bobBundle.signedPreKeyPub)
      dh4    = X25519.computeSharedSecret(ekPair.privateKey,  bobBundle.oneTimePreKeyPub)
      f      = Array.fill(32)(0xFF.toByte)
      ikm    = f ++ dh1 ++ dh2 ++ dh3 ++ dh4
      ms     = HKDF.expand(ikm, info = "bunker-x3dh-v1", len = 32)
      _      <- ZIO.succeed { dh1.fill(0); dh2.fill(0); dh3.fill(0); dh4.fill(0); ikm.fill(0) }
    yield MasterSecret(ms, ekPair.publicKey)
```

```java
@PlatformThread
public KillSwitchResult triggerKillSwitch(String emergencyPassword) {
    // 0. Верифицировать аварийный пароль
    if (!bcrypt.matches(emergencyPassword, config.getEmergencyPasswordHash())) {
        auditLog.record(null, "KILL_SWITCH_FAIL_AUTH", null);
        throw new UnauthorizedException("Invalid emergency password");
    }

    // 1. Атомарно заблокировать Redis (Lua script)
    redis.eval(KILL_SWITCH_LUA, Collections.emptyList(), Collections.emptyList());

    // 2. Параллельно:
    CompletableFuture<Void> revokeTokens = CompletableFuture.runAsync(() ->
        refreshTokenRepo.revokeAll()  // UPDATE refresh_tokens SET revoked=TRUE
    );
    CompletableFuture<Void> kafkaEvent = CompletableFuture.runAsync(() ->
        kafkaProducer.send("system-events", "SYSTEM_LOCKED",
            "{\"timestamp\":" + Instant.now().toEpochMilli() + "}")
    );
    CompletableFuture<Void> dbReadOnly = CompletableFuture.runAsync(() ->
        jdbcTemplate.execute("SET SESSION CHARACTERISTICS AS TRANSACTION READ ONLY")
    );

    CompletableFuture.allOf(revokeTokens, kafkaEvent, dbReadOnly).join();

    auditLog.record(null, "KILL_SWITCH", Map.of("timestamp", Instant.now()));
    return KillSwitchResult.SUCCESS;
}
```

**BIP39 Recovery — схема:**
```
Генерация recovery мнемоники (при регистрации):
  1. entropy = SecureRandom.nextBytes(32)    // 256 bits → 24 слова BIP39
  2. passphrase = SecureRandom.nextBytes(16) // 12 слов дополнительных (BIP39 extension)
  3. seed = BIP39.mnemonicToSeed(words_24, passphrase_words_12)  // PBKDF2-HMAC-SHA512
  4. recovery_mk = Argon2id(seed[0:32], static_recovery_salt, m=131072, t=4, p=4)
  5. Верифицировать: recovery_mk должен расшифровать ECK категории "Recovery"
  6. Показать пользователю слова (24 + 12 = 36) для записи на бумаге
  7. recovery_mk немедленно затереть после верификации

Использование (триггер: скрытый жест 3 пальца × 3 секунды):
  1. Ввести 24 основных слова → 12 дополнительных
  2. Вычислить recovery_mk (медленный Argon2id — намеренно)
  3. Расшифровать ECK из PostgreSQL → получить CK → расшифровать данные
  4. Разблокировать аккаунт (SYSTEM_LOCKED → false) через специальный endpoint
  5. Пользователь может сменить пароль и зарегистрировать новое кольцо
```

### 10.1 Матрица угроз и защит

| Угроза | Защита |
|---|---|
| Взлом пароля (brute force) | Argon2id(m=65536): 64МБ RAM per попытку, GPU неэффективен |
| Кража телефона | MK в Secure Enclave: без биометрии/PIN телефона MK не извлечь |
| Клонирование NFC-кольца | WebAuthn signCount: clone всегда даёт ≤ stored count → REJECT |
| MITM подмена соли | HMAC-SHA256(salt) + TOFU fingerprint → клиент детектирует |
| Компрометация Sentinel | Vault изолирован за WireGuard; данные и ключи на Vault |
| Redis memory forensics | Redis на Vault (127.0.0.1), CK живёт 30с, no-persistence |
| Heap dump атака на JVM | Platform Thread + Arrays.fill + VarHandle.fullFence |
| Компрометация БД (SQL dump) | Все данные зашифрованы на уровне приложения; ключей в БД нет |
| Принуждение (coercion) | Kill Switch: аварийный пароль → полная блокировка |
| Потеря кольца | BIP39 recovery (36 слов): математическое восстановление MK |
| Потеря устройства | MK в Secure Enclave не извлекаем; recovery через BIP39 |

### 10.2 Полная Kill Switch последовательность

```
Триггер: POST /auth/kill-switch {emergency_password}

1. [AUTH]    Верифицировать bcrypt(emergency_password, stored_hash)
             Если fail → audit log "KILL_SWITCH_FAIL" → reject

2. [ATOMIC]  Redis Lua script (один EVAL — атомарно):
             SET SYSTEM_LOCKED 1 NX
             + DEL всех "ck:*" ключей (все активные CK мгновенно)
             + SET SYSTEM_LOCKED 1 (восстановить после KEYS scan)

3. [INSTANT] Gateway начинает возвращать 503 на ВСЕ /api/* запросы
             (проверяет SYSTEM_LOCKED в Redis при каждом запросе)

4. [PARALLEL, не блокируют друг друга]:
   4a. PostgreSQL: UPDATE refresh_tokens SET revoked=TRUE, revoked_at=NOW()
       → все Refresh Token инвалидированы
   4b. Kafka: produce("system-events", "SYSTEM_LOCKED", {ts, reason})
       → compacted topic: состояние сохраняется между рестартами
   4c. PostgreSQL: ALTER SYSTEM SET default_transaction_read_only=on;
       SELECT pg_reload_conf();
       → БД только на чтение (данные не изменяемы)

5. [LOG]     audit_log: event_type='KILL_SWITCH', metadata={reason, timestamp}

6. [RESPONSE] → 200 OK {status: LOCKED, message: "System locked"}

Разблокировка (только через BIP39):
  POST /auth/recovery {words_24, passphrase_words_12, new_password}
  1. Вычислить recovery_mk из BIP39 (медленно, намеренно: 4с на быстром устройстве)
  2. Расшифровать ECK категории "Recovery" → CK → верифицировать
  3. ALTER SYSTEM RESET default_transaction_read_only; SELECT pg_reload_conf();
  4. DEL SYSTEM_LOCKED в Redis
  5. Разрешить смену пароля и регистрацию нового кольца
  6. audit_log: KILL_SWITCH_UNLOCK
```
