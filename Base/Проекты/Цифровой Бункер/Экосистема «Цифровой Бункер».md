## 0. Принципы системы

Система строится на трёх абсолютных принципах, которые не могут быть нарушены ни одним решением:

1. **Zero-Trust by Default** — ни одна зона, сервис или сетевой сегмент не считается доверенным автоматически. Каждый запрос проходит явную верификацию identity и прав.
2. **Envelope Encryption** — сервер никогда не видит сырых данных. Ключи существуют в plaintext только в RAM клиента и изолированной Vault-зоне, только в момент операции, только в виде `byte[]` с немедленным обнулением.
3. **Аппаратная привязка** — Master Key и сессия привязаны к физическому устройству (Titan M2 / Secure Enclave + NFC-кольцо). Компрометация одного фактора не компрометирует систему.

## 1. Инфраструктура и топология сети

Система физически и логически разделена на две изолированные зоны, соединённые зашифрованным L3-туннелем.

### 1.1 Sentinel (Узел-Щит)

- **Среда:** Публичный VPS, Ubuntu 24.04 LTS.
- **Роль:** Точка входа, терминация TLS, JWT-валидация, WebAuthn-регистрация/аутентификация, маршрутизация трафика к Vault, push-уведомления.
- **Данных не хранит.** Redis на Sentinel отсутствует.
- **Защита:** nftables разрешает входящий трафик только на 443/tcp. Все исходящие соединения — только через WireGuard-интерфейс к Vault.
- **DDoS:** rate-limiting на уровне Gateway (Redis sliding window на стороне Vault через gRPC).

### 1.2 Vault (Узел-Хранилище)

- **Среда:** Домашний мини-ПК Firebat, без публичного IP.
- **Роль:** Изолированное ядро. Хранит все данные, обрабатывает криптографию и логику мессенджера.
- **Принимает соединения** только из WireGuard-интерфейса от Sentinel.
- **Межсервисная авторизация:** SPIFFE/SPIRE — каждый микросервис имеет SVID-сертификат. `bunker-storage` и `bunker-chat` принимают gRPC только от сервисов с валидным SVID в предустановленном trust bundle.

### 1.3 Клиенты

- **Mobile:** Flutter на Pixel 8 (Titan M2 — хранение MK).
- **Desktop:** macOS на MacBook (Secure Enclave — хранение MK).
- **Аппаратный фактор:** NFC-кольцо (FIDO2 CTAP2 — не U2F!). Проверяется `signCount` для детекции клонирования.

## 2. Криптография и контроль доступа

### 2.1 Иерархия ключей (Envelope Encryption)

```
Пароль пользователя + Salt (верифицированная HMAC)
           │
           ▼ Argon2id (m=65536, t=3, p=4)
     Master Key (MK)
     [хранится в Secure Enclave / Titan M2, никогда не покидает устройство]
           │
           ▼ AES-256-GCM decrypt
  Category Key (CK) для категории "Documents", "Finance", ...
  [существует в RAM только во время операции, сразу обнуляется]
           │
           ▼ AES-256-GCM decrypt (EDK = Encrypted Data Key)
     Data Key (DK) для конкретного файла/секрета
     [существует в RAM только во время потокового дешифрования]
           │
           ▼ Tink StreamingAead (AES-256-GCM-HKDF-4KB)
        Plaintext
```

### 2.2 Генерация Master Key

1. Пользователь вводит пароль на устройстве.
2. Клиент запрашивает `GET /auth/salt/{username}` → получает `{salt: bytes, salt_hmac: bytes}`.
3. **Клиент верифицирует HMAC соли** (`HMAC-SHA256(salt, serverStaticKey)`). `serverStaticKey` — публичный ключ, закреплённый в приложении при первой регистрации через WebAuthn attestation (TOFU + pin). Если HMAC не совпадает — отказ. Это защита от MITM-подмены соли для rainbow table атаки.
4. `MK = Argon2id(password, salt, m=65536, t=3, p=4)` — 32 байта.
5. MK сохраняется в аппаратном анклаве. По сети не передаётся никогда.

### 2.3 Инъекция ключей — исправленная схема

> [!WARNING] Исправлено по аудиту
> Исходная схема хранила CK в Redis на публичном Sentinel. **Исправлено:** Redis перенесён на Vault. CK никогда не покидает изолированную зону.

**Исправленный поток:**
```
1. Client расшифровывает ECK своим MK → получает CK (plaintext, только в RAM)
2. Client создаёт ephemeral X25519 keypair (EK_priv, EK_pub)
3. Client вычисляет shared secret: shared = ECDH(EK_priv, Vault_pub)
4. Client шифрует CK: EncCK = AES-256-GCM(CK, shared)
5. Client отправляет запрос на Vault: {EncCK, EK_pub, session_id}
   (через TLS + gRPC mTLS — двойное шифрование транспорта)
6. Vault вычисляет shared secret: shared = ECDH(Vault_priv, EK_pub)
7. Vault расшифровывает CK → кладёт в локальный Redis с TTL=30s
   SET "ck:{session_id}" {CK} EX 30
8. Vault немедленно начинает операцию с файлом/секретом
9. После операции: Arrays.fill(ck, 0) + VarHandle.fullFence() + Redis DEL
```

**Redis на Vault — конфигурация безопасности:**
```conf
requirepass <256-bit-random-token>
tls-port 6380
bind 127.0.0.1            # только loopback — нет сетевого доступа
maxmemory-policy noeviction  # CK не вытесняется в swap
rename-command KEYS ""    # запрет опасных команд
rename-command DEBUG ""
rename-command CONFIG ""
save ""                   # отключить RDB persistence (CK не должны быть на диске)
appendonly no             # отключить AOF persistence
```

### 2.4 WebAuthn / NFC-кольцо (аппаратный фактор)

**Требования к кольцу:** FIDO2 CTAP2 (не FIDO U2F). Поддерживает `clientPin`, user presence, user verification.

**Поток аутентификации:**
```
1. Клиент отправляет username → получает challenge (32 случайных байта)
2. Клиент инициирует WebAuthn assertion:
   - Платформа запрашивает NFC-кольцо (касание)
   - Кольцо подписывает {challenge + clientDataJSON + rpId hash} своим приватным ключом
3. Клиент отправляет {id, response.authenticatorData, response.signature, response.clientDataJSON}
4. bunker-auth верифицирует подпись публичным ключом кольца (из PostgreSQL)
5. Проверяет: signCount > stored_count (защита от клонирования кольца)
6. Обновляет stored signCount
7. Если ок → выдаёт JWT (ECDSA P-256, 15 мин) + Refresh Token (90 дней, в PostgreSQL)
```

**Flutter-интеграция:**
- Android 14+: `android.security.identity` + `CredentialManager` API.
- iOS/macOS: `ASAuthorizationController` (passkeys).
- Пакет: `passkeys` (pub.dev) — нативные платформенные WebAuthn API.

### 2.5 Защита памяти в Java (Production-паттерны)

> [!WARNING] Исправлено по аудиту
> JIT может устранить `Arrays.fill` как dead write. Virtual Threads сериализуют стек на heap. Оба вектора закрыты.

```java
/**
 * ПРАВИЛО 1: Крипто-операции только в Platform Thread.
 * Virtual Thread при parking сериализует стек на heap — секрет утекает.
 */
CompletableFuture<byte[]> decryptAsync(byte[] encData, UUID sessionId) {
    return CompletableFuture.supplyAsync(
        () -> decryptInPlatformThread(encData, sessionId),
        platformCryptoExecutor  // Executors.newFixedThreadPool(4) — platform threads
    );
}

/**
 * ПРАВИЛО 2: Zeroing с memory fence — запрет JIT dead-write elimination.
 */
private byte[] decryptInPlatformThread(byte[] encData, UUID sessionId) {
    byte[] ck = redis.get("ck:" + sessionId);
    try {
        return tinkAead.decrypt(encData, sessionId.toString().getBytes());
    } finally {
        Arrays.fill(ck, (byte) 0);
        VarHandle.fullFence();          // барьер: JIT не уберёт fill как dead write
        redis.del("ck:" + sessionId);   // немедленно, не ждать TTL
    }
}

/**
 * ПРАВИЛО 3: Никогда не конвертировать секрет в String.
 * String интернируется, GC не гарантирует немедленную очистку.
 */
// НЕПРАВИЛЬНО:
String password = new String(secretBytes);  // ❌ String → String pool

// ПРАВИЛЬНО:
byte[] password = secretBytes;  // ✅ byte[] → Arrays.fill после использования
char[] passwordChars = new char[secretBytes.length];
// ... использование ...
Arrays.fill(passwordChars, '\0');
VarHandle.fullFence();
```

**JVM-флаги для production:**
```
-XX:-HeapDumpOnOutOfMemoryError    # запрет автодампов с секретами
-XX:+UseZGC                        # ZGC — меньше stop-the-world, меньше окон уязвимости
-Djava.security.egd=file:/dev/urandom  # SecureRandom без блокировки
```

### 2.6 Аварийное восстановление (Recovery)

- **Триггер:** скрытый жест в UI (например, 3-пальцевое нажатие 3 секунды).
- **Механизм:** 24 слова BIP39 (стандарт) + passphrase (12 дополнительных слов как BIP39 extension).
- **Алгоритм:** `MK = Argon2id(BIP39_seed_bytes, static_recovery_salt, m=131072, t=4, p=4)`.
- **Важно:** CLI-утилита верификации мнемоники должна быть протестирована на реальных устройствах до деплоя Phase 1.

### 2.7 Kill Switch — атомарная процедура

> [!WARNING] Исправлено по аудиту
> Неатомарная последовательность создавала временное окно для атакующего. Исправлено через Lua-скрипт.

```lua
-- kill_switch.lua — атомарно выполняется на Redis Vault
-- Вызывается через EVAL одним вызовом
local locked = redis.call('SET', 'SYSTEM_LOCKED', '1', 'NX')
if locked then
    redis.call('FLUSHDB')                   -- удалить все CK
    redis.call('SET', 'SYSTEM_LOCKED', '1') -- восстановить флаг после flush
end
return locked
```

**Полная последовательность Kill Switch:**
```
1. [Atomic] Lua script → SET SYSTEM_LOCKED=1 + FLUSHDB на Redis Vault
2. [Immediate] bunker-gateway начинает возвращать 503 на все запросы
   (проверяет SYSTEM_LOCKED при каждом запросе — через gRPC к Vault)
3. [Parallel] Revoke всех Refresh Tokens в PostgreSQL:
   UPDATE refresh_tokens SET revoked = TRUE WHERE revoked = FALSE
4. [Parallel] Kafka event: topic=system-events, key=SYSTEM_LOCKED, value={timestamp, reason}
   (compacted topic — state сохраняется между рестартами)
5. [PostgreSQL] Переключить в read-only:
   ALTER SYSTEM SET default_transaction_read_only = on; SELECT pg_reload_conf();
6. [Audit] INSERT INTO audit_log (event_type='KILL_SWITCH', metadata={reason})
```

**Разблокировка:** только через Recovery-фразу BIP39, которая генерирует MK → разблокирует административный endpoint.

## 3. Микросервисная архитектура (The Polyglot Stack)

### Межсервисная авторизация (SPIFFE/SPIRE)

> [!WARNING] Исправлено по аудиту
> Отсутствие service-to-service auth внутри Vault оставляло все сервисы открытыми при компрометации Sentinel. Закрыто через SPIFFE.

```
SPIRE Server (Vault) выдаёт SVID каждому воркеру:
  spiffe://bunker.local/sentinel/gateway     → может вызывать storage, chat
  spiffe://bunker.local/vault/storage        → может читать Redis, PostgreSQL
  spiffe://bunker.local/vault/chat           → может читать Kafka, ClickHouse
  spiffe://bunker.local/vault/notify         → может читать Kafka (только consumer)

Каждый gRPC-вызов: mutual TLS с SVID сертификатами.
Vault-сервисы проверяют SPIFFE ID caller'а через TrustDomain policy.
```

**Минимальная альтернатива (если SPIFFE избыточен для v1):** mTLS с клиентскими сертификатами, выданными внутренним CA (step-ca). Каждый сервис имеет свою пару cert/key. Валидность сертификата = авторизация.

### A. `bunker-gateway` — Java 21, Spring Cloud Gateway 4.x

**Узел:** Sentinel. **Роль:** единственная точка входа в систему.

```
Цепочка фильтров (последовательно):
  1. TLS termination (сертификат Let's Encrypt / Caddy)
  2. Rate limit (gRPC вызов к bunker-storage → Redis sliding window на Vault)
  3. JWT validation (ECDSA P-256 подпись, issuer=bunker-auth, exp check)
  4. SYSTEM_LOCKED check (gRPC → Vault, cached 1s в local Caffeine cache)
  5. WebAuthn session validation (проверка hardware_verified флага в JWT claims)
  6. Route: /api/storage/* → gRPC → bunker-storage (Vault)
             /api/chat/*   → gRPC → bunker-chat (Vault)
             /auth/*       → bunker-auth (local, Sentinel)
```

**Virtual Threads:** все gateway-операции (I/O-bound) используют Virtual Threads через `spring.threads.virtual.enabled=true`.

**Важно:** Gateway никогда не проксирует CK. Он маршрутизирует только бизнес-запросы.

### B. `bunker-auth` — Java 21, Spring Boot 3.4, Spring Security 6

**Узел:** Sentinel.

```
Endpoints:
  POST /auth/register
       → генерировать salt (32 random bytes)
       → вычислить salt_hmac = HMAC-SHA256(salt, server_static_key)
       → INSERT users(username, argon2_salt, salt_hmac)
       → ответ: {user_id, webauthn_registration_options}

  GET  /auth/salt/{username}
       → SELECT argon2_salt, salt_hmac FROM users WHERE username = ?
       → ответ: {salt, salt_hmac}
       (клиент ОБЯЗАН верифицировать HMAC перед Argon2id)

  POST /auth/webauthn/register
       → библиотека: webauthn4j-spring-security 0.10+
       → проверить attestation, сохранить {credential_id, public_key_cose, aaguid}
       → закрепить fingerprint salt в Secure Enclave клиента

  POST /auth/webauthn/login
       → верифицировать assertion подпись
       → проверить signCount > stored_count (детекция клонирования)
       → обновить signCount
       → выдать JWT (ECDSA P-256, exp=15m, claims={user_id, hardware_verified=true})
       → выдать Refresh Token (random 256 bits) → хранить SHA-256(token) в PostgreSQL

  POST /auth/kill-switch
       → проверить emergency_password (bcrypt, отдельный от MK)
       → gRPC вызов к bunker-storage: TriggerKillSwitch()
       → выполнить атомарную процедуру (секция 2.7)

  POST /auth/refresh
       → проверить SHA-256(token) в PostgreSQL
       → проверить device_fp совпадает
       → выдать новый JWT
```

**Библиотеки:** `com.webauthn4j:webauthn4j-spring-security:0.10+`, `io.jsonwebtoken:jjwt-api:0.12+` (ECDSA), `de.mkammerer:argon2-jvm:2.11`.

**Streaming шифрование файлов (Tink StreamingAead):**
```java
// Upload: шифрование потоком, без загрузки в память
@PlatformThread  // кастомная аннотация для enforcement
public void uploadFile(InputStream rawStream, Path destPath, byte[] ck) {
    byte[] dk = generateDataKey();  // 32 random bytes
    byte[] edk = aesGcm.encrypt(dk, associatedData);  // зашифровать DK через CK

    KeysetHandle handle = TinkProtoKeysetFormat.parseKeyset(buildKeyset(dk), ...);
    StreamingAead streamingAead = handle.getPrimitive(StreamingAead.class);

    try (FileOutputStream fos = new FileOutputStream(destPath.toFile());
         OutputStream encStream = streamingAead.newEncryptingStream(fos, associatedData)) {
        rawStream.transferTo(encStream);  // потоковая запись без буферизации в heap
    } finally {
        Arrays.fill(dk, (byte) 0);
        VarHandle.fullFence();
    }
    // Сохранить в PostgreSQL: edk, edk_iv, edk_tag, storage_path, encrypted_name
}

// Download: потоковое дешифрование
public void downloadFile(UUID fileId, OutputStream clientStream, byte[] ck) {
    FileMetadataRow meta = repo.findById(fileId);
    byte[] dk = aesGcm.decrypt(meta.edk(), meta.edkIv(), ck);  // platform thread

    KeysetHandle handle = buildHandle(dk);
    StreamingAead streamingAead = handle.getPrimitive(StreamingAead.class);

    try (FileInputStream fis = new FileInputStream(meta.storagePath());
         InputStream decStream = streamingAead.newDecryptingStream(fis, associatedData)) {
        decStream.transferTo(clientStream);
    } finally {
        Arrays.fill(dk, (byte) 0);
        VarHandle.fullFence();
        // CK зануляется выше в InjectCategoryKey обработчике
    }
}
```

**Secure Delete:**
```java
public void secureDelete(Path path) {
    long size = Files.size(path);
    // Три прохода: нули, единицы, случайные байты (DoD 5220.22-M lite)
    try (FileChannel ch = FileChannel.open(path, WRITE)) {
        ch.write(ByteBuffer.allocate((int) size));          // pass 1: zeros
        ch.position(0);
        ByteBuffer ones = ByteBuffer.allocate((int) size);
        Arrays.fill(ones.array(), (byte) 0xFF);
        ch.write(ones);                                     // pass 2: ones
        ch.position(0);
        byte[] random = SecureRandom.getSeed((int) size);
        ch.write(ByteBuffer.wrap(random));                  // pass 3: random
        Arrays.fill(random, (byte) 0);
        VarHandle.fullFence();
    }
    Files.delete(path);
}
```

### D. `bunker-chat` — Scala 3.4, ZIO 2.1, ZIO HTTP, fs2-kafka

**Узел:** Vault.

**Почему ZIO:**
- `ZIO STM` — транзакционный in-memory state без locks для статусов сообщений и presence.
- `ZIO Fiber` — 1 fiber per WebSocket connection, нет thread-per-connection overhead.
- `ZIO Scope` — управление lifecycle соединений без resource leak.
- `ZIO Schedule` — retry/backoff политики для Kafka producer без boilerplate.

```scala
// WebSocket handler — 1 fiber per connection
def handleConnection(userId: UserId, ws: WebSocketChannel): ZIO[Scope, Throwable, Unit] =
  for
    _         <- presenceService.markOnline(userId)
    inbox     <- Queue.unbounded[OutboundMessage]
    // Fiber 1: читать входящие сообщения и публиковать в Kafka
    sendFiber <- ws.receiveStream
                   .mapZIO(msg => validateAndPublish(msg, userId))
                   .runDrain
                   .fork
    // Fiber 2: читать из inbox и отправлять клиенту
    recvFiber <- ZStream.fromQueue(inbox)
                   .mapZIO(msg => ws.send(msg.toFrame))
                   .runDrain
                   .fork
    _         <- sendFiber.join race recvFiber.join
    _         <- presenceService.markOffline(userId)
  yield ()
```

**Kafka как слепой курьер:**
```scala
// Сервер НИКОГДА не декодирует payload — только маршрутизирует зашифрованные байты
case class KafkaMessage(
  chatId:           UUID,
  senderIdHash:     Array[Byte],  // HMAC(userId, serverKey) — не userId напрямую
  encryptedPayload: Array[Byte],  // зашифровано Chat Key (X3DH) — сервер не знает DK
  timestamp:        Instant
)
// topic: "chat-messages-{chatId}" (один топик на чат, partitioned by chatId)
```

## 4. Мессенджер — E2E шифрование (X3DH + Double Ratchet)

> [!WARNING] Исправлено по аудиту
> Исходная схема хранила Chat Key в Infisical (потенциально внешний облачный сервис). Заменено на Signal Protocol X3DH с хранением в PostgreSQL.

### 4.1 Протокол X3DH — начальный обмен ключами

```
Предварительная подготовка (Bob публикует в PostgreSQL):
  IK_B   = Identity Key (X25519, постоянный)
  SPK_B  = Signed PreKey (X25519, меняется каждые 7 дней)
  Sig_B  = Ed25519(IK_B, SPK_B)  — подпись для верификации
  OPK_B  = One-Time PreKey (X25519, используется однократно)

Alice инициирует чат с Bob:
  1. Alice получает из PostgreSQL: {IK_B_pub, SPK_B_pub, Sig_B, OPK_B_pub}
  2. Alice верифицирует: Ed25519.verify(IK_B_pub, SPK_B_pub, Sig_B) → ok
  3. Alice генерирует ephemeral key: EK_A (X25519)
  4. Alice вычисляет 4 DH:
       DH1 = X25519(IK_A_priv, SPK_B_pub)
       DH2 = X25519(EK_A_priv, IK_B_pub)
       DH3 = X25519(EK_A_priv, SPK_B_pub)
       DH4 = X25519(EK_A_priv, OPK_B_pub)
  5. master_secret = KDF(DH1 || DH2 || DH3 || DH4)
  6. Chat Key = HKDF(master_secret, info="bunker-chat-v1", len=32)
  7. Alice шифрует первое сообщение Chat Key + встраивает {IK_A_pub, EK_A_pub, OPK_B_id}
  8. Используемый OPK_B_id → DELETE из PostgreSQL (one-time!)

Bob получает первое сообщение:
  1. Bob извлекает {IK_A_pub, EK_A_pub, OPK_B_id} из сообщения
  2. Повторяет те же 4 DH операции зеркально
  3. Получает идентичный Chat Key
  4. Расшифровывает сообщение

После этого: Double Ratchet для Forward Secrecy каждого сообщения.
```

### 4.2 Хранение (без Infisical — всё в PostgreSQL)

```sql
-- Prekey bundles хранятся в PostgreSQL на Vault
-- x3dh_prekeys + one_time_prekeys (см. SQL-схему, секция 6)
-- Chat Key НЕ ХРАНИТСЯ на сервере — вычисляется независимо каждым устройством
-- История чатов — ClickHouse, только зашифрованные payload bytes
```

### 4.3 Поток обмена сообщениями

```
Alice (клиент):
  1. Шифрует текст текущим Double Ratchet ключом → bytes
  2. WebSocket → bunker-chat: {chatId, encryptedPayload}

bunker-chat (сервер):
  3. Валидирует JWT sender
  4. Публикует в Kafka: topic=chat-{chatId}, value={encryptedPayload, timestamp}
  5. Публикует в Kafka: topic=notifications, value={recipientId, chatId}
  (сервер НЕ читает payload, видит только zашифрованные байты)

bunker-notify (Sentinel):
  6. Kafka consumer → FCM wake-up push к Bob (только chatId, без контента)

Bob (клиент):
  7. Получает push → открывает приложение
  8. Загружает encryptedPayload через WebSocket
  9. Расшифровывает Double Ratchet ключом → plaintext
```

## 6. SQL-схема PostgreSQL (Production-ready)

```sql
-- ============================================================
-- EXTENSIONS
-- ============================================================
CREATE EXTENSION IF NOT EXISTS "pgcrypto";   -- gen_random_uuid(), gen_random_bytes()
CREATE EXTENSION IF NOT EXISTS "pg_trgm";    -- fuzzy search по зашифрованным именам (если нужно)

-- ============================================================
-- USERS & AUTH
-- ============================================================

CREATE TABLE users (
    id            UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    username      TEXT        NOT NULL UNIQUE,
    -- Соль для Argon2id: публичная, 32 байта случайных данных
    argon2_salt   BYTEA       NOT NULL,
    -- HMAC-SHA256(argon2_salt, server_static_key): integrity защита соли от подмены
    salt_hmac     BYTEA       NOT NULL,
    is_locked     BOOLEAN     NOT NULL DEFAULT FALSE,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_login_at TIMESTAMPTZ
);

-- FIDO2 / WebAuthn credentials (NFC-кольцо, Touch ID, Passkey)
CREATE TABLE webauthn_credentials (
    id               UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id          UUID    NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    credential_id    BYTEA   NOT NULL UNIQUE,        -- FIDO2 credential ID (variable len)
    public_key_cose  BYTEA   NOT NULL,               -- COSE-encoded public key
    sign_count       BIGINT  NOT NULL DEFAULT 0,     -- монотонный счётчик (replay protection)
    aaguid           BYTEA,                          -- authenticator model GUID
    device_label     TEXT,                           -- "NFC Ring", "MacBook Touch ID"
    transports       TEXT[],                         -- ["nfc"], ["internal"], ["hybrid"]
    backed_up        BOOLEAN NOT NULL DEFAULT FALSE, -- passkey synced to cloud (риск!)
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_used_at     TIMESTAMPTZ
);

-- JWT Refresh Tokens: не в JWT body, хранятся в БД
CREATE TABLE refresh_tokens (
    id           UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      UUID        NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash   BYTEA       NOT NULL UNIQUE,    -- SHA-256(raw_token) — raw не хранится
    device_fp    TEXT        NOT NULL,           -- device fingerprint для валидации
    expires_at   TIMESTAMPTZ NOT NULL,
    revoked      BOOLEAN     NOT NULL DEFAULT FALSE,
    revoked_at   TIMESTAMPTZ,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_refresh_tokens_user_active
    ON refresh_tokens(user_id)
    WHERE revoked = FALSE;

-- ============================================================
-- KEY MANAGEMENT (Envelope Encryption Layer)
-- ============================================================

-- Категории хранилища: Documents, Passwords, Finance, Work, ...
-- Для каждой категории — свой ECK (Encrypted Category Key)
CREATE TABLE categories (
    id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id    UUID        NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name       TEXT        NOT NULL,
    -- ECK = Category Key, зашифрованный Master Key пользователя (AES-256-GCM)
    -- MK используется только на клиенте для decrypt(ECK) → CK
    eck        BYTEA       NOT NULL,   -- ciphertext
    eck_iv     BYTEA       NOT NULL,   -- 12 bytes (GCM nonce)
    eck_tag    BYTEA       NOT NULL,   -- 16 bytes (GCM auth tag)
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(user_id, name)
);

-- ============================================================
-- FILE STORAGE
-- ============================================================

CREATE TABLE file_metadata (
    id               UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id          UUID    NOT NULL REFERENCES users(id),
    category_id      UUID    NOT NULL REFERENCES categories(id),

    -- Имя файла зашифровано Category Key на клиенте перед отправкой
    encrypted_name   BYTEA   NOT NULL,
    name_iv          BYTEA   NOT NULL,   -- 12 bytes
    name_tag         BYTEA   NOT NULL,   -- 16 bytes

    -- UUID-based путь на диске: /vault/files/{user_uuid}/{file_uuid}
    -- Не выводим из имени файла
    storage_path     TEXT    NOT NULL,

    size_bytes       BIGINT  NOT NULL,
    mime_type_enc    BYTEA,             -- зашифрованный MIME или NULL

    -- EDK = Data Encryption Key, зашифрованный Category Key
    -- Позволяет ротировать CK без перешифровки файлов (re-key только EDK)
    edk              BYTEA   NOT NULL,   -- Encrypted Data Key (AES-256-GCM)
    edk_iv           BYTEA   NOT NULL,
    edk_tag          BYTEA   NOT NULL,

    -- Tink StreamingAead параметры (без ключа — только метаданные шифрования)
    tink_keyset_info BYTEA,

    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at       TIMESTAMPTZ,        -- soft delete
    wiped_at         TIMESTAMPTZ         -- подтверждение secure overwrite воркером
);

CREATE INDEX idx_file_metadata_user_category
    ON file_metadata(user_id, category_id)
    WHERE deleted_at IS NULL;

-- ============================================================
-- PASSWORD VAULT (менеджер паролей)
-- ============================================================

CREATE TABLE vault_secrets (
    id             UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id        UUID    NOT NULL REFERENCES users(id),
    category_id    UUID    NOT NULL REFERENCES categories(id),

    -- Зашифрованный JSON-blob структуры:
    -- {title, login, password, url, notes, totp_secret, custom_fields[]}
    -- Шифруется: AES-256-GCM(json_bytes, CK)
    encrypted_blob BYTEA   NOT NULL,
    blob_iv        BYTEA   NOT NULL,   -- 12 bytes
    blob_tag       BYTEA   NOT NULL,   -- 16 bytes

    -- EDK для этой записи (для индивидуальной ротации ключей)
    edk            BYTEA,
    edk_iv         BYTEA,
    edk_tag        BYTEA,

    version        INT     NOT NULL DEFAULT 1,  -- optimistic locking
    created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at     TIMESTAMPTZ
);

CREATE INDEX idx_vault_secrets_user_category
    ON vault_secrets(user_id, category_id)
    WHERE deleted_at IS NULL;

-- ============================================================
-- MESSENGER — X3DH (Signal Protocol)
-- ============================================================

-- Identity + Signed PreKey bundle (публичные части)
CREATE TABLE x3dh_prekeys (
    id                 UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id            UUID    NOT NULL REFERENCES users(id) ON DELETE CASCADE,

    -- Identity Key: постоянный X25519 public key (32 bytes)
    -- Меняется только при потере устройства
    identity_key_pub   BYTEA   NOT NULL,

    -- Signed PreKey: ротируется каждые 7 дней
    signed_prekey_id   INT     NOT NULL,
    signed_prekey_pub  BYTEA   NOT NULL,   -- X25519 public key
    signed_prekey_sig  BYTEA   NOT NULL,   -- Ed25519 подпись Identity Key над SPK

    -- Timestamp для ротации SPK
    spk_created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    created_at         TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(user_id)    -- один актуальный bundle per user
);

-- One-Time PreKeys: каждый используется ровно один раз → DELETE после использования
CREATE TABLE one_time_prekeys (
    id           UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      UUID    NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    prekey_id    INT     NOT NULL,
    prekey_pub   BYTEA   NOT NULL,   -- X25519 public key (32 bytes)
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(user_id, prekey_id)
);
-- Клиент должен пополнять OPK до >=20 штук (сервер присылает предупреждение при <5)

-- Метаданные чатов (без контента — контент только в ClickHouse зашифрованным)
CREATE TABLE chats (
    id         UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
    is_group   BOOLEAN NOT NULL DEFAULT FALSE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE chat_members (
    chat_id    UUID NOT NULL REFERENCES chats(id) ON DELETE CASCADE,
    user_id    UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    joined_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (chat_id, user_id)
);

-- ============================================================
-- AUDIT & SYSTEM STATE
-- ============================================================

-- Партиционированная таблица для audit log (очистка старых партиций)
CREATE TABLE audit_log (
    id         BIGSERIAL   NOT NULL,
    user_id    UUID        REFERENCES users(id),  -- NULL для системных событий
    event_type TEXT        NOT NULL,
    -- Типы: LOGIN_SUCCESS, LOGIN_FAIL, WEBAUTHN_FAIL, WEBAUTHN_CLONE_DETECTED,
    --       KEY_INJECT, FILE_UPLOAD, FILE_ACCESS, FILE_DELETE, SECRET_READ,
    --       SECRET_WRITE, KILL_SWITCH, KILL_SWITCH_UNLOCK, RECOVERY_START,
    --       OPK_DEPLETED, SYSTEM_LOCKED
    ip_address INET,
    user_agent TEXT,
    -- Дополнительный контекст БЕЗ чувствительных данных
    metadata   JSONB,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);

-- Создавать партиции скриптом каждый месяц
CREATE TABLE audit_log_y2026m03
    PARTITION OF audit_log
    FOR VALUES FROM ('2026-03-01') TO ('2026-04-01');

CREATE INDEX idx_audit_log_user_time
    ON audit_log(user_id, created_at DESC);

CREATE INDEX idx_audit_log_event_type
    ON audit_log(event_type, created_at DESC);

-- Системное состояние (singleton-строки)
CREATE TABLE system_state (
    key        TEXT PRIMARY KEY,
    value      TEXT        NOT NULL,
    reason     TEXT,
    set_by     UUID        REFERENCES users(id),
    set_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Seed: начальное разблокированное состояние
INSERT INTO system_state(key, value, reason) VALUES
    ('SYSTEM_LOCKED', 'false', 'initial'),
    ('MAINTENANCE',   'false', 'initial');

-- ============================================================
-- HELPER VIEWS
-- ============================================================

-- Активные пользователи с их credential'ами
CREATE VIEW active_user_credentials AS
    SELECT u.id, u.username, u.last_login_at,
           array_agg(wc.device_label) AS devices
    FROM users u
    LEFT JOIN webauthn_credentials wc ON wc.user_id = u.id
    WHERE u.is_locked = FALSE
    GROUP BY u.id;

-- Предупреждение о низком количестве OPK
CREATE VIEW opk_low_stock AS
    SELECT user_id, COUNT(*) AS remaining_opks
    FROM one_time_prekeys
    GROUP BY user_id
    HAVING COUNT(*) < 5;
```
