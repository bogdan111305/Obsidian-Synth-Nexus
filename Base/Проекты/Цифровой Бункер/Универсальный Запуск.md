# Универсальный Запуск «Цифровой Бункер»

> [!INFO] Версия: 1.0 — 2026-03-20
> Цель: план развёртывания, который работает на одной машине (WSL/Docker) и без правки кода переносится на разные физические узлы (Sentinel VPS + Vault Firebat).
> Основан на мастер-плане: [[Экосистема «Цифровой Бункер»]]

---

## 0. Принцип «Deploy Once, Run Anywhere»

Весь конфиг вынесен в переменные окружения. Сервисы обращаются друг к другу через DNS-имена, а не IP. mTLS включён с первого дня — даже внутри одной Docker-сети.

```
Сейчас (один системник):              Потом (два узла):
┌─────────────────────────┐            ┌──────────────┐    ┌──────────────┐
│  Docker Desktop / WSL   │            │  Sentinel VPS│    │  Vault Firebat│
│                         │            │              │    │              │
│  external_bridge        │   →→→→→→   │ external_    │    │ internal_    │
│  ├─ bunker-gateway      │            │ bridge net   │    │ secure net   │
│  ├─ bunker-auth         │            │ ├─ gateway   │    │ ├─ storage   │
│  └─ bunker-notify       │            │ ├─ auth      │    │ ├─ chat      │
│                         │            │ └─ notify    │    │ ├─ postgres  │
│  internal_secure        │            └──────────────┘    │ ├─ redis    │
│  ├─ bunker-storage      │               WireGuard →→→→   │ ├─ kafka    │
│  ├─ bunker-chat         │                                 │ └─ clickhouse│
│  ├─ postgres            │                                 └──────────────┘
│  ├─ redis               │
│  ├─ kafka               │
│  └─ clickhouse          │
└─────────────────────────┘
Один docker-compose.yml → при разносе: два отдельных compose-файла
```

## 4. Network Policy — изоляция данных

Docker `internal: true` запрещает любые маршруты из `internal_secure` наружу. Дополнительно — явные правила кто к кому может обращаться:

```
Разрешено:
  external_bridge → bunker-gateway:8443 (единственная точка входа)
  bunker-gateway  → bunker-auth     (gRPC mTLS, internal_secure)
  bunker-gateway  → bunker-storage  (gRPC mTLS, internal_secure)
  bunker-gateway  → bunker-chat     (gRPC mTLS, internal_secure)
  bunker-auth     → postgres        (SSL, port 5432)
  bunker-auth     → redis           (TLS, port 6380)
  bunker-storage  → postgres        (SSL, port 5432)
  bunker-storage  → redis           (TLS, port 6380)
  bunker-chat     → postgres        (SSL, port 5432)
  bunker-chat     → kafka           (SSL, port 9094)
  bunker-notify   → kafka           (SSL, port 9094, consumer only)

Запрещено (Docker network isolation):
  external_bridge → postgres        ← БД недоступна снаружи
  external_bridge → redis           ← Redis недоступен снаружи
  external_bridge → kafka           ← Kafka недоступна снаружи
  external_bridge → bunker-storage  ← Storage только через Gateway
  external_bridge → bunker-chat     ← Chat только через Gateway
  bunker-storage  → kafka           ← Storage не пишет в Kafka напрямую
  bunker-auth     → kafka           ← Auth не читает сообщения чата
```

> [!INFO] Redis остаётся в `internal_secure` даже на одной машине
> Это не условность локального старта — это архитектурное требование. Redis с CK никогда не должен быть доступен из external_bridge. Даже в dev-режиме.

## 5. mTLS между сервисами

mTLS включён с первого дня. Если всё крутится в одном compose — это не overhead, это привычка кода, которая не потребует правок при разносе на узлы.

### 5.1 Генерация сертификатов (скрипт)

```bash
#!/bin/bash
# scripts/gen-certs.sh
# Генерирует CA + сертификаты для каждого сервиса
# Запускать один раз: ./scripts/gen-certs.sh

set -euo pipefail

CERTS_DIR="./certs"
DAYS=365
SERVICES="gateway auth storage chat notify redis postgres kafka"

echo "→ Создаём CA..."
mkdir -p "$CERTS_DIR/ca"
openssl genrsa -out "$CERTS_DIR/ca/ca.key" 4096
openssl req -new -x509 -days $DAYS \
  -key "$CERTS_DIR/ca/ca.key" \
  -out "$CERTS_DIR/ca/ca.crt" \
  -subj "/CN=bunker-local-ca/O=Bunker/C=RU"

echo "→ Генерируем ECDSA P-256 ключ для JWT..."
mkdir -p "$CERTS_DIR/jwt"
openssl ecparam -name prime256v1 -genkey -noout \
  -out "$CERTS_DIR/jwt/ec-private.pem"
openssl ec -in "$CERTS_DIR/jwt/ec-private.pem" \
  -pubout -out "$CERTS_DIR/jwt/ec-public.pem"

for SERVICE in $SERVICES; do
  echo "→ Сертификат для $SERVICE..."
  mkdir -p "$CERTS_DIR/$SERVICE"

  # Генерируем ключ
  openssl genrsa -out "$CERTS_DIR/$SERVICE/server.key" 2048

  # CSR с SAN (Subject Alternative Names) для Docker DNS
  cat > "/tmp/$SERVICE-san.cnf" <<EOF
[req]
req_extensions = v3_req
distinguished_name = req_distinguished_name
[req_distinguished_name]
[v3_req]
subjectAltName = @alt_names
[alt_names]
DNS.1 = $SERVICE
DNS.2 = bunker-$SERVICE
DNS.3 = localhost
IP.1  = 127.0.0.1
EOF

  openssl req -new \
    -key "$CERTS_DIR/$SERVICE/server.key" \
    -out "/tmp/$SERVICE.csr" \
    -subj "/CN=$SERVICE/O=Bunker/C=RU" \
    -config "/tmp/$SERVICE-san.cnf"

  # Подписываем CA
  openssl x509 -req -days $DAYS \
    -in "/tmp/$SERVICE.csr" \
    -CA "$CERTS_DIR/ca/ca.crt" \
    -CAkey "$CERTS_DIR/ca/ca.key" \
    -CAcreateserial \
    -out "$CERTS_DIR/$SERVICE/server.crt" \
    -extensions v3_req \
    -extfile "/tmp/$SERVICE-san.cnf"

  # Клиентский сертификат (для mTLS как клиент)
  cp "$CERTS_DIR/$SERVICE/server.crt" "$CERTS_DIR/$SERVICE/client.crt"
  cp "$CERTS_DIR/$SERVICE/server.key" "$CERTS_DIR/$SERVICE/client.key"

  # Симлинк на CA
  ln -sf "../ca/ca.crt" "$CERTS_DIR/$SERVICE/ca.crt"

  echo "  ✓ $CERTS_DIR/$SERVICE/ готов"
done

# Kafka требует JKS keystore
echo "→ Генерируем Kafka JKS keystores..."
KAFKA_CERT_DIR="$CERTS_DIR/kafka"

# Server keystore
openssl pkcs12 -export \
  -in "$KAFKA_CERT_DIR/server.crt" \
  -inkey "$KAFKA_CERT_DIR/server.key" \
  -chain -CAfile "$CERTS_DIR/ca/ca.crt" \
  -name kafka-server \
  -out "/tmp/kafka-server.p12" \
  -passout pass:${KAFKA_KEYSTORE_PASSWORD:-changeme}

keytool -importkeystore \
  -srckeystore "/tmp/kafka-server.p12" \
  -srcstoretype PKCS12 \
  -srcstorepass ${KAFKA_KEYSTORE_PASSWORD:-changeme} \
```

### 5.2 mTLS в Java-сервисах (Spring Boot)

```java
// Универсальная конфигурация через переменные окружения
// Работает и локально (Docker), и в продакшне (разные узлы)

@Configuration
public class MtlsConfig {

    @Value("${TLS_CA_CERT}")
    private String caCertPath;

    @Value("${TLS_SERVER_CERT}")
    private String serverCertPath;

    @Value("${TLS_SERVER_KEY}")
    private String serverKeyPath;

    // gRPC сервер — принимает mTLS соединения
    @Bean
    public SslContext grpcServerSslContext() throws Exception {
        return SslContextBuilder.forServer(
                new File(serverCertPath),
                new File(serverKeyPath)
            )
            .trustManager(new File(caCertPath))
            .clientAuth(ClientAuth.REQUIRE)  // требуем клиентский сертификат
            .build();
    }

    // gRPC клиент — подключается с mTLS к другим сервисам
    @Bean
    public SslContext grpcClientSslContext() throws Exception {
        return SslContextBuilder.forClient()
            .keyManager(
                new File(serverCertPath),    // наш сертификат как клиента
                new File(serverKeyPath)
            )
            .trustManager(new File(caCertPath))
            .build();
    }

    // ManagedChannel — адрес через ENV, не хардкод
    @Bean
    public ManagedChannel storageChannel(
            @Value("${STORAGE_HOST}") String host,
            @Value("${STORAGE_GRPC_PORT}") int port,
            SslContext grpcClientSslContext) {
        return NettyChannelBuilder.forAddress(host, port)
            .sslContext(grpcClientSslContext)
            .build();
    }
}
```

```dart
// lib/network/secure_client.dart
import 'dart:io';
import 'package:flutter/services.dart';

Future<HttpClient> createSecureClient() async {
  final client = HttpClient();

  final caCert = AppConfig.customCaCert;
  if (caCert != null && caCert.isNotEmpty) {
    // Локальная разработка: загружаем наш CA
    final certBytes = await rootBundle.load(caCert);
    final context = SecurityContext();
    context.setTrustedCertificatesBytes(certBytes.buffer.asUint8List());
    return HttpClient(context: context);
  }

  // Продакшн: системный CA (Let's Encrypt)
  return client;
}
```

Когда появится VPS (Sentinel) и Firebat (Vault):

```bash
# Шаг 1: На Sentinel — запустить только sentinel-сервисы
# docker-compose.sentinel.yml (переопределяет compose.yml)
DEPLOY_MODE=sentinel \

               up -d bunker-gateway bunker-auth bunker-notify

# Шаг 2: На Vault (Firebat) — запустить только vault-сервисы
DEPLOY_MODE=vault \
docker compose -f docker-compose.yml \
               -f docker-compose.vault.yml \
               up -d postgres redis kafka clickhouse bunker-storage bunker-chat

# Единственное изменение: значения ENV (адреса хостов)
# Код сервисов — не меняется
```

**Что изменится при миграции:**
| Параметр | Локально (сейчас) | Продакшн (потом) |
|----------|-------------------|------------------|
| `STORAGE_HOST` | `bunker-storage` (Docker DNS) | `storage.vault.local` (WireGuard DNS) |
| `POSTGRES_HOST` | `postgres` | `postgres` (тот же, на Vault) |
| `WEBAUTHN_RP_ID` | `localhost` | `bunker.yourdomain.com` |
| `API_BASE_URL` | `https://192.168.1.100:8443` | `https://bunker.yourdomain.com` |
| Сертификаты | самоподписанные (gen-certs.sh) | Let's Encrypt (Sentinel) + step-ca (Vault) |
| WireGuard | не нужен | между Sentinel и Vault |
